# v1.3 Sync Failure Visibility Design

**Date**: 2026-04-28
**Branch**: `feature/phase-1-v1`
**Status**: Spec, ready for plan
**Builds on**: v1.2 (`docs/superpowers/specs/2026-04-28-task-v1.2-periodic-sync-and-freshness-design.md`)

## 0. Goal & Non-goals

**Goal**: Make persistent auto-sync failure **visible** so the user does not need to manually press "Sync Now" to discover the sync layer has been silently broken for 15+ minutes.

**Non-goals (deferred)**:
- Sidebar / menu-bar status indicator (deferred to v1.4)
- Configurable failing threshold (defaults to `periodicInterval`)
- Per-host failure surface (host-level errors are out of scope; only sync-cycle errors)
- Refresh token / silent re-auth (deferred per v1.2 §8)
- Network reachability gating via `NWPathMonitor` (one failed `URLSessionDataTask` is cheaper than polling reachability)

## 1. Architecture

Three coordinated changes, no new module:

```
SyncPreferences                HostSyncStore                       SyncSettingsView
─────────────                  ────────────                        ────────────────
+ notifyOnFailureEnabled       + lastSyncAttemptedAt (@Published)  + Toggle "Notify when sync fails"
  (UserDefaults key:           + lastSyncErrorKind   (@Published)  + Red failure text + "since" time
   catermNotifyOnFailureEnabled,    (.auth / .other)              + If toggle OFF, show
   default false)              + failingSince        (@Published)    "Turn on 'Notify…'" hint
                               + wasFailing (private, transient)
                               + manualInProgress (private)
                               + syncFailureState(now:…) free fn
                               + performSync catch edge detection
                               + UNUserNotificationCenter delivery
```

### Data flow (one failing auto-cycle)

1. `Timer.publish` / wake / mutation-debounce → `scheduleAutoSync()` → auth gate ✓ → `performSync()` starts
2. `performSync()` head writes `lastSyncAttemptedAt = Date()`
3. Op loop `createHost` throws → catch path writes `lastSyncErrorKind = .other` (401 → `.auth`); does **not** update `lastSyncedAt` (v1.2 carve-out preserved)
4. Catch path: `let nowFailing = isCurrentlyFailing()`; if `!wasFailing && nowFailing && !manualInProgress && preferences.notifyOnFailureEnabled` → fire `UNUserNotification`; if `!wasFailing && nowFailing` also writes `failingSince = lastSyncAttemptedAt`
5. Writes `wasFailing = nowFailing`, then rethrows
6. `SyncSettingsView`'s `TimelineView(.periodic by: 30)` re-renders: `syncFailureState(...)` returns `.failing` → red text appears with `failingSince` time

### Invariants

- **Manual sync does not trigger notification** (spec §3 Cycle 5 — manual failure is already visible to the user; gated via `manualInProgress`).
- **Manual sync still updates `lastSyncAttemptedAt`** (consistency: the user-visible "last attempted" must be the actual most recent attempt).
- **Auth-priority**: existing v1.2 `accountState == .sessionExpired` red banner takes precedence; `.failing` is the secondary state. Precedence: `.sessionExpired` > `.failing` > `.normal`.
- **`periodicSyncEnabled = false`** ⇒ `syncFailureState` returns `.normal` (user-paused sync should not nag; consistent with spec §3.2).
- **`lastSyncedAt == nil`** ⇒ `.normal` (a fresh-install user that has never had a successful sync is in the initial state, not "failing").
- **`failingSince` semantics**: timestamp of *first* entering failing state, not the most recent attempt. Reset to `nil` on recovery. Not persisted (re-derived on cold start; see §3 Cycle 1).

### 2.2 `HostSyncStore` extension

**File**: `apps/macos/Sources/HostSyncStore/HostSyncStore.swift`

#### 2.2.1 New types (file-top, alongside existing `SyncError`)

```swift
public enum SyncErrorKind: Equatable, Sendable {
    case auth     // 401 / .sessionExpired
    case other    // network / 5xx / decode error / etc.
}

public enum SyncFailureState: Equatable {
    case normal
    case failing(reason: SyncErrorKind, since: Date)
}

public protocol NotificationDelivering: Sendable {
    func add(_ request: UNNotificationRequest) async throws
}

public struct LiveNotificationCenter: NotificationDelivering {
    public init() {}
    public func add(_ request: UNNotificationRequest) async throws {
        try await UNUserNotificationCenter.current().add(request)
    }
}
```

#### 2.2.2 New `@Published` properties

```swift
@Published public private(set) var lastSyncAttemptedAt: Date?
@Published public private(set) var lastSyncErrorKind: SyncErrorKind?
@Published public private(set) var failingSince: Date?
```

Persistence:
```swift
private static let lastSyncAttemptedAtKey = "catermLastSyncAttemptedAt"
// lastSyncErrorKind and failingSince are NOT persisted; re-derived on cold start.
```

`init` hydrate (symmetric to `lastSyncedAt`):
```swift
let attemptedTimestamp = userDefaults.object(forKey: Self.lastSyncAttemptedAtKey) as? Double
self.lastSyncAttemptedAt = attemptedTimestamp.map(Date.init(timeIntervalSince1970:))
```

#### 2.2.3 Transient state (not @Published, not persisted)

```swift
private var wasFailing: Bool = false
private var manualInProgress: Bool = false
```

> **Why not persist `wasFailing`**: the cold-start path re-evaluates via `isCurrentlyFailing()` and lets the next cycle trigger the edge naturally — avoids stale state and ensures a meaningful notification on the first post-restart failure (see §3 Cycle 1).

#### 2.2.4 Pure derivation (free function, not on `@MainActor`, no `self`)

Place at file top:

```swift
public func syncFailureState(
    now: Date,
    lastSyncedAt: Date?,
    lastSyncAttemptedAt: Date?,
    lastSyncErrorKind: SyncErrorKind?,
    periodicSyncEnabled: Bool,
    failingThreshold: TimeInterval
) -> SyncFailureState {
    guard periodicSyncEnabled else { return .normal }
    guard let attempted = lastSyncAttemptedAt else { return .normal }
    guard let succeeded = lastSyncedAt else { return .normal }
    guard attempted > succeeded else { return .normal }
    guard now.timeIntervalSince(succeeded) > failingThreshold else { return .normal }
    let reason = lastSyncErrorKind ?? .other
    return .failing(reason: reason, since: attempted)
}
```

> Note: `since` is the *most recent attempted* timestamp. The store-level `failingSince` (the **first** failure of a run) is supplied to the view layer separately; the pure function operates on inputs alone (testable in isolation). The view derives display text from `store.failingSince ?? attempted`.

#### 2.2.5 `performSync()` modifications

**Head (after `setSyncing(true)`, before any client call)**:
```swift
let attempted = Date()
lastSyncAttemptedAt = attempted
userDefaults.set(attempted.timeIntervalSince1970, forKey: Self.lastSyncAttemptedAtKey)
```

**Success path (after the partial-apply success block)**:
```swift
lastSyncedAt = Date()
userDefaults.set(lastSyncedAt!.timeIntervalSince1970, forKey: Self.lastSyncedAtKey)
lastSyncErrorKind = nil
wasFailing = false
failingSince = nil
```

**Catch path**:
```swift
} catch {
    lastSyncErrorKind = classifySyncError(error)
    let nowFailing = isCurrentlyFailing()
    if !wasFailing && nowFailing {
        failingSince = lastSyncAttemptedAt
        if !manualInProgress && preferences.notifyOnFailureEnabled {
            await fireFailureNotification()
        }
    }
    wasFailing = nowFailing
    throw error
}
```

Helpers:
```swift
private func classifySyncError(_ error: Error) -> SyncErrorKind {
    if case .sessionExpired = error as? SyncError { return .auth }
    return .other
}

private func isCurrentlyFailing() -> Bool {
    if case .failing = syncFailureState(
        now: Date(),
        lastSyncedAt: lastSyncedAt,
        lastSyncAttemptedAt: lastSyncAttemptedAt,
        lastSyncErrorKind: lastSyncErrorKind,
        periodicSyncEnabled: preferences.periodicSyncEnabled,
        failingThreshold: periodicInterval
    ) {
        return true
    }
    return false
}

private func fireFailureNotification() async {
    let content = UNMutableNotificationContent()
    content.title = "Caterm sync is failing"
    content.body = "Click Sync Now in Sync Settings, or check your connection."
    let request = UNNotificationRequest(
        identifier: "caterm.sync.failing.\(UUID().uuidString)",
        content: content,
        trigger: nil
    )
    try? await notificationCenter.add(request)
}
```

#### 2.2.6 Manual sync entry — `sync()` wrapper

```swift
public func sync() async throws {
    manualInProgress = true
    defer { manualInProgress = false }
    try await startSync().value
}
```

> The `defer` runs even if `startSync().value` throws, so `manualInProgress` cannot stick.

#### 2.2.7 `handlePeriodicEnabled(false)` cleanup

When the user toggles Background sync OFF:

```swift
// existing v1.2 logic: cancel timer, cancel wake observer
// new: clear failure state so re-enabling starts clean
wasFailing = false
failingSince = nil
lastSyncErrorKind = nil
```

> See §4.5 for rationale: avoids spurious notification suppression when user re-enables.

```swift
private func handleNotifyToggle(_ newValue: Bool) async {
    if !newValue {
        preferences.notifyOnFailureEnabled = false
        return

    let center = UNUserNotificationCenter.current()
    do {
        let granted = try await center.requestAuthorization(options: [.alert, .sound])
```

Inside the existing `TimelineView(.periodic(from: .now, by: 30))`, after the "Last sync:" row, add:

```swift
if accountState != .sessionExpired,

```

```swift
    let formatter = RelativeDateTimeFormatter()
    formatter.unitsStyle = .short

}
```

## 3. Data Flow (per cycle)

### Cycle 1: Cold start with persisted failing history

```
Process launch
  ↓
HostSyncStore.init hydrate from UserDefaults:
  lastSyncedAt        = T0   (1 day ago)
  lastSyncAttemptedAt = T1   (15 min ago, > T0 ⇒ persisted failing)
  lastSyncErrorKind   = nil  (not persisted)
  failingSince        = nil  (not persisted)
  wasFailing          = false (init default)
  ↓
SyncSettingsView TimelineView calls syncFailureState(...):
  attempted > succeeded ✓
  now - succeeded > 15min ✓
  ⇒ .failing(reason: .other, since: T1)
  → red text appears (since uses store.failingSince ?? attempted = T1)
  ↓
Launch sync trigger fires performSync():
  attempted := now (T2)
  ┌─ If success: lastSyncedAt=T2, errorKind=nil, wasFailing→false, failingSince=nil
  │              → red text disappears, no notification (silent recovery)
  └─ If failure: errorKind=.other
                 nowFailing = isCurrentlyFailing() → true
                 !wasFailing(false→¬=true) && nowFailing(true) ✓
                 → failingSince = T2
                 → if !manualInProgress(launch trigger is auto, so true) && notifyEnabled ⇒ fire
                 → wasFailing = true
```

> **Result**: cold-start with stale failing state, then another auto-failure ⇒ user gets a notification. This is intentional — restarting the machine and finding sync still broken deserves a heads-up.

### Cycle 2: Auto-fail (normal → failing edge)

```
T0:    lastSyncedAt = T0, attempted = T0, errorKind = nil, wasFailing = false, failingSince = nil
T0+15: Timer.publish → scheduleAutoSync() → auth gate ✓ → performSync()
       attempted = T0+15
       client.listHosts throws NetworkError
       catch:
         errorKind = .other
         nowFailing = true
         !wasFailing(true) ✓ → failingSince = T0+15
         !manualInProgress(true) && notifyEnabled(assume true) ⇒ fire notification
         wasFailing = true
       SyncSettingsView next 30s tick:
         syncFailureState → .failing(.other, since: T0+15)
         red text: "Sync failing since 0 minutes ago" (uses store.failingSince = T0+15)
```

### Cycle 3: Consecutive auto-fail (no duplicate notification)

```
T0+15: wasFailing = true, failingSince = T0+15
T0+30: Timer → performSync() → fails again
       attempted = T0+30
       errorKind = .other
       nowFailing = true
       !wasFailing(false) ⇒ edge does not fire ✓
       wasFailing = true (maintained)
       failingSince unchanged (= T0+15)  ← view shows "Sync failing since 15 minutes ago"
```

### Cycle 4: Auto-recovery (failing → normal, silent)

```
wasFailing = true, failingSince = T0+15, errorKind = .other
T0+45: Timer → performSync() → success
       attempted = T0+45, succeeded = T0+45
       errorKind = nil
       wasFailing = false
       failingSince = nil
       (no notification — silent)
       SyncSettingsView next tick:
         syncFailureState → .normal
         red text disappears, "Last sync: just now"
```

### Cycle 5: Manual fail (user clicks Sync Now)

```
T0: state normal
User clicks Sync Now → manualInProgress = true (via sync() wrapper)
performSync() → attempted = now → throw
catch:
  errorKind = .other
  nowFailing = true
  !wasFailing(true) ✓ → failingSince = now
  !manualInProgress ⇒ FALSE ⇒ does NOT fire notification ✓
  wasFailing = true
defer: manualInProgress = false

Result: red text appears (state machine intact), no OS notification
        (the user already saw the failure inline; suppress redundant alert)
```

### Cycle 6: Auth-fail priority

```
T0: normal
Auto Timer → performSync() → AuthSession session expired (cookie present, server 401)
attempted = now
errorKind = .auth
nowFailing → true (reason = .auth)
fire notification (if notifyEnabled)
wasFailing = true

SyncSettingsView render priority:
  if accountState == .sessionExpired:
    show "Sign In Again…" CTA (existing v1.2 banner; takes precedence)
  else if syncFailureState == .failing:
    show v1.3 red text
  else: normal

The v1.3 red text is hidden under the CTA → no UI duplication ✓
Subsequent auto-cycles get gated by scheduleAutoSync's auth gate (AuthSession.isSignedIn = false) → no notification spam
```

### Edge matrix

| from \ to | normal | failing(.other) | failing(.auth) |
|-----------|--------|-----------------|----------------|
| **normal** | — (no-op) | auto: notify if enabled; manual: silent. Red text appears | same; view shows .sessionExpired CTA instead |
| **failing(.other)** | silent. Red text disappears | no edge. Red text persists (since unchanged) | reason changes but `wasFailing` stays → no re-fire; UI swaps to CTA |
| **failing(.auth)** | silent | UI swaps from CTA back to v1.3 red text | no edge |

## 4. Error Handling & Edge Cases

### 4.1 UNUserNotificationCenter permission flow

```
User toggles "Notify when sync fails" ON
  ↓
handleNotifyToggle(true) async:
  let granted = try await center.requestAuthorization(options: [.alert, .sound])
  if granted: preferences.notifyOnFailureEnabled = true
  else:       preferences.notifyOnFailureEnabled = false  ← Toggle UI bounces back
```

**Branches**:
- `requestAuthorization` throws (rare — unsigned bundle / missing entitlement) → catch → revert toggle to false; **do not propagate to caller**.
- User previously denied → `requestAuthorization` returns `granted = false` immediately (no system prompt). Same as deny branch.
- We do **not** check `getNotificationSettings()` ahead of time, and do **not** redirect users to System Settings if denied (v1.3 keeps it minimal; Apple-style "open settings" guidance can come in v1.4 if needed).

### 4.2 Notification delivery failure

```swift
try? await notificationCenter.add(request)   // silently swallowed
```

- Red text remains visible (minimum guarantee).
- Each edge uses a fresh UUID identifier — no folding/coalescing logic in v1.3.
- `trigger: nil` ⇒ immediate delivery; no scheduled timers.

### 4.3 UserDefaults missing

| Key | Missing meaning | Hydrate result |
|-----|-----------------|----------------|
| `catermNotifyOnFailureEnabled` | User never set | `false` (opt-in default) |
| `catermLastSyncAttemptedAt` | Never attempted (fresh install) | `nil` → `syncFailureState` short-circuits to `.normal` |

`SyncErrorKind` is not persisted; on cold start with persisted failing inputs, the first `syncFailureState` evaluation uses `reason = .other` (fallback). This is acceptable approximation — the next sync cycle refreshes the kind, and red text wording does not vary by kind in v1.3.

### 4.4 Threshold strictness

`now.timeIntervalSince(succeeded) > failingThreshold` uses strict `>`. Timer scheduling has sub-second jitter, so equality is unreachable in practice. Strict comparison avoids edge-case oscillation if the implementation is later changed to `>=`.

### 4.5 User toggles `periodicSyncEnabled` OFF mid-failing

```
State: wasFailing=true, failingSince=T0
User toggles Background sync OFF
  → preferences.periodicSyncEnabled = false (didSet)
  → handlePeriodicEnabled(false): cancel timer + wake observer (v1.2)
  → NEW v1.3: wasFailing = false; failingSince = nil; lastSyncErrorKind = nil

SyncSettingsView next tick:
  syncFailureState → .normal (periodicSyncEnabled false short-circuits)
  red text disappears ✓

User re-toggles ON later:
  Timer restarts. Next cycle fails:
    isCurrentlyFailing() → true
    !wasFailing(true after reset) ✓ → notification fires (correct: this is a fresh edge)
```

> **Why clean state on toggle off**: makes the state machine round-trip cleanly. If we kept `wasFailing=true` across the disable/enable cycle, the user's first post-enable failure would silently miss the notification — surprising behavior.

### 4.6 Cold-start persisted failing (rationale)

`wasFailing` init = `false`. When the user starts the app and the next sync cycle fails, the edge `!wasFailing && nowFailing` fires a notification. This is intentional: restarting and discovering "still broken" is exactly when a heads-up is helpful. The notification body ("Click Sync Now in Sync Settings, or check your connection") provides context.

Auth failures are absorbed by the existing `.sessionExpired` CTA (cookie still present but server-side session expired); subsequent cycles get gated by the auth check before they can spam notifications.

### 4.7 Notification suppression — full enumeration

| Scenario | Reason |
|----------|--------|
| `notifyOnFailureEnabled = false` | Toggle OFF |
| `manualInProgress = true` | User already sees the manual failure |
| `wasFailing && nowFailing` | No edge transition |
| `!nowFailing` | Not in failing state (incl. recovery) |
| `periodicSyncEnabled = false` | Short-circuits to `.normal`, `nowFailing` always false |
| Permission denied | `UNUserNotificationCenter.add(...)` rejects (and toggle has been reverted to false anyway) |

## 10. Locked Decisions

1. **`since` semantics**: store-level `failingSince` = first failure of a run (not most recent attempt). Reset on recovery and on `periodicSyncEnabled = false`.
2. **`failingSince` not persisted**: cold-start re-derives via attempted/succeeded comparison; reduces state-machine surface area.
3. **`notifyOnFailureEnabled` default OFF**: HIG-aligned opt-in. UNUserNotification permission requested only on toggle-on.
4. **`failingThreshold` = `periodicInterval`** (15 min in v1.3). Auto-links to v1.4 interval picker if added.
5. **`>` strict comparison** in threshold check (Timer jitter makes equality unreachable; explicit choice).
6. **Permission deny silently reverts toggle**: no Apple-style System Settings redirect in v1.3.
7. **Notification UUID per edge**: no folding/coalescing logic.
8. **`handlePeriodicEnabled(false)` clears `wasFailing`/`failingSince`/`lastSyncErrorKind`**: clean round-trip on disable/enable.
9. **`lastSyncErrorKind` not persisted**: refreshed on next cycle; `.other` fallback on cold start is acceptable (UI text doesn't vary by kind in v1.3).
10. **No new SyncSettingsView tests**: pure-fn coverage + store-level coverage suffice; YAGNI.
11. **Manual sync updates state but suppresses notification**: state machine unifies auto and manual paths; only notification path differs.
12. **Auth failure precedence preserved**: existing v1.2 `.sessionExpired` CTA outranks v1.3 red text in view layer.
