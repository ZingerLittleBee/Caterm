# v1.3 Sync Failure Visibility Design

**Date**: 2026-04-28
**Branch**: `feature/phase-1-v1`
**Status**: Spec, ready for plan (revised per code review — P1+P2 addressed)
**Builds on**: v1.2 (`docs/superpowers/specs/2026-04-28-task-v1.2-periodic-sync-and-freshness-design.md`)

## 0. Goal & Non-goals

**Goal**: Make persistent auto-sync failure **visible** so the user does not need to manually press "Sync Now" to discover the sync layer has been silently broken for 15+ minutes.

**Non-goals (deferred)**:
- Sidebar / menu-bar status indicator (deferred to v1.4)
- Configurable failing threshold (defaults to `periodicInterval`)
- Per-host failure surface (host-level errors are out of scope; only sync-cycle errors)
- Refresh token / silent re-auth (deferred per v1.2 §8)
- Network reachability gating via `NWPathMonitor` (one failed `URLSessionDataTask` is cheaper than polling reachability)

## 1. Architecture

Three coordinated changes, no new module:

```
SyncPreferences                HostSyncStore                       SyncSettingsView
─────────────                  ────────────                        ────────────────
+ notifyOnFailureEnabled       + lastSyncAttemptedAt (@Published)  + Toggle "Notify when sync fails"
  (UserDefaults key:           + lastSyncErrorKind   (@Published)  + Red failure text + "since" time
   catermNotifyOnFailureEnabled,    (.auth / .other)              + If toggle OFF, show
   default false)              + failingSince        (@Published)    "Turn on 'Notify…'" hint
                               + wasFailing (private, transient)   + accountState extended to
                               + init param: notificationCenter      consult store.lastSyncErrorKind
                               + syncFailureState(now:…) free fn     (auto-only 401 → .sessionExpired)
                               + performSync wrap in do/catch
                               + UNUserNotificationCenter delivery
                               (manualInProgress already exists
                                from v2.10 — only READ in v1.3)
```

### Data flow (one failing auto-cycle)

1. `Timer.publish` / wake / mutation-debounce → `scheduleAutoSync()` → auth gate ✓ → `performSync()` starts
2. `performSync()` head writes `lastSyncAttemptedAt = Date()`
3. Op loop `createHost` throws → catch path writes `lastSyncErrorKind = .other` (401 → `.auth`); does **not** update `lastSyncedAt` (v1.2 carve-out preserved)
4. Catch path: `let nowFailing = isCurrentlyFailing()`; if `!wasFailing && nowFailing && !manualInProgress && preferences.notifyOnFailureEnabled` → fire `UNUserNotification`; if `!wasFailing && nowFailing` also writes `failingSince = lastSyncAttemptedAt`
5. Writes `wasFailing = nowFailing`, then rethrows
6. `SyncSettingsView`'s `TimelineView(.periodic by: 30)` re-renders: `syncFailureState(...)` returns `.failing` → red text appears with `failingSince` time

### Invariants

- **Manual sync does not trigger notification** (spec §3 Cycle 5 — manual failure is already visible to the user; gated via `manualInProgress`, which v1.3 *reads* — the flip itself is owned by the existing v2.10 `sync()` Task).
- **Manual sync still updates `lastSyncAttemptedAt`** (consistency: the user-visible "last attempted" must be the actual most recent attempt). Manual `sync()` flows through the same `performSync()` body, so the head-of-perform write covers both paths.
- **Auth-priority**: existing v1.2 `accountState == .sessionExpired` CTA takes precedence over `.failing` red text. Precedence: `.sessionExpired` > `.failing` > `.normal`. v1.3 extends `accountState(...)` to ALSO derive `.sessionExpired` from `store.lastSyncErrorKind == .auth` (not only the manual-path `lastSyncError @State`) so auto-only 401 also surfaces as the CTA. See §2.3 and Decision #3.
- **`periodicSyncEnabled = false`** ⇒ `syncFailureState` returns `.normal` (user-paused sync should not nag; consistent with spec §3.2).
- **`lastSyncedAt == nil`** ⇒ `.normal` (a fresh-install user that has never had a successful sync is in the initial state, not "failing").
- **`failingSince` semantics**: timestamp of *first* entering failing state during a run, not the most recent attempt. Reset to `nil` on recovery. Not persisted — cold-start displays `lastSyncAttemptedAt` as a best-effort fallback (the truly-first failure of the original run is lost across restarts; acceptable approximation, see Decision #2).
- **TimelineView mandatory for failing-since label** (parallel to v1.2 §3.3): `formatFailingSince(...)` resolves relative against `Date()` at body-eval time. Without `TimelineView(.periodic(by: 30))` the phrase freezes when `failingSince` doesn't advance — exactly the failure case we want users to notice. The new red-text `Text(...)` MUST live inside the existing v1.2 `TimelineView` block; see §2.3.

#### 2.2.4 Pure derivation (free function, not on `@MainActor`, no `self`)

Place at file top:

```swift
public func syncFailureState(
    now: Date,
    lastSyncedAt: Date?,
    lastSyncAttemptedAt: Date?,
    lastSyncErrorKind: SyncErrorKind?,
    periodicSyncEnabled: Bool,
    failingThreshold: TimeInterval
) -> SyncFailureState {
    guard periodicSyncEnabled else { return .normal }
    guard let attempted = lastSyncAttemptedAt else { return .normal }
    guard let succeeded = lastSyncedAt else { return .normal }
    guard attempted > succeeded else { return .normal }
    guard now.timeIntervalSince(succeeded) > failingThreshold else { return .normal }
    let reason = lastSyncErrorKind ?? .other
    return .failing(reason: reason, since: attempted)
}
```

> Note: `since` is the *most recent attempted* timestamp. The store-level `failingSince` (the **first** failure of a run) is supplied to the view layer separately; the pure function operates on inputs alone (testable in isolation). The view derives display text from `store.failingSince ?? attempted` where `attempted` is bound out of the `.failing(reason:since:)` payload (see §2.3).

#### 2.2.5 `performSync()` modifications

The existing `performSync()` (HostSyncStore.swift:188-202) has no do/catch — exceptions just propagate. v1.3 wraps the body in `do/catch` so the catch path can record failure state. Final shape:

```swift
private func performSync() async throws {
    // Head: record attempt (covers both manual and auto paths).
    let attempted = Date()
    lastSyncAttemptedAt = attempted
    userDefaults.set(attempted, forKey: Self.lastSyncAttemptedAtKey)

    do {
        let remote = try await client.listHosts()
        try Task.checkCancellation()
        let ops = HostSyncReconciler.reconcile(local: sessionStore.hosts,
                                                remote: remote)
        for op in ops {
            try Task.checkCancellation()
            try await apply(op)
        }
        // Success — only after op loop finishes without throwing.
        let now = Date()
        lastSyncedAt = now
        userDefaults.set(now, forKey: Self.lastSyncedAtKey)
        lastSyncErrorKind = nil
        wasFailing = false
        failingSince = nil
    } catch is CancellationError {
        // Chained cancel-and-drain replaces this Task with a newer one;
        // treat as benign re-arm, NOT a failure. Don't classify, don't
        // fire notification, don't update wasFailing.
        throw CancellationError()
    } catch {
        lastSyncErrorKind = classifySyncError(error)
        let nowFailing = isCurrentlyFailing()
        if !wasFailing && nowFailing {
            failingSince = lastSyncAttemptedAt
            if !manualInProgress && preferences.notifyOnFailureEnabled {
                await fireFailureNotification()
            }
        }
        wasFailing = nowFailing
        throw error
    }
}
```

> **`CancellationError` carve-out**: `startSync()` cancels the previous in-flight Task and drains it (HostSyncStore.swift:173-184). The drained task can surface `CancellationError` from any of three sites: `try await client.listHosts()` (URLSession task cancelled mid-flight), `try Task.checkCancellation()` (between op-loop iterations or after `await prev?.result`), or `try await apply(op)` (any nested cancellation). The carve-out at the catch site covers all three because it's a type-pattern match, not a site-specific check. Without the carve-out, all three sites would be classified `.other` and trip the failure edge.
> **Head-write order side effect**: `lastSyncAttemptedAt` is written *before* the do/catch (so the catch path has a value to use as `failingSince`). On a cancelled-replace path the persisted attempt timestamp may briefly reflect a Task that didn't actually run to completion — the *replacing* Task overwrites it within milliseconds. Acceptable behavior; the UI's "Last attempt" semantic is approximate at sub-second resolution either way.
> **`lastSyncedAt` invariant preserved (v1.2 §4.2)**: success-path writes are still gated by the for-loop completing — partial-apply still leaves `lastSyncedAt` untouched. The new success-path writes (`lastSyncErrorKind = nil`, `wasFailing = false`, `failingSince = nil`) are colocated with the existing `lastSyncedAt = now`, so they share the same all-or-nothing gating.

Helpers:
```swift
private func classifySyncError(_ error: Error) -> SyncErrorKind {
    guard let err = error as? ServerSyncError else { return .other }
    switch err {
    case .http(status: 401, _):    return .auth
    case .orpc(_, status: 401, _): return .auth   // oRPC envelope wraps 401, NOT .http
    case .authFailed:              return .auth
    case .notSignedIn:             return .auth
    default:                       return .other
    }
}

private func isCurrentlyFailing() -> Bool {
    if case .failing = syncFailureState(
        now: Date(),
        lastSyncedAt: lastSyncedAt,
        lastSyncAttemptedAt: lastSyncAttemptedAt,
        lastSyncErrorKind: lastSyncErrorKind,
        periodicSyncEnabled: preferences.periodicSyncEnabled,
        failingThreshold: periodicInterval
    ) {
        return true
    }
    return false
}

private func fireFailureNotification() async {
    let content = UNMutableNotificationContent()
    content.title = "Caterm sync is failing"
    content.body = "Click Sync Now in Sync Settings, or check your connection."
    let request = UNNotificationRequest(
        identifier: "caterm.sync.failing.\(UUID().uuidString)",
        content: content,
        trigger: nil
    )
    try? await notificationCenter.add(request)
}
```

> **Why `classifySyncError` duplicates the view-layer `isAuthFailure`**: lifting the view's private predicate to a shared module is a larger refactor; v1.3 keeps both inline. Decision #5 pins the lockstep requirement.

#### 2.2.6 Manual sync entry — `sync()` is unchanged

The existing `sync()` (HostSyncStore.swift:89-108) already:
- Sets `manualInProgress = true` at the start of its inner Task body (line 95).
- Clears it in `defer` (line 97).
- Owns `currentManualTask` dedup and `pendingAutoAfterManual` replay (lines 96-102).

#### 2.2.7 `handlePeriodicEnabled(false)` cleanup

The existing `handlePeriodicEnabled(_:)` (HostSyncStore.swift:142-151) cancels and recreates the timer subscription. v1.3 adds three assignments inside the **`enabled == false`** branch:

```swift
private func handlePeriodicEnabled(_ enabled: Bool) {
    periodicTimerCancellable?.cancel()
    periodicTimerCancellable = nil
    guard enabled else {
        // v1.3 add — clear failure state so re-enable starts clean.
        wasFailing = false
        failingSince = nil
        lastSyncErrorKind = nil
        return
    }
    periodicTimerCancellable = Timer.publish(every: periodicInterval,
                                              on: .main,
                                              in: .common)
        .autoconnect()
        .sink { [weak self] _ in self?.scheduleAutoSync() }
}
```

> **Why inside the `else { return }` branch**: the sink fires synchronously on init with `enabled = true`, then on every value change. Clearing on `enabled = true` would erase legitimate state. See §4.5 for rationale on why clearing on toggle-off makes the round-trip clean.

#### 2.3.1 Extend `accountState(...)` to derive `.sessionExpired` from auto-only auth failures

The existing free function (SyncSettingsView.swift:18-22) only sees `lastSyncError` — a `@State` populated exclusively by the manual `syncNow()` catch path (line 132). An auto-only 401 never sets this state, so without the extension, auto-only auth failures would show as v1.3 red text *instead of* the `.sessionExpired` CTA, contradicting the auth-priority invariant.

v1.3 extends the signature to also take `lastSyncErrorKind` (the store-side surface, written by both auto and manual paths via `classifySyncError`):

```swift
func accountState(isSignedIn: Bool,
                  lastSyncError: ServerSyncError?,
                  lastSyncErrorKind: SyncErrorKind?) -> AccountState {
    if isAuthFailure(lastSyncError) || lastSyncErrorKind == .auth { return .sessionExpired }
    return isSignedIn ? .signedIn : .signedOut
}
```

> **Two call sites must be updated together** (transcription hazard):
> 1. **Account section switch** (SyncSettingsView.swift:65) — must pass `lastSyncErrorKind: syncStore.lastSyncErrorKind`.
> 2. **Sync section TimelineView guard** (new, §2.3.3) — must compute `derivedAccountState` with the same arg.
>
> Updating only one will desync the auth-priority surface (one place renders the CTA, the other renders red text simultaneously). Existing tests in `SyncSettingsAccountStateTests` need their call sites updated to pass `lastSyncErrorKind: nil` (back-compat). One new test added for the auto-path (see §5.5).

#### 2.3.3 Failing red text — inside the existing TimelineView

Inside the existing `TimelineView(.periodic(from: .now, by: 30))` block (SyncSettingsView.swift:100-103), **after** the `Text("Last sync: …")` row. Co-location is mandatory (see §1 invariant — `formatFailingSince` resolves relative against `Date()` at body-eval time):

```swift
let failure = currentFailureState(now: context.date)
let derivedAccountState = accountState(
    isSignedIn: authSession.isSignedIn,
    lastSyncError: lastSyncError,
    lastSyncErrorKind: syncStore.lastSyncErrorKind
)
if derivedAccountState != .sessionExpired,
   case let .failing(_, attempted) = failure {
    let displaySince = syncStore.failingSince ?? attempted
    Text("Sync failing since \(formatFailingSince(displaySince, now: context.date))")
        .foregroundColor(.red)
        .font(.caption)
    if !preferences.notifyOnFailureEnabled {
        Text("Turn on 'Notify when sync fails' to be alerted next time.")
            .foregroundColor(.secondary)
            .font(.caption2)
    }
}
```

> **Fallback `?? attempted`, NOT `?? Date()`**: `attempted` (bound out of the `.failing(_, attempted)` payload, which the pure fn populates from `lastSyncAttemptedAt`) is the cold-start best-effort source for the truly-first failure timestamp. `Date()` would always read as "0 minutes ago" because the formatter's `relativeTo` would be the same instant.

## 3. Data Flow (per cycle)

### Cycle 1: Cold start with persisted failing history

```
Process launch
  ↓
HostSyncStore.init hydrate from UserDefaults:
  lastSyncedAt        = T0   (1 day ago)
  lastSyncAttemptedAt = T1   (15 min ago, > T0 ⇒ persisted failing)
  lastSyncErrorKind   = nil  (not persisted)
  failingSince        = nil  (not persisted)
  wasFailing          = false (init default)
  ↓
SyncSettingsView TimelineView calls syncFailureState(...):
  attempted > succeeded ✓
  now - succeeded > 15min ✓
  ⇒ .failing(reason: .other, since: T1)
  → red text appears (since uses store.failingSince ?? attempted = T1)
  ↓
Launch sync trigger fires performSync():
  attempted := now (T2)
  ┌─ If success: lastSyncedAt=T2, errorKind=nil, wasFailing→false, failingSince=nil
  │              → red text disappears, no notification (silent recovery)
  └─ If failure: errorKind=.other
                 nowFailing = isCurrentlyFailing() → true
                 !wasFailing(false→¬=true) && nowFailing(true) ✓
                 → failingSince = T2
                 → if !manualInProgress(launch trigger is auto, so true) && notifyEnabled ⇒ fire
                 → wasFailing = true
```

> **Result**: cold-start with stale failing state, then another auto-failure ⇒ user gets a notification. This is intentional — restarting the machine and finding sync still broken deserves a heads-up.

### Cycle 2: Auto-fail (normal → failing edge)

```
T0:    lastSyncedAt = T0, attempted = T0, errorKind = nil, wasFailing = false, failingSince = nil
T0+15: Timer.publish → scheduleAutoSync() → auth gate ✓ → performSync()
       attempted = T0+15
       client.listHosts throws NetworkError
       catch:
         errorKind = .other
         nowFailing = true
         !wasFailing(true) ✓ → failingSince = T0+15
         !manualInProgress(true) && notifyEnabled(assume true) ⇒ fire notification
         wasFailing = true
       SyncSettingsView next 30s tick:
         syncFailureState → .failing(.other, since: T0+15)
         red text: "Sync failing since 0 minutes ago" (uses store.failingSince = T0+15)
```

### Cycle 3: Consecutive auto-fail (no duplicate notification)

```
T0+15: wasFailing = true, failingSince = T0+15
T0+30: Timer → performSync() → fails again
       attempted = T0+30
       errorKind = .other
       nowFailing = true
       !wasFailing(false) ⇒ edge does not fire ✓
       wasFailing = true (maintained)
       failingSince unchanged (= T0+15)  ← view shows "Sync failing since 15 minutes ago"
```

### Cycle 4: Auto-recovery (failing → normal, silent)

```
wasFailing = true, failingSince = T0+15, errorKind = .other
T0+45: Timer → performSync() → success
       attempted = T0+45, succeeded = T0+45
       errorKind = nil
       wasFailing = false
       failingSince = nil
       (no notification — silent)
       SyncSettingsView next tick:
         syncFailureState → .normal
         red text disappears, "Last sync: just now"
```

### Cycle 5: Manual fail (user clicks Sync Now)

```
T0: state normal
User clicks Sync Now → manualInProgress = true (set inside the existing v2.10 sync() Task body)
performSync() → attempted = now → throw
catch:
  errorKind = .other
  nowFailing = true
  !wasFailing(true) ✓ → failingSince = now
  !manualInProgress ⇒ FALSE ⇒ does NOT fire notification ✓
  wasFailing = true
defer: manualInProgress = false

Result: red text appears (state machine intact), no OS notification
        (the user already saw the failure inline; suppress redundant alert)
```

### Cycle 6: Auth-fail priority (auto path)

```
T0: normal
Auto Timer → performSync() → 401 (cookie present, server expired session)
attempted = now
errorKind = .auth                 ← classifySyncError matches .http(401, _) / .orpc(_, 401, _)
nowFailing → true (reason = .auth)
fire notification (if notifyEnabled, !manualInProgress)
wasFailing = true

SyncSettingsView render priority (extended accountState):
  derivedAccountState = accountState(
      isSignedIn: true,
      lastSyncError: nil,                          ← NOT set on auto path
      lastSyncErrorKind: syncStore.lastSyncErrorKind = .auth   ← v1.3 extension fires here
  ) → .sessionExpired
  → "Sign In Again…" CTA in Account section (v1.2 banner)
  → v1.3 red text guard `derivedAccountState != .sessionExpired` evaluates false → red text hidden ✓

Subsequent auto-cycles: scheduleAutoSync's auth gate (AuthSession.isSignedIn) — note
that `isSignedIn` is cookie-presence, not server-validity. So a server-expired-cookie-present
state will keep auto-cycling and re-firing 401s. v2.10 §10 disclaims this; v1.3 inherits.
The notification edge gate (`!wasFailing && nowFailing`) prevents notification spam
across these consecutive 401s.
```

> **Manual sync after auto-only auth failure**: when the user clicks "Sign In Again…", `signOut()` clears the cookie → `authSession.isSignedIn = false`. The next `syncNow()` would set `lastSyncError = .notSignedIn`. After successful re-sign-in, the next `performSync()` success clears `lastSyncErrorKind` → `derivedAccountState` returns to `.signedIn`, CTA disappears.

### Edge matrix

| from \ to | normal | failing(.other) | failing(.auth) |
|-----------|--------|-----------------|----------------|
| **normal** | — (no-op) | auto: notify if enabled; manual: silent. Red text appears | same; view shows .sessionExpired CTA instead |
| **failing(.other)** | silent. Red text disappears | no edge. Red text persists (since unchanged) | reason changes but `wasFailing` stays → no re-fire; UI swaps to CTA |
| **failing(.auth)** | silent | UI swaps from CTA back to v1.3 red text | no edge |

## 4. Error Handling & Edge Cases

### 4.1 UNUserNotificationCenter permission flow

```
User toggles "Notify when sync fails" ON
  ↓
handleNotifyToggle(true) async:
  let granted = try await center.requestAuthorization(options: [.alert, .sound])
  if granted: preferences.notifyOnFailureEnabled = true
  else:       preferences.notifyOnFailureEnabled = false  ← Toggle UI bounces back
```

**Branches**:
- `requestAuthorization` throws (unsigned bundle / missing entitlement / SwiftPM CLI shape) → catch → revert toggle to false; **do not propagate to caller**.
- User previously denied → `requestAuthorization` returns `granted = false` immediately (no system prompt). Same as deny branch.
- We do **not** check `getNotificationSettings()` ahead of time, and do **not** redirect users to System Settings if denied (v1.3 keeps it minimal; Apple-style "open settings" guidance can come in v1.4 if needed).

**Dev / SwiftPM-CLI shape**: this app currently builds as a SwiftPM executable (not an `.app` bundle). `UNUserNotificationCenter` may refuse authorization without a proper bundle identifier / `Info.plist`. v1.3 treats this gracefully — the toggle silently reverts to false, and the red text remains the only failure surface. This still solves the original goal (the user *sees* the failure when they open Sync Settings); notifications are an enhancement layered on top. The Phase 1.11 release work (DMG + signed bundle) will unlock notifications without further v1.3 code changes.

**Granted-then-disabled-in-System-Settings drift**: if the user grants permission, then later disables Caterm notifications via System Settings, subsequent `add(...)` calls silently fail (caught by `try?`). The in-app toggle stays `true` even though delivery is disabled. v1.3 does **not** detect this drift — deferred to v1.4 if reported.

### 4.2 Notification delivery failure

```swift
try? await notificationCenter.add(request)   // silently swallowed
```

- Red text remains visible (minimum guarantee).
- Each edge uses a fresh UUID identifier — no folding/coalescing logic in v1.3.
- `trigger: nil` ⇒ immediate delivery; no scheduled timers.

### 4.3 UserDefaults missing

| Key | Missing meaning | Hydrate result |
|-----|-----------------|----------------|
| `catermNotifyOnFailureEnabled` | User never set | `false` (opt-in default) |
| `catermLastSyncAttemptedAt` | Never attempted (fresh install) | `nil` → `syncFailureState` short-circuits to `.normal` |

`SyncErrorKind` is not persisted; on cold start with persisted failing inputs, the first `syncFailureState` evaluation uses `reason = .other` (fallback). This is acceptable approximation — the next sync cycle refreshes the kind, and red text wording does not vary by kind in v1.3.

### 4.4 Threshold strictness

`now.timeIntervalSince(succeeded) > failingThreshold` uses strict `>`. Timer scheduling has sub-second jitter, so equality is unreachable in practice. Strict comparison avoids edge-case oscillation if the implementation is later changed to `>=`.

### 4.5 User toggles `periodicSyncEnabled` OFF mid-failing

```
State: wasFailing=true, failingSince=T0
User toggles Background sync OFF
  → preferences.periodicSyncEnabled = false (didSet)
  → handlePeriodicEnabled(false): cancel timer + wake observer (v1.2)
  → NEW v1.3: wasFailing = false; failingSince = nil; lastSyncErrorKind = nil

SyncSettingsView next tick:
  syncFailureState → .normal (periodicSyncEnabled false short-circuits)
  red text disappears ✓

User re-toggles ON later:
  Timer restarts. Next cycle fails:
    isCurrentlyFailing() → true
    !wasFailing(true after reset) ✓ → notification fires (correct: this is a fresh edge)
```

> **Why clean state on toggle off**: makes the state machine round-trip cleanly. If we kept `wasFailing=true` across the disable/enable cycle, the user's first post-enable failure would silently miss the notification — surprising behavior.

### 4.6 Cold-start persisted failing (rationale)

`wasFailing` init = `false`. When the user starts the app and the next sync cycle fails, the edge `!wasFailing && nowFailing` fires a notification. This is intentional: restarting and discovering "still broken" is exactly when a heads-up is helpful. The notification body ("Click Sync Now in Sync Settings, or check your connection") provides context.

Auth failures are absorbed by the existing `.sessionExpired` CTA (cookie still present but server-side session expired); subsequent cycles get gated by the auth check before they can spam notifications.

### 4.7 Notification suppression — full enumeration

| Scenario | Reason |
|----------|--------|
| `notifyOnFailureEnabled = false` | Toggle OFF |
| `manualInProgress = true` | User already sees the manual failure |
| `wasFailing && nowFailing` | No edge transition |
| `!nowFailing` | Not in failing state (incl. recovery) |
| `periodicSyncEnabled = false` | Short-circuits to `.normal`, `nowFailing` always false |
| Permission denied | `UNUserNotificationCenter.add(...)` rejects (and toggle has been reverted to false anyway) |
| `CancellationError` thrown | Chained cancel-and-drain — benign re-arm, not a failure |

### 4.8 Manual-vs-auto interleaving (race disclaimer)

The `manualInProgress` check in the catch path is racy with respect to manual-click timing relative to an in-flight auto pass. Concrete sequence:

```
T0: auto pass enters performSync()        (manualInProgress = false)
T0: auto pass network call begins
T1: user clicks Sync Now
    → currentManualTask is nil (auto holds inFlight, not currentManualTask)
    → manual Task created, manualInProgress = true
T2: auto pass network call throws
    → catch path runs on @MainActor
    → reads manualInProgress = true (because manual click landed first on the actor)
    → notification suppressed, even though the failure is auto-origin
```

**Acceptable**: when this happens, the user is staring at the Sync Settings sheet (they just clicked Sync Now). Suppressing the OS notification is the correct UX — they'll see the inline manual-failure surface seconds later. The race is benign; no fix in v1.3.

The matching test (`testNotification_notFired_whenManualSync`) does **not** rely on this race — it drives `try? await store.sync()` end-to-end, where `manualInProgress` is set deterministically before the throw inside the same Task. The race exists only when manual click fires *during* an in-flight auto pass; that scenario is not unit-tested.

> **Determinism trace for `testNotification_notFired_whenManualSync`**: `sync()` (HostSyncStore.swift:89-108) runs synchronously on `@MainActor` until the inner `Task`'s `try await startSync().value` (line 104) — i.e. `manualInProgress = true` (line 95) happens *before* any suspension. The catch path inside `performSync` re-acquires `@MainActor` after `client.listHosts()` returns. Both writes are on the main actor, no suspension between them, so `manualInProgress == true` when the catch runs. The `defer` block (lines 96-103) only fires after the inner Task body returns to `try await task.value` (line 107), which is *after* the catch path completes. Same precedent as v2.10's `testManualBlocksAutoFire`.

## 10. Locked Decisions

Ordered by impact (highest first):

1. **`since` semantics — first failure of a run**: store-level `failingSince` = timestamp of *first* entering the failing state during a run, NOT the most-recent attempt. Reset on recovery and on `periodicSyncEnabled = false`. Pinned by `testFailingSince_pinnedAcrossConsecutiveFailures`.

2. **`failingSince` not persisted, cold-start best-effort fallback**: cold-start re-derives nothing — `failingSince` starts `nil`. The view falls back to `attempted` (bound out of the pure fn's `.failing(_, attempted)` payload, populated from persisted `lastSyncAttemptedAt`). This is *not* the truly-first failure of the original run (lost across restarts) — acceptable approximation. The phrase "Sync failing since N minutes ago" reads as "since N minutes ago" of the most recent attempt rather than the original break point. Decision #1's invariant holds *within a single process lifetime*.

3. **Auth failure precedence — store-side derivation extends `accountState`**: `accountState(...)` gains a `lastSyncErrorKind: SyncErrorKind?` parameter. `.sessionExpired` is returned when *either* the manual-path `lastSyncError` is auth-shaped (existing v1.2 behavior) OR the store-side `lastSyncErrorKind == .auth` (v1.3 extension covering auto-only 401s). v1.3 red text guard checks the *extended* result. This keeps the auth surface unified across manual and auto paths.

4. **TimelineView mandatory for failing-since label** (parallel to v1.2 §3.3 `lastSyncedAt`): the new `Text("Sync failing since …")` MUST live inside the existing `TimelineView(.periodic(by: 30))` block. `formatFailingSince` resolves relative against `Date()` at body-eval time; without periodic re-render the phrase freezes when `failingSince` doesn't advance — exactly the failure case we want to surface.

5. **`classifySyncError` inline switch on `ServerSyncError`** (kept in lockstep with view-layer `isAuthFailure`): the existing private `isAuthFailure` predicate in `SyncSettingsView.swift:24-33` is the source of truth for "is this an auth error." v1.3 inlines the same four cases (`.http(401,_)` / `.orpc(_,401,_)` / `.authFailed` / `.notSignedIn`) in `classifySyncError`. If a new auth-shaped case lands in `ServerSyncError`, **both predicates must be updated**. Lifting to a shared module is deferred to a future refactor.

6. **`CancellationError` carve-out in `performSync` catch**: chained cancel-and-drain (HostSyncStore.swift:173-184) throws `CancellationError` from the drained Task. The catch path must NOT classify this as a failure — separate `catch is CancellationError { throw CancellationError() }` clause keeps the failure surface scoped to server / network errors.

7. **`sync()` itself unchanged**: existing v2.10 logic owns `currentManualTask` dedup, `pendingAutoAfterManual` replay, and the `manualInProgress` flip. v1.3 only *reads* `manualInProgress` from the new catch path. Modifying `sync()` would silently break v2.10 invariants pinned by `HostSyncStoreAutoSyncTests`.

8. **Persistence shape uniform with v1.2**: `lastSyncAttemptedAt` round-trips as `Date` directly via `as? Date` / `set(now, ...)` — same as `lastSyncedAt` in v1.2. Do NOT mix in `Double` (timeIntervalSince1970).

9. **`handlePeriodicEnabled(false)` clears `wasFailing`/`failingSince`/`lastSyncErrorKind`**: clean round-trip on disable/enable. The cleanup lives **inside** the `else { return }` branch (NOT before the guard), so re-arming the timer on `enabled = true` doesn't erase legitimate state.

10. **`notifyOnFailureEnabled` default OFF**: HIG-aligned opt-in. UNUserNotification permission requested only on toggle-on. Permission throw / deny silently reverts toggle to false.

11. **Permission failure modes are silent**: throw on `requestAuthorization` (unsigned bundle / SwiftPM CLI shape), user-denial, and post-grant System Settings disable all degrade gracefully. The red text remains the failure surface; v1.3 still solves the original visibility goal even when notifications can't be delivered.

12. **`failingThreshold` = `periodicInterval`** (15 min in v1.3). Auto-links to v1.4 interval picker if added.

13. **`>` strict comparison** in threshold check (Timer jitter makes equality unreachable; explicit choice).

14. **Notification UUID per edge**: no folding/coalescing logic.

15. **`lastSyncErrorKind` not persisted**: refreshed on next cycle; `.other` fallback on cold start is acceptable (UI text doesn't vary by kind in v1.3).

16. **`RelativeDateTimeFormatter.unitsStyle = .full`** for `formatFailingSince`, matching v1.2 `formatLastSyncedAt` for visual consistency.

17. **No view rendering tests**: `accountState` extension is a pure-fn test; v1.3 UI bindings are direct and trust SwiftUI; `handleNotifyToggle` async path is YAGNI.

18. **Manual sync state-vs-notification split**: state machine unifies auto and manual paths (both write `lastSyncAttemptedAt`, `lastSyncErrorKind`, `failingSince`); only notification path differs (manual suppresses).

19. **Manual-vs-auto race acceptable** (§4.8): `manualInProgress` check in catch is racy with respect to manual-click timing during in-flight auto pass. When the race elides a notification, the user is staring at Sync Settings — suppression is correct UX. Not unit-tested.
