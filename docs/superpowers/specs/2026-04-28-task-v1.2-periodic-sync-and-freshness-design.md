# Task v1.2 ‚Äî Periodic Sync + Freshness Display (Design)

**Date:** 2026-04-28
**Author:** brainstorm session continuing Phase 2 v1.1
**Status:** approved (pending user review of this file)
**Scope:** macOS Caterm app
**Phase:** v1.2 (follow-up to 2.10 auto-sync triggers)

## 1. Goal

Today (after 2.10), sync fires on:
- (a) launch (`syncIfSignedIn` from `WindowGroup .task`)
- (b) post-mutation, debounced 2 s (`SessionStore.mutationsForSync` publisher)
- manual ("Sync Now" button)

The remaining freshness gap is **the device that sat idle while another device wrote** ‚Äî neither (a) nor (b) fires for that device until the user actively touches the app. v1.2 closes this with:

- **(c) Periodic background sync** every 15 min while the app is alive and the user is signed in
- **`lastSyncedAt` freshness display** in `SyncSettingsView` so users have a visible signal that auto-sync is working

These are scoped together because periodic sync makes silent failures *more frequent* (more fires = more chances to fail), and the timestamp gives users a way to notice. Without (B), (A) would be invisible to the user. Without (A), (B) would be a nicer no-op.

## 2. Non-goals

- **Configurable interval.** Toggle is binary on/off. No "every 1 / 5 / 15 / 30 / 60 min" picker. Defer to v1.3 if explicit demand surfaces.
- **`lastSyncAttemptedAt` + "failing" badge.** Currently `lastSyncedAt` only updates on success. A user with a persistent 401 sees a stale timestamp but no inline indication "sync is failing." The 2.10.4 `.sessionExpired` Account-state derivation picks up *manual* failures, but auto-only failures stay invisible until the user clicks Sync Now. Deferred ‚Äî keeping v1.2 minimal; revisit if users actually report it.
- **Refresh-token / silent re-auth.** Carried over from 2.10 ¬ß10 unchanged. Investigating better-auth's refresh-cookie support is a v1.3 spike. Today's "Sign In Again‚Ä¶" path remains the recovery surface for 401s; periodic sync makes session expiration *more visible* via stale `lastSyncedAt` ‚Äî that signal alone is the v1.2 contribution.
- **Per-op retry policy.** Same reasoning as 2.10 ¬ß10. Periodic sync at 15 min actually makes this less painful ‚Äî recovery latency is bounded by cadence rather than waiting for user action.
- **Sidebar / menu-bar status indicator.** A persistent "Sync OK / Failing / Disabled" pill always visible. Genuinely useful but a separate UI design conversation. v1.3+.
- **Network reachability gating** (`NWPathMonitor` to skip fires when offline). Cost of a fire over no network is one fast-failed `URLSessionDataTask` ‚Äî cheap enough that polling reachability isn't justified.
- **Battery-state gating.** macOS already throttles background apps' timers under battery duress. Trust the OS.

## 3. Architecture

Three coordinated changes to the existing `HostSyncStore` + a new tiny preferences object.

### 3.1 `SyncPreferences` (new, public, in `HostSyncStore` module)

`@MainActor public final class SyncPreferences: ObservableObject` with:

```swift
@Published public var periodicSyncEnabled: Bool
private let defaults: UserDefaults
private static let periodicEnabledKey = "catermPeriodicSyncEnabled"

public init(defaults: UserDefaults = .standard) {
    self.defaults = defaults
    let stored = defaults.object(forKey: Self.periodicEnabledKey) as? Bool
    self.periodicSyncEnabled = stored ?? true   // default true when key missing
}
```

Plus a `didSet` (or explicit setter) that writes back to `defaults`:

```swift
@Published public var periodicSyncEnabled: Bool {
    didSet {
        defaults.set(periodicSyncEnabled, forKey: Self.periodicEnabledKey)
    }
}
```

Single source of truth for the toggle. Initialized once in `CatermApp.init` and threaded into `HostSyncStore.init` (matches existing `AuthSession` / `SessionStore` injection pattern). The `defaults` parameter exists for test isolation (each test passes a fresh `UserDefaults(suiteName: "test-\(UUID())")`).

**Module choice:** lives in the existing `HostSyncStore` SwiftPM target. `HostSyncStore.init` is the primary consumer (co-locating avoids circular-dep risk); `Caterm` already depends on `HostSyncStore` (so the view can `import HostSyncStore` and use `SyncPreferences`); tests in `HostSyncStoreTests` can use it via `@testable import HostSyncStore` without crossing target boundaries. No new SwiftPM target.

### 3.2 `HostSyncStore` extends with periodic + wake + freshness state

New stored state:
```swift
private let preferences: SyncPreferences
private let userDefaults: UserDefaults
private let periodicInterval: TimeInterval
private var periodicTimerCancellable: AnyCancellable?
@Published public private(set) var lastSyncedAt: Date?

private static let lastSyncedAtKey = "catermLastSyncedAt"
```

`init` signature gains three parameters (existing 2.10.3a parameters preserved):

```swift
public init(
    client: ServerSyncClient,
    sessionStore: SessionStore,
    authSession: AuthSessionProtocol,
    preferences: SyncPreferences,                          // new
    debounceInterval: TimeInterval = 2.0,
    periodicInterval: TimeInterval = 15 * 60,              // new ‚Äî 900 s production
    userDefaults: UserDefaults = .standard                 // new ‚Äî for lastSyncedAt persistence
)
```

`init` body adds three subscriptions on top of the existing 2.10.3b mutationsForSync debounce:

```swift
// 1. Hydrate lastSyncedAt from persistence
self.lastSyncedAt = userDefaults.object(forKey: Self.lastSyncedAtKey) as? Date

// 2. Track preferences.periodicSyncEnabled ‚Äî start/stop the periodic timer
preferences.$periodicSyncEnabled
    .sink { [weak self] enabled in self?.handlePeriodicEnabled(enabled) }
    .store(in: &cancellables)

// 3. Listen for system wake (laptop opened from sleep)
NSWorkspace.shared.notificationCenter
    .publisher(for: NSWorkspace.didWakeNotification)
    .sink { [weak self] _ in self?.handleSystemWake() }
    .store(in: &cancellables)
```

Note: `NSWorkspace` notifications post to `NSWorkspace.shared.notificationCenter`, **not** `NotificationCenter.default`. Important detail ‚Äî using the wrong center silently misses every wake event.

`handlePeriodicEnabled` starts/stops the timer:

```swift
private func handlePeriodicEnabled(_ enabled: Bool) {
    periodicTimerCancellable?.cancel()
    periodicTimerCancellable = nil
    guard enabled else { return }
    periodicTimerCancellable = Timer.publish(every: periodicInterval,
                                             on: .main,
                                             in: .common)
        .autoconnect()
        .sink { [weak self] _ in self?.scheduleAutoSync() }
}
```

`@Published`'s `.sink` fires immediately with the current value ‚Äî so on `init`, this synchronously invokes `handlePeriodicEnabled(true)` (the default), starting the timer.

`handleSystemWake` fires an immediate sync and re-arms the timer (so the next periodic fire is 15 min from wake, not 15 min from the pre-sleep schedule):

```swift
private func handleSystemWake() {
    scheduleAutoSync()
    if preferences.periodicSyncEnabled {
        handlePeriodicEnabled(true)  // re-create the timer with a fresh schedule
    }
}
```

Sleep notification is deliberately ignored ‚Äî `Timer.publish` naturally pauses across sleep (the runloop is suspended), and any spurious post-wake double-fire is absorbed by the existing 2.10.3c chained cancel-and-drain.

`performSync` gains the `lastSyncedAt` update at the end of the success path:

```swift
private func performSync() async throws {
    let remote = try await client.listHosts()
    try Task.checkCancellation()
    let ops = HostSyncReconciler.reconcile(local: sessionStore.hosts, remote: remote)
    for op in ops {
        try Task.checkCancellation()
        try await apply(op)
    }
    let now = Date()
    lastSyncedAt = now
    userDefaults.set(now, forKey: Self.lastSyncedAtKey)
}
```

Critically: the update is **after** the op loop, so partial application (one op throws) does NOT update `lastSyncedAt`. See ¬ß4.2 for the precise semantics.

### 3.3 `SyncSettingsView` adds two rows

In the existing Sync section, add a Toggle bound to `preferences.periodicSyncEnabled`:

```swift
Section("Sync") {
    Toggle("Periodic sync (every 15 minutes)",
           isOn: $preferences.periodicSyncEnabled)
    Button("Sync Now") { Task { await syncNow() } }
        .disabled(!authSession.isSignedIn || isSyncing)
    Text("Last sync: \(formatLastSyncedAt(syncStore.lastSyncedAt))")
        .font(.caption).foregroundColor(.secondary)
    if let lastSyncError {
        Text(lastSyncError.description).foregroundColor(.red).font(.caption)
    }
}
```

`formatLastSyncedAt` is a free function (file-internal scope, like `accountState` from 2.10.4), tested directly:

```swift
func formatLastSyncedAt(_ date: Date?) -> String {
    guard let date else { return "Never synced" }
    let formatter = RelativeDateTimeFormatter()
    formatter.unitsStyle = .full
    return formatter.localizedString(for: date, relativeTo: Date())
}
```

`RelativeDateTimeFormatter` automatically produces "2 minutes ago" / "1 hour ago" / "yesterday" / "Apr 27" ‚Äî system-localized.

The view declares both `preferences` AND `syncStore` as `@ObservedObject`:

```swift
@ObservedObject var preferences: SyncPreferences
@ObservedObject var syncStore: HostSyncStore
```

(Not `@StateObject` ‚Äî the view does not own the lifetime; `CatermApp` does.)

**Why `syncStore` changes from `let` to `@ObservedObject`:** the existing 2.10.4/2.10.5 code declares `let syncStore: HostSyncStore`. That worked because the view didn't read any `@Published` property of it ‚Äî every observable state was a local `@State`. v1.2 introduces `syncStore.lastSyncedAt` (a `@Published` property) into the view body, so the view must observe `syncStore` to re-render on changes. The fix is mechanical: `let syncStore: HostSyncStore` ‚Üí `@ObservedObject var syncStore: HostSyncStore`. Caller code (the sheet constructor in `CatermApp`) is unchanged because the constructor argument's type is identical.

### 3.5 Load-bearing invariant from 2.10 still holds

Periodic and wake fires both go through `scheduleAutoSync()` ‚Üí `startSync()` ‚Üí the chained cancel-and-drain. They get serialized identically to mutation-debounce fires. **No new code paths bypass the chain.** This means every concurrency property pinned by the 2.10.3c/d tests automatically applies to v1.2's new fire sources.

## 4. Error handling + invariants

### 4.1 Errors in periodic / wake fires

Behave identically to mutation-debounced fires (2.10 ¬ß6.1): both go through `scheduleAutoSync() ‚Üí startSync()`, the resulting `Task<Void, Error>` is stored in `inFlight` and *never observed*. Failures (offline, 401, 5xx, decode error) sit in the unobserved Result and are dropped when `inFlight` is replaced by the next schedule.

UI stays silent; the only signal is `lastSyncedAt` not advancing. User opens Sync Settings ‚Üí manual "Sync Now" surfaces the error in red text ‚Üí the 2.10.4 `.sessionExpired` derivation handles the 401 case. **No new error path introduced by v1.2.**

### 4.2 `lastSyncedAt` update rule (precise)

Inside `performSync()`, immediately after the `for op in ops { try await apply(op) }` loop completes without throwing, set `lastSyncedAt = Date()` and write to UserDefaults. **Do not** update on:

- `client.listHosts()` throwing (we never reached the op loop)
- Any `apply(op)` throwing (partial application ‚Äî last sync was *not* fully applied)
- `Task.checkCancellation()` throwing (chain replaced this pass; the next pass will eventually update)
- Cancel-during-drain (same logic)

This means `lastSyncedAt` reflects "last fully-applied sync", which is the most honest signal a user can observe. A long string of partial-failure passes will leave `lastSyncedAt` stale even though `listHosts` is succeeding ‚Äî that's the right behavior: the user sees "last sync 2 hours ago", clicks Sync Now, sees the actual error.

### 4.3 Concurrent fires (timer + wake + mutation + manual)

All coalesced by the existing chained cancel-and-drain (2.10.3c) and the `manualInProgress` / `pendingAutoAfterManual` / `currentManualTask` flags (2.10.3d). Worst case: timer fires, wake fires, debounced mutation fires, user clicks Sync Now ‚Äî all within 1 second. The chain runs at most one `performSync` to completion (manual takes priority via `currentManualTask`); auto fires that arrive during manual go to `pendingAutoAfterManual` and replay once after manual exits.

**No new chain logic needed for v1.2.** This is the load-bearing benefit of putting periodic and wake fires through the same `scheduleAutoSync()` entry point as mutation debounce.

### 4.4 Toggle-off during in-flight sync

The flag flip cancels only `periodicTimerCancellable` (future fires). The currently-executing sync's `Task` is `inFlight` and runs to completion or until replaced by manual/mutation. This is intentional: a user toggling off doesn't expect to lose freshness *they already have* ‚Äî they expect "no more periodic fires."

Manual "Sync Now" remains available regardless of the toggle. Mutation-debounced sync (2.10.3b) is also unaffected ‚Äî the toggle gates only the periodic timer.

### 4.5 UserDefaults write failure

Best-effort. `UserDefaults.standard.set(_:forKey:)` doesn't throw; if the disk write fails silently (which it almost never does), the in-memory `@Published` value still drives the live UI, and next app launch falls back to the previous persisted value. No user-facing handling.

### 4.6 App quit during sync

Unstructured Task in `inFlight` is terminated by process exit; `lastSyncedAt` may not have been persisted for a sync that was milliseconds away from completing. Acceptable ‚Äî next launch's `syncIfSignedIn()` will quickly fetch fresh state and update `lastSyncedAt`.

### 4.7 Surviving 2.10 invariants ‚Äî still load-bearing

- `apply(.createRemote)` does NOT have `Task.checkCancellation()` between `client.createHost` and `setServerId` (spec 2.10 ¬ß4.2.1).
- `mutationsForSyncSubject.send()` is only inside `addHost / updateHost / deleteHost` (spec 2.10 ¬ß3.2).
- `HostSyncStore` is held as long-lived `@StateObject` in `CatermApp` (spec 2.10 ¬ß3.1).

If any of these regress, every v1.2 invariant collapses too.

## 5. Data flow

### 5.1 Path A ‚Äî Startup with periodic enabled (default)

```
CatermApp.init()
  ‚Üí SyncPreferences(defaults: .standard) reads UserDefaults
       periodicSyncEnabled defaults to true if key missing
  ‚Üí HostSyncStore.init(... preferences:, userDefaults:)
       lastSyncedAt = defaults.object(forKey: "catermLastSyncedAt") as? Date
       preferences.$periodicSyncEnabled.sink { handlePeriodicEnabled($0) }
                                          .store(in: cancellables)
       NSWorkspace.shared.notificationCenter.publisher(for: .didWakeNotification)
                                            .sink { handleSystemWake() }
                                            .store(in: cancellables)
  ‚Üí handlePeriodicEnabled(true) fires synchronously from the .sink (current-value semantics):
       periodicTimerCancellable = Timer.publish(every: 900, on: .main, in: .common)
                                       .autoconnect()
                                       .sink { scheduleAutoSync() }
  ‚Üí WindowGroup .task { syncStore.syncIfSignedIn() }   (existing 2.10.5 path)
```

### 5.2 Path B ‚Äî Periodic timer fire (every 15 min)

```
Timer.publish 900s elapsed
  ‚Üí sink ‚Üí scheduleAutoSync()      (gated by manualInProgress per 2.10.3d)
       ‚îú‚îÄ manualInProgress? ‚Üí pendingAutoAfterManual = true; return
       ‚îî‚îÄ otherwise ‚Üí startSync() ‚Üí chain ‚Üí performSync()
                                              ‚îú‚îÄ success ‚Üí lastSyncedAt = Date()
                                              ‚îÇ           defaults.set(date, forKey: ...)
                                              ‚îÇ           @Published triggers view update
                                              ‚îî‚îÄ throws ‚Üí drop unobserved Result; lastSyncedAt unchanged
```

### 5.3 Path C ‚Äî Wake from sleep

```
Laptop opened ‚Üí
NSWorkspace.didWakeNotification posts on NSWorkspace.shared.notificationCenter
  ‚Üí handleSystemWake()
       scheduleAutoSync()        // immediate freshness fetch
       guard preferences.periodicSyncEnabled else { return }
       handlePeriodicEnabled(true)   // re-arm timer, next fire = wake + 15 min
                                     // (replaces the pre-sleep timer's drift)
```

### 5.4 Path D ‚Äî User toggles periodic OFF (and back ON)

```
SyncSettingsView Toggle flipped ‚Üí
  preferences.periodicSyncEnabled = false
       didSet ‚Üí defaults.set(false, forKey: "catermPeriodicSyncEnabled")
       @Published emits false
  ‚Üí HostSyncStore's sink: handlePeriodicEnabled(false)
       periodicTimerCancellable?.cancel()
       periodicTimerCancellable = nil
       (any in-flight sync continues ‚Äî we don't yank running syncs;
        next automatic fire simply won't happen)

User toggles back ON ‚Üí
  preferences.periodicSyncEnabled = true
  ‚Üí handlePeriodicEnabled(true) starts fresh Timer.publish (next fire = now + 15 min)
```

### 5.5 Invariants

1. **`lastSyncedAt` reflects last fully-applied sync** ‚Äî set only after the op loop completes without throwing. Tested by `testLastSyncedAtUnchangedOnError`.
2. **Toggle off cancels future fires only** ‚Äî in-flight sync continues. Tested by `testTogglingDisabledStopsFutureFires` (call count doesn't grow past the in-flight one).
3. **Toggle on re-creates the timer with a fresh schedule** ‚Äî next fire is `now + interval`, not "whatever the old schedule said". Tested by `testTogglingEnabledRestartsTimer`.
4. **Wake-from-sleep re-arms the periodic timer** ‚Äî without this, post-wake sync uses the leftover schedule from before sleep. Implementation detail covered by `testDidWakeNotificationTriggersSync` (the immediate-fire side); the re-arm is enforced by code review.
5. **No new bypass of the chain** ‚Äî periodic and wake both go through `scheduleAutoSync()`. Compile-time guarantee (private API).
6. **`lastSyncedAt` survives app restart** ‚Äî persisted in UserDefaults under `catermLastSyncedAt`. Tested by `testLastSyncedAtPersistsAcrossInstances`.
7. **`periodicSyncEnabled` survives app restart** ‚Äî persisted under `catermPeriodicSyncEnabled`. Tested by `testEnabledRoundtripsThroughUserDefaults`.

### 7.3 Test-only injection: `periodicInterval`

Production hard-codes `15 * 60` in the default value of `HostSyncStore.init(periodicInterval: TimeInterval = 15 * 60)`. Tests pass `0.05`. No new public abstraction; matches the existing `debounceInterval` pattern from 2.10.3a.

### 7.4 Wake-from-sleep ‚Äî what's *not* tested

`NSWorkspace.shared.notificationCenter` is a real notification center, and `testDidWakeNotificationTriggersSync` synthesizes the notification via `.post(name:object:)` to validate the wiring. The actual sleep/wake lifecycle is a system-level signal that XCTest can't simulate authentically ‚Äî that path is covered by manual smoke (close laptop, wait 1 min, open, observe `lastSyncedAt` jumping).

The "re-arm the periodic timer on wake" behavior is not unit-tested either ‚Äî testing it would require precise timer timing assertions that are flaky on CI. Code review enforces it instead. The risk is small: even without re-arming, the leftover timer would fire eventually; we'd just have a slightly worse "first sync after wake" latency.
