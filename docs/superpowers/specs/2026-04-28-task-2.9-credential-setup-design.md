# Task 2.9 — needsCredentialSetup UX (Design)

**Date:** 2026-04-28
**Author:** brainstorm session continuing Phase 2 v1.1
**Status:** approved
**Scope:** macOS Caterm app
**Phase:** 2.9 (follow-up to v1.1 baseline)

## 1. Goal

Close the dead-end UX hole created by v1.1's host-sync feature. When a user double-clicks (or right-click → Connect) a host that was pulled from the cloud and has no local credential configured for this device, today's behavior is silent failure (the new tab opens but SSH cannot authenticate). After 2.9, the same action opens a credential-setup sheet first; saving the credentials follows through with the original connect intent automatically.

This design assumes Phase 2 v1.1 baseline is shipped (commits `df6caf6` … `e92a159`) and that `SessionStore.needsCredentialSetup(_:)` already exists.

## 2. Non-goals

- Bulk credential setup across multiple locked hosts (deferred to a future plan).
- Syncing credentials/secrets to the server. Per spec §7.1.2, credentials remain a device-local overlay forever.
- Connection failure UX once credentials are set (wrong password / unreachable host) — that is the existing reconnect overlay's territory.
- Editing host metadata (name / hostname / port / username) from this sheet. The synced metadata is server source-of-truth; the sheet is auth-only by construction.

## 3. User flow

```
locked host (lock badge in sidebar)
    │
    ├─ double-click  ─┐
    │                 │
    └─ ⌃-click→Connect┘
            │
            ▼
HostListSidebar.connect(host)
            │
   needsCredentialSetup(host)?
       │           │
      YES          NO
       │           │
       ▼           ▼
[CredentialSetupView sheet]   existing path
       │                      (openTab + .catermOpenTab)
       │
   [Cancel] ── do nothing, drop original connect intent
       │
   [Save]
       │
       ▼
SessionStore.setHostSecret(secret, hostId: ..., kind: ...)  // step 1, may throw
SessionStore.setCredentialOnly(cred, for: host.id)          // step 2
       │
       ▼
sheet dismisses (only on success of both steps)
       │
       ▼
connect(updatedHost)         // re-enter, this time needsCredentialSetup == false
       │
       ▼
existing openTab path
```

Cancel intentionally drops the connect intent; the user clicked into the locked host but changed their mind.

## 4. Components

### 4.1 New: `CredentialSetupView.swift`

`apps/macos/Sources/Caterm/Views/CredentialSetupView.swift` — sheet view.

**Inputs**

```swift
struct CredentialSetupView: View {
    let host: SSHHost                                                  // for banner
    var onSaved: (CredentialSource, String?) async throws -> Void      // throws → inline error
    var onCancel: () -> Void
}
```

`onSaved` is `async throws`. The sheet wraps the call in a `Task { do { try await onSaved(...) } catch { errorMessage = ... } }` and only dismisses on success. This keeps the failure presentation **inside the sheet** — the parent doesn't need to know how to render an error in a child sheet's body.

**Internal state**:

- `credKind: CredKind` (`.password | .keyFile | .agent`) — local enum, mirrors `HostFormView.CredKind`
- `keyPath: String`
- `hasPassphrase: Bool`
- `pendingSecret: String`
- `errorMessage: String?` — failure text shown inline; cleared on the next Save attempt
- `isSaving: Bool` — disables buttons during the in-flight save

**UI layout** (top to bottom):

1. **Read-only banner**: two lines — `host.name` (font.headline) + `host.username@host.hostname:host.port` (font.caption, secondary). No editable controls.
2. **Section "Authentication"**:
   - `Picker(.segmented)` for credKind
   - `if credKind == .keyFile`: key path TextField + "Browse…" button (NSOpenPanel into `~/.ssh`) + Toggle "Key has passphrase"
   - `if credKind == .password`: SecureField "Password (stored in Keychain)"
   - `if credKind == .keyFile && hasPassphrase`: SecureField "Passphrase (stored in Keychain)"
3. **Footer**: HStack `[Cancel]  Spacer()  [Save]` — Save disabled when `!isValid || isSaving`.
4. **Inline error row** (only visible when `errorMessage != nil`): red Text below the footer.

**`isValid` gating** (must pass before Save is enabled):

```
.password  → !pendingSecret.isEmpty
.keyFile   → canonicalizedKeyPath() != nil
             AND (!hasPassphrase || !pendingSecret.isEmpty)
.agent     → always valid
```

`canonicalizedKeyPath()` is the function that resolves `~`/`~user` and verifies file existence:

```swift
private func canonicalizedKeyPath() -> String? {
    let expanded = (keyPath as NSString).expandingTildeInPath
    guard FileManager.default.fileExists(atPath: expanded) else { return nil }
    return expanded
}
```

This closes the failure mode where the user types `~/.ssh/id_ed25519` literally: without expansion, `FileManager.fileExists` returns false, `needsCredentialSetup` keeps returning true, and the sheet would re-open in a tight loop after every Save. The expanded absolute path is the value that gets baked into `CredentialSource.keyFile(keyPath: …)` on submit.

If the user types a non-existent path, Save stays disabled — no Save attempt, no loop, no inline error needed (the disabled Save button + visible empty-or-invalid path are the affordance).

**Reuse vs. extraction**: HostFormView implements an equivalent auth section. We duplicate ~60 lines rather than extract a shared subview. Justification: the two views diverge in surrounding context (banner vs. full edit form) and a shared abstraction would force both call sites to thread state through a less-tailored interface. YAGNI.

`onSaved` is invoked with `(CredentialSource, String?)`:

- `.password` → secret = pendingSecret (non-empty enforced by isValid)
- `.keyFile(canonicalizedPath, hasPassphrase: false)` → secret = nil
- `.keyFile(canonicalizedPath, hasPassphrase: true)` → secret = pendingSecret
- `.agent` → secret = nil

### 4.2 Modified: `HostListSidebar.swift`

Add `@State private var pendingCredentialHost: SSHHost?` and a `.sheet(item: $pendingCredentialHost)` modifier next to the existing add/edit sheets.

`connect(_:)` is the single chokepoint used by both double-tap (line 31) and the context menu Connect (line 23). Modify it to dispatch on the new pure helper `resolveConnectIntent`:

```swift
private func connect(_ host: SSHHost) {
    switch resolveConnectIntent(for: host, in: store) {
    case .promptCredentials:
        pendingCredentialHost = host
    case .openTab:
        let tabId = store.openTab(host: host)
        NotificationCenter.default.post(
            name: .catermOpenTab, object: nil, userInfo: ["tabId": tabId]
        )
    }
}
```

`resolveConnectIntent` is a free function alongside `HostListSidebar.swift`. **It must be `@MainActor`** because `SessionStore` is `@MainActor`-isolated (`SessionStore.swift:11`) and `needsCredentialSetup` is therefore main-actor-isolated too — calling it from non-isolated context fails Swift 6 isolation analysis:

```swift
enum ConnectIntent: Equatable {
    case openTab
    case promptCredentials
}

@MainActor
func resolveConnectIntent(for host: SSHHost, in store: SessionStore) -> ConnectIntent {
    store.needsCredentialSetup(host) ? .promptCredentials : .openTab
}
```

The single-line body looks trivial, but extracting it lets the test suite verify the gating without instantiating any SwiftUI view. See §7.

The new sheet — note `onSaved` is `async throws`, the sheet's button handler does the do/catch:

```swift
.sheet(item: $pendingCredentialHost) { host in
    CredentialSetupView(host: host) { cred, secret in
        // Order is intentional: write Keychain (the operation that fails) FIRST.
        // If it throws, no SessionStore mutation happens → Cancel is truly a no-op.
        if let secret, let kind = secretKind(for: cred) {
            try store.setHostSecret(secret, hostId: host.id, kind: kind)
        }
        try store.setCredentialOnly(cred, for: host.id)
        // Sheet dismisses on success (handled inside CredentialSetupView).
        // Resume the user's original connect intent with the refreshed host.
        if let refreshed = store.hosts.first(where: { $0.id == host.id }) {
            await MainActor.run {
                pendingCredentialHost = nil
                connect(refreshed)
            }
        }
    } onCancel: {
        pendingCredentialHost = nil
    }
}
```

`secretKind(for:)` is a small helper mapping `CredentialSource` → optional `SessionStore.SecretKind` (`SecretKind` is a nested enum at `SessionStore.swift:43`, so the fully qualified name is required at non-extension call sites):

```swift
private func secretKind(for cred: CredentialSource) -> SessionStore.SecretKind? {
    switch cred {
    case .password: return .password
    case .keyFile(_, hasPassphrase: true): return .keyPassphrase
    default: return nil  // .agent and key-without-passphrase have no secret to store
    }
}
```

The closure's `if let secret, let kind = secretKind(for: cred)` guards both: if either is nil, `setHostSecret` is skipped. By construction (§4.1), they are always either both non-nil or both nil — but the guard formalizes the invariant for the type system.

### 4.3 Modified: `SessionStore.swift`

Add one method, parallel to the v1.1 sync support block. The implementation uses **copy-save-assign** so that a `HostPersistence.save` failure leaves `self.hosts` untouched (atomicity at the in-memory level):

```swift
/// Replace credential overlay for an existing host. Does NOT bump updatedAt
/// — credential is a device-local concept that never propagates to the
/// server, so it must not trigger reconciler `.updateRemote` ops.
///
/// Atomicity: persists to a local copy first; only assigns to `self.hosts`
/// after `HostPersistence.save` returns. A disk-write failure throws
/// without mutating in-memory state, so callers can treat the call as
/// all-or-nothing for SessionStore-side state.
public func setCredentialOnly(_ source: CredentialSource,
                              for hostId: UUID) throws {
    guard let idx = hosts.firstIndex(where: { $0.id == hostId }) else { return }
    var updated = hosts
    updated[idx].credential = source
    try HostPersistence.save(updated, to: hostsURL)
    hosts = updated
}
```

This is intentionally narrower than `updateHost(_:)`, which (a) takes a full SSHHost and (b) bumps `updatedAt` (because metadata edits ARE meant to sync). The two methods coexist; callers pick by intent.

## 5. Data flow & invariants

- **`Host.credential` is device-local.** It is part of `Host`'s persisted JSON in `~/Library/Application Support/Caterm/hosts.json`, but it is never sent in `RemoteHostCreateInput` / `RemoteHostUpdateInput` (those hardcode `authType: "key"` per spec §7.1.2). Other devices' Hosts pulled via Sync Now default to `.password` (set by `SessionStore.addRemoteHost`), which is what makes them locked until 2.9's sheet runs.
- **`updatedAt` is a sync clock.** It must only advance for changes the user wants the server to learn about (rename, hostname change, etc.). Credential overlay changes never sync, so `setCredentialOnly` does not advance it. Skipping the bump means after the user sets credentials, a subsequent Sync Now produces zero ops for this host — exactly right.
- **Keychain key remains the local UUID.** `setHostSecret(_:hostId:kind:)` already keys on `hostId.uuidString`, which doesn't change between sync passes (per spec §7.1.3 dual-id mapping). So secrets persist correctly across re-syncs.

## 6. Error handling

The sheet drives all error presentation **inline** via its local `errorMessage` state (set in the `catch` of its own `Task` wrapper around `onSaved`). The parent's `errorMessage` (the existing global `.alert` mechanism on `HostListSidebar`) is reserved for non-sheet errors and is **not** used by 2.9.

Save sequence is ordered to minimize side-effect surface: Keychain first, SessionStore credential second. Combined with `setCredentialOnly`'s copy-save-assign atomicity (§4.3):

| Failure | Side effects | Sheet behavior |
|---|---|---|
| `setHostSecret` throws (step 1; Keychain unavailable) | None. Step 2 never runs. | Sheet stays open, inline `errorMessage` filled. Cancel afterward is a clean no-op (Keychain unchanged, SessionStore unchanged). |
| `setCredentialOnly` throws (step 2; `HostPersistence.save` fails — disk full / permission denied) | **Keychain has a stale entry.** SessionStore in-memory and on-disk are unchanged thanks to copy-save-assign. | Sheet stays open, inline `errorMessage` filled. Cancel afterward is **not** a perfect no-op: the Keychain entry written in step 1 stays. See "Stale Keychain entry" below. |
| `keyPath` doesn't exist after tilde expansion | None. | `isValid` returns false → Save button disabled → no Save attempt at all. Disabled button is the affordance, no `errorMessage`. |
| Cancel from a clean state (no failed Save before) | None. | `pendingCredentialHost = nil`, original connect intent dropped. |
| Subsequent SSH connection failure (wrong password, unreachable host) | N/A | Existing reconnect overlay handles it. Out of scope for 2.9. |

**Stale Keychain entry on step 2 failure** — accepted v1.1 behavior. Adding compensating `keychain.delete` would (a) require either passing the freshly-derived `kind` into a rollback closure or wrapping both writes in a SessionStore-side transactional method, both adding code surface, and (b) the failure trigger (step 2 disk write fails after step 1 succeeded) is a basically-impossible state (Keychain is on the same disk; if it were full the Keychain write would have failed first). The stale entry is overwritten on the user's next successful Save with the same credential kind, or purged by `deleteHost`. We document it instead of fixing it. This means **Cancel after a step-2 failure is a no-op for SessionStore but leaves a Keychain footprint** — an honest tradeoff over silent compensation.

## 7. Testing

Coverage adds **one new test file** in the existing `SessionStoreTests` target (focused on `setCredentialOnly` semantics including its atomicity contract) plus **one new test target** `CatermTests` (covering the resolver). All tests are `@MainActor` because `SessionStore` is `@MainActor`-isolated.

What is and isn't covered automatically:

- ✅ Locked host gates to `.promptCredentials`
- ✅ Unlocked host gates to `.openTab`
- ✅ End-to-end "Save unlocks Connect" — Keychain write + setCredentialOnly transitions resolver to `.openTab`
- ✅ `setCredentialOnly` does not bump `updatedAt`
- ✅ `setCredentialOnly` rolls back in-memory state on `HostPersistence.save` failure (atomicity contract)
- ❌ Sheet's inline `errorMessage` rendering — tested only via manual smoke (no SwiftUI view tests in repo)
- ❌ Cancel-as-no-op end-to-end — partial: the SessionStore-side no-op is implied by the rollback test; the SwiftUI dismissal path is manual smoke

### 7.1 `SessionStoreCredentialOnlyTests` (new file in existing `SessionStoreTests` target)

`apps/macos/Tests/SessionStoreTests/SessionStoreCredentialOnlyTests.swift`:

- `testSetCredentialOnlyChangesCredentialButNotUpdatedAt`: add a host, capture `updatedAt`, call `setCredentialOnly(.agent, for:)`, assert host's credential is `.agent` AND `updatedAt` is unchanged AND `hosts.json` reflects the new credential. Pins the "no sync side-effect" invariant.
- `testSetCredentialOnlyNonexistentHostIsNoOp`: call with random UUID, no throw, hosts unchanged.
- `testSetCredentialOnlyRollsBackOnSaveFailure`: configure SessionStore with a writable `tmpHostsURL`, add a host (succeeds, hosts.json written), then make the file unwritable via `chmod 0o400`, then call `setCredentialOnly(.agent, for: host.id)` and `XCTAssertThrowsError`. After the throw, assert `store.hosts[0].credential` is **the original value** (not `.agent`) — proves the copy-save-assign rollback. Restore permissions in `tearDown`. Pins finding #1's atomicity claim.

### 7.2 `ConnectIntentResolverTests` (new `CatermTests` test target)

`apps/macos/Tests/CatermTests/ConnectIntentResolverTests.swift`. Class is `@MainActor` because `resolveConnectIntent` is `@MainActor` (§4.2):

- `testLockedHostResolvesToPromptCredentials`: insert a host with `serverId != nil`, `credential = .password`, no Keychain entry → resolver returns `.promptCredentials`.
- `testUnlockedAgentHostResolvesToOpenTab`: insert a host with `credential = .agent` → resolver returns `.openTab`.
- `testCredentialSetupTransitionsLockedToUnlocked`: insert a locked host, assert resolver = `.promptCredentials`, then call `setHostSecret(...)` + `setCredentialOnly(.password, for:)` (in that order), assert resolver now returns `.openTab`. Pins end-to-end "Save unlocks Connect".

Why a new `CatermTests` target rather than reusing `SessionStoreTests`? Because `resolveConnectIntent` lives in the `Caterm` executable target (with `HostListSidebar`), not in `SessionStore`. SwiftPM requires a separate test target with the executable's name as a dep. Package.swift gains one new `.testTarget`.

Alternative considered: move `resolveConnectIntent` into the `SessionStore` library so it can be tested under `SessionStoreTests`. Rejected because the resolver references `ConnectIntent` which is a UI-flow concept, not a session-management concept; bleeding it into SessionStore makes that lib's API surface noisier without benefit.

### 7.3 No new SwiftUI view tests

Repository pattern is XCTest unit tests for stores and pure helpers; views are validated via manual smoke. Adding a UI test framework is out of scope. The throwing-closure design (finding #1) makes the sheet's error handling testable in principle via a fake `onSaved` that throws — but that's testing SwiftUI plumbing, not user-visible behavior. We rely on manual smoke for the sheet's inline-error visibility.

### 7.4 Manual smoke acceptance criteria

(These live in the implementation plan, not here, but are listed for design-completeness.)

1. With a locked host present, double-click → sheet appears, banner reads correctly, lock badge still visible.
2. Pick `.agent`, Save → sheet dismisses → tab opens.
3. Pick `.password`, type password, Save → sheet dismisses → tab opens → askpass satisfied from Keychain → tab connected.
4. Pick `.keyFile` with literal `~/.ssh/id_ed25519` (typed, not browsed) → Save enabled (tilde expansion works), Save → tab opens with the absolute path baked into the credential.
5. Pick `.keyFile` with a path that doesn't exist → Save remains disabled, no error displayed.
6. Cancel → sheet dismisses, no tab opens, lock badge still present.
7. Simulate Keychain failure if practical (deny prompt, etc.): sheet stays open, inline red error shown, Cancel afterward leaves host in original locked state (no orphan credential change on disk).
8. After step 2/3/4, lock badge is gone, and ⌘⇧, → Sync Now produces no ops for that host (verified via server log: no `sshHost.update` POST). Confirms the "credential is device-local overlay" invariant.
