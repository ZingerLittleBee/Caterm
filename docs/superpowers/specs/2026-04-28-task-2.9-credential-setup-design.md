# Task 2.9 — needsCredentialSetup UX (Design)

**Date:** 2026-04-28
**Author:** brainstorm session continuing Phase 2 v1.1
**Status:** approved
**Scope:** macOS Caterm app
**Phase:** 2.9 (follow-up to v1.1 baseline)

## 1. Goal

Close the dead-end UX hole created by v1.1's host-sync feature. When a user double-clicks (or right-click → Connect) a host that was pulled from the cloud and has no local credential configured for this device, today's behavior is silent failure (the new tab opens but SSH cannot authenticate). After 2.9, the same action opens a credential-setup sheet first; saving the credentials follows through with the original connect intent automatically.

This design assumes Phase 2 v1.1 baseline is shipped (commits `df6caf6` … `e92a159`) and that `SessionStore.needsCredentialSetup(_:)` already exists.

## 2. Non-goals

- Bulk credential setup across multiple locked hosts (deferred to a future plan).
- Syncing credentials/secrets to the server. Per spec §7.1.2, credentials remain a device-local overlay forever.
- Connection failure UX once credentials are set (wrong password / unreachable host) — that is the existing reconnect overlay's territory.
- Editing host metadata (name / hostname / port / username) from this sheet. The synced metadata is server source-of-truth; the sheet is auth-only by construction.

## 3. User flow

```
locked host (lock badge in sidebar)
    │
    ├─ double-click  ─┐
    │                 │
    └─ ⌃-click→Connect┘
            │
            ▼
HostListSidebar.connect(host)
            │
   needsCredentialSetup(host)?
       │           │
      YES          NO
       │           │
       ▼           ▼
[CredentialSetupView sheet]   existing path
       │                      (openTab + .catermOpenTab)
       │
   [Cancel] ── do nothing, drop original connect intent
       │
   [Save]
       │
       ▼
SessionStore.setCredentialOnly(cred, for: host.id)
SessionStore.setHostSecret(secret, hostId: ..., kind: ...)  // when applicable
       │
       ▼
sheet dismisses
       │
       ▼
connect(updatedHost)         // re-enter, this time needsCredentialSetup == false
       │
       ▼
existing openTab path
```

Cancel intentionally drops the connect intent; the user clicked into the locked host but changed their mind.

## 4. Components

### 4.1 New: `CredentialSetupView.swift`

`apps/macos/Sources/Caterm/Views/CredentialSetupView.swift` — sheet view.

**Inputs**

```swift
struct CredentialSetupView: View {
    let host: SSHHost                                 // for read-only banner
    var onSaved: (CredentialSource, String?) -> Void  // (cred, optional secret)
    var onCancel: () -> Void
}
```

**Internal state** mirrors HostFormView's auth section:

- `credKind: CredKind` (`.password | .keyFile | .agent`) — local enum, mirrors `HostFormView.CredKind`
- `keyPath: String`
- `hasPassphrase: Bool`
- `pendingSecret: String`
- `errorMessage: String?` (Keychain failures, etc.)

**UI layout** (top to bottom):

1. **Read-only banner**: two lines — `host.name` (font.headline) + `host.username@host.hostname:host.port` (font.caption, secondary). No editable controls; this is purely context so the user knows which host they're configuring.
2. **Section "Authentication"**:
   - `Picker(.segmented)` for credKind
   - `if credKind == .keyFile`: key path TextField + "Browse…" button (NSOpenPanel into `~/.ssh`) + Toggle "Key has passphrase"
   - `if credKind == .password`: SecureField "Password (stored in Keychain)"
   - `if credKind == .keyFile && hasPassphrase`: SecureField "Passphrase (stored in Keychain)"
   - `if errorMessage != nil`: red Text below
3. **Footer**: HStack `[Cancel]  Spacer()  [Save]` — Save disabled until `isValid` (key file mode requires non-empty `keyPath`; password mode requires non-empty `pendingSecret`; agent always valid).

**Reuse vs. extraction**: HostFormView already implements an equivalent auth section. We choose to duplicate ~60 lines rather than extract a shared subview. Justification: the two views diverge in surrounding context (banner vs. full edit form), share only structural similarity, and a shared abstraction would force both call sites to thread state through a less-tailored interface. YAGNI.

`onSaved` is invoked with `(CredentialSource, String?)` where the optional String is the secret to persist:

- `.password` → secret = pendingSecret (non-empty enforced by isValid)
- `.keyFile(_, hasPassphrase: false)` → secret = nil
- `.keyFile(_, hasPassphrase: true)` → secret = pendingSecret
- `.agent` → secret = nil

### 4.2 Modified: `HostListSidebar.swift`

Add `@State private var pendingCredentialHost: SSHHost?` and a `.sheet(item: $pendingCredentialHost)` modifier next to the existing add/edit sheets.

`connect(_:)` is the single chokepoint already used by both double-tap (line 31) and the context menu Connect (line 23). Modify it:

```swift
private func connect(_ host: SSHHost) {
    if store.needsCredentialSetup(host) {
        pendingCredentialHost = host
        return
    }
    let tabId = store.openTab(host: host)
    NotificationCenter.default.post(
        name: .catermOpenTab, object: nil, userInfo: ["tabId": tabId]
    )
}
```

The new sheet:

```swift
.sheet(item: $pendingCredentialHost) { host in
    CredentialSetupView(host: host) { cred, secret in
        do {
            try store.setCredentialOnly(cred, for: host.id)
            if let secret { try persistSecret(host: host, secret: secret, cred: cred) }
            pendingCredentialHost = nil
            // Re-fetch host with new credential and follow through.
            if let refreshed = store.hosts.first(where: { $0.id == host.id }) {
                connect(refreshed)
            }
        } catch {
            errorMessage = error.localizedDescription
        }
    } onCancel: {
        pendingCredentialHost = nil
    }
}
```

`persistSecret` already exists on `HostListSidebar` (line 100 of the current file) but takes `host: SSHHost` and reads `host.credential`. Since we just changed `host.credential` via `setCredentialOnly`, the existing `persistSecret(host:secret:)` will pick up the new credential value if we re-fetch. Or we add an overload `persistSecret(host:secret:cred:)` that takes the new credential explicitly to avoid the round-trip read. Implementation choice deferred to plan stage; both work.

### 4.3 Modified: `SessionStore.swift`

Add one method, parallel to the v1.1 sync support block:

```swift
/// Replace credential overlay for an existing host. Does NOT bump updatedAt
/// — credential is a device-local concept that never propagates to the
/// server, so it must not trigger reconciler `.updateRemote` ops.
public func setCredentialOnly(_ source: CredentialSource,
                              for hostId: UUID) throws {
    guard let idx = hosts.firstIndex(where: { $0.id == hostId }) else { return }
    hosts[idx].credential = source
    try HostPersistence.save(hosts, to: hostsURL)
}
```

This is intentionally narrower than `updateHost(_:)`, which (a) takes a full SSHHost and (b) bumps `updatedAt` (because metadata edits ARE meant to sync). The two methods coexist; callers pick by intent.

## 5. Data flow & invariants

- **`Host.credential` is device-local.** It is part of `Host`'s persisted JSON in `~/Library/Application Support/Caterm/hosts.json`, but it is never sent in `RemoteHostCreateInput` / `RemoteHostUpdateInput` (those hardcode `authType: "key"` per spec §7.1.2). Other devices' Hosts pulled via Sync Now default to `.password` (set by `SessionStore.addRemoteHost`), which is what makes them locked until 2.9's sheet runs.
- **`updatedAt` is a sync clock.** It must only advance for changes the user wants the server to learn about (rename, hostname change, etc.). Credential overlay changes never sync, so `setCredentialOnly` does not advance it. Skipping the bump means after the user sets credentials, a subsequent Sync Now produces zero ops for this host — exactly right.
- **Keychain key remains the local UUID.** `setHostSecret(_:hostId:kind:)` already keys on `hostId.uuidString`, which doesn't change between sync passes (per spec §7.1.3 dual-id mapping). So secrets persist correctly across re-syncs.

## 6. Error handling

| Failure | Behavior |
|---|---|
| Save: `setCredentialOnly` throws (rare — only `HostPersistence.save` could fail on disk) | sheet stays open, `errorMessage` filled, no `pendingCredentialHost = nil` |
| Save: `setHostSecret` throws (Keychain unavailable) | same as above; credential change rolled back? **No.** Credential is on-disk and won't sync without a metadata change anyway; the Keychain error is the user-visible blocker. Sheet stays open with `errorMessage`; user can retry. |
| Cancel | `pendingCredentialHost = nil`, no other state changes, original connect intent dropped |
| Subsequent SSH connection failure (wrong password, unreachable host) | Existing reconnect overlay handles it. Out of scope for 2.9. |

The asymmetry on Keychain failure (credential lands locally but secret doesn't) is acceptable because: the next time the user clicks Connect on the same host, `needsCredentialSetup` will return true again (no Keychain entry under `<id>.password`), so the sheet reopens. Self-healing.

## 7. Testing

- **`SessionStoreCredentialOnlyTests`** (new XCTest):
  - `testSetCredentialOnlyDoesNotBumpUpdatedAt`: load host, capture `updatedAt`, call `setCredentialOnly(.agent, for:)`, assert credential changed AND `updatedAt` unchanged AND hosts.json reflects the new credential.
  - `testSetCredentialOnlyNonexistentHostIsNoOp`: call with random UUID, no throw, hosts unchanged.
- **No new SwiftUI view tests** — repository pattern is XCTest unit tests for stores/logic; views are validated via manual smoke. Adding a UI test framework is out of scope.
- **Manual smoke acceptance criteria** (will live in plan, not here):
  1. With a locked host present, double-click → sheet appears, banner reads correctly, lock badge still visible.
  2. Pick `.agent`, Save → sheet dismisses → tab opens connecting (succeeds if server has SSH agent forwarding configured, otherwise shows live SSH error in tab — both acceptable).
  3. Pick `.password`, type password, Save → sheet dismisses → tab opens connecting → askpass prompt is satisfied from Keychain → tab connected.
  4. Pick `.keyFile` with `~/.ssh/id_ed25519`, no passphrase, Save → tab opens → connected.
  5. Cancel → sheet dismisses, no tab opens, lock badge still on the host.
  6. After step 2/3/4, lock badge is gone (since `needsCredentialSetup` now returns false), and a subsequent ⌘⇧, → Sync Now produces no ops for that host (verified via server log: no `sshHost.update` POST).

## 9. Open follow-ups (not in 2.9)

- **2.10 Auto-sync triggers** — orthogonal, no dependency on 2.9.
- **Bulk credential setup** — once user signs in to a third device, a list of N locked hosts could be tedious. Consider a "Configure all" affordance in v1.2.
- **Credential rotation UX** — if the stored secret stops working (server-side password change), `needsCredentialSetup` still returns false (Keychain entry exists) so the sheet won't auto-open. User must edit via the existing HostFormView edit sheet. Acceptable for v1.1; flag if friction emerges.
