# v1.6 Remote terminfo auto-install â€” Design

**Status:** Approved (brainstorming â†’ spec, 2026-04-29)
**Branch:** `barcelona-v1` (continuation)
**Predecessor:** v1.5 (`d73359f` defaults `term = xterm-256color` for compat)

## 1. Goal

Let users opt into Ghostty's full terminfo (`xterm-ghostty`) on remote SSH hosts without running any install commands themselves. When the toggle is ON, every SSH connection ensures `xterm-ghostty` terminfo is available on the remote â€” installing it idempotently if missing â€” and falls back to `xterm-256color` automatically when installation isn't possible. When the toggle is OFF (default), behavior matches the current v1.5 release: `TERM=xterm-256color` everywhere, no remote mutation.

## 2. Non-goals (deferred)

- **Per-host override** of the toggle. YAGNI â€” single global preference.
- **Manual "Reinstall terminfo on this host" button.** B' is self-healing; redundant.
- **ControlMaster connection multiplexing.** B' uses a single SSH session, so dedup is unnecessary.
- **Renaming `SyncPreferences` â†’ `Preferences`.** Naming mismatch acknowledged; cleanup deferred to v1.7.
- **Cloud-syncing the toggle.** Pure local UX preference; no cross-device need.
- **Auto-detecting remote shell.** POSIX `/bin/sh` syntax used in the wrapper â€” works on Bash / Zsh / Dash / Ash / Busybox sh.

## 3. Architecture

The toggle drives **two coordinated changes**:

| `installTerminfoEnabled` | `term =` in Caterm-managed config | SSH command shape |
|---|---|---|
| `false` (default) | `xterm-256color` | `ssh ... 'user'@'host'` (current v1.5 shape) |
| `true` | `xterm-ghostty` | `ssh ... -t 'user'@'host' '<inline wrapper>'` |

The terminfo source dump is **bundled as a static resource** in the SSHCommandBuilder SwiftPM target. We deliberately do *not* rely on a local Ghostty.app install or shell out to `infocmp` at runtime â€” most Caterm users won't have Ghostty.app, and shelling out introduces a startup-time dependency.

The bundle is a snapshot of `infocmp -x xterm-ghostty` from the libghostty version we vendor. When `Frameworks/GhosttyKit.xcframework` is bumped, the bundle must be regenerated by hand. A short `Resources/README.md` documents the regen one-liner.

## 4. Component summary

| Component | Kind | Responsibility |
|---|---|---|
| `SSHCommandBuilder/Resources/xterm-ghostty.terminfo` | New static resource | Bundled `infocmp -x xterm-ghostty` dump (~3 KB). |
| `SSHCommandBuilder/Resources/README.md` | New static doc | Regen instruction + libghostty version pin. |
| `TerminfoSource.terminfoDump()` | New internal func in SSHCommandBuilder | Loads bundle resource once, caches in a `static let`. Returns `String?` â€” `nil` only on missing-bundle build error. |
| `SSHCommandBuilder.build(..., installTerminfo:)` | Existing func, new param | When `false`: existing behavior. When `true`: emit a `-t` flag and append a quoted remote command (the inline wrapper). |
| `ConfigStore.writeManagedConfig(useGhosttyTerminfo:)` | Existing func, new param | Picks `xterm-ghostty` or `xterm-256color` for the `term =` line. Default `false` preserves v1.5 default. |
| `SyncPreferences.installTerminfoEnabled` | New `@Published` field | Default `false`; persisted to UserDefaults via `didSet`. Emits to subscribers so view reloads + managed config rewrites. |
| `SyncSettingsView` Toggle | New UI row | Label + caption + binds to preference. Settings sheet, between existing "Background sync" and account section. |
| Session spawn site | Existing call site adjustment | Reads `installTerminfoEnabled` once before invoking `SSHCommandBuilder.build`. |

## 5. Detail

### 5.1 Inline wrapper (POSIX sh, runs on remote)

```sh
if ! infocmp xterm-ghostty >/dev/null 2>&1; then
  if command -v tic >/dev/null 2>&1; then
    tic -x - <<'TERMINFO_EOF'
<terminfo dump>
TERMINFO_EOF
    [ $? -ne 0 ] && export TERM=xterm-256color
  else
    export TERM=xterm-256color
  fi
fi
exec $SHELL -l
```

**Why each piece:**

- `infocmp xterm-ghostty >/dev/null 2>&1` â€” idempotency check. If user (or a previous Caterm connect) already installed it, skip the work and the data transfer.
- `command -v tic` â€” guard against `tic` missing. BusyBox sh has it as a builtin only sometimes; Alpine without `ncurses` ships no `tic`. We don't try to install ncurses.
- `<<'TERMINFO_EOF'` â€” **single-quoted** delimiter. Disables `$` expansion and backtick substitution in the body. Critical because `infocmp -x` output contains escape sequences (`\E`, `\033`) and `$<...>` delay markers that look like shell variables.
- `tic -x -` â€” read terminfo source from stdin (`-`), enable extended capabilities (`-x`). Without `-x`, the `XT`, `Tc`, `Su`, etc. caps are dropped â€” graceful loss; basic terminfo still works.
- `[ $? -ne 0 ] && export TERM=xterm-256color` â€” runtime install failure (read-only home, EPERM). Falls back in the same shell session before the user gets a prompt.
- `exec $SHELL -l` â€” replaces wrapper script with user's login shell, in-place. Equivalent to what `ssh user@host` does by default. `-l` so PATH / aliases / etc. behave as login shell.

**Why no `set -e` / no `trap`:** every failure is locally handled (each branch ends in either `tic` success, `export TERM=xterm-256color`, or skipping the block). `set -e` would abort the wrapper on the `infocmp` exit-1 path (the "not installed" detection itself). `trap` adds noise without value.

**Why no `[ -t 0 ]` / `tput init`:** SSH `-t` already allocates a TTY for us. The wrapper executes inside that TTY.

**Why no `if [ -z "$SHELL" ]; then SHELL=/bin/sh; fi`:** every Unix shell login sets `$SHELL`. Defensive code for a non-existent failure.

### 5.2 SSHCommandBuilder integration

Current builder ends with:
```
ssh ... -p 22 'user'@'host'
```

With `installTerminfo: true`:
```
ssh ... -p 22 -t 'user'@'host' '<wrapper-string>'
```

The `-t` flag must come **before** `user@host` (or anywhere in the options block); the remote-command must come **after** `user@host`. Existing `Arg.raw / .quoted / .verbatim` enum is enough â€” wrapper string is one big `.quoted(...)`.

The wrapper string is built as:

```swift
let wrapper = """
if ! infocmp xterm-ghostty >/dev/null 2>&1; then
  if command -v tic >/dev/null 2>&1; then
    tic -x - <<'TERMINFO_EOF'
\(terminfoDump)
TERMINFO_EOF
    [ $? -ne 0 ] && export TERM=xterm-256color
  else
    export TERM=xterm-256color
  fi
fi
exec $SHELL -l
"""
```

`ShellQuote.posix` already escapes embedded single quotes correctly (the `'\''` dance), so the wrapper â€” including the heredoc body â€” is safe inside the SSH command's outer single quotes.

**Skip-if-bundle-missing semantics:** If `TerminfoSource.terminfoDump()` returns `nil` (build problem), `SSHCommandBuilder.build(installTerminfo: true)` falls back to the `installTerminfo: false` shape and logs once via `NSLog`. Caller sees no error â€” toggle becomes a no-op rather than a hard failure. The `term =` config still says `xterm-ghostty` in this degenerate case, but on first connect the absent terminfo will surface a transient `clear: unknown terminal type` until the user notices and disables. We accept this â€” it's a build-time/CI failure, not a runtime user-facing issue.

### 5.3 ConfigStore wiring

Current `managedConfigContent` is a `static let`. Becomes a function:

```swift
static func managedConfigContent(useGhosttyTerminfo: Bool) -> String {
    let term = useGhosttyTerminfo ? "xterm-ghostty" : "xterm-256color"
    return """
        # Caterm-managed; do not edit. Override in your user config at:
        #   ~/Library/Application Support/Caterm/config

        term = \(term)

        keybind = super+up=scroll_page_lines:-1
        keybind = super+down=scroll_page_lines:1
        keybind = super+page_up=scroll_page_fractional:-1
        keybind = super+page_down=scroll_page_fractional:1
        keybind = super+home=scroll_to_top
        keybind = super+end=scroll_to_bottom
        keybind = super+k=clear_screen
        """
}
```

`writeManagedConfig` gains the parameter:

```swift
@MainActor
public static func writeManagedConfig(useGhosttyTerminfo: Bool) throws {
    // existing path / atomic compare-and-skip logic, but using
    // managedConfigContent(useGhosttyTerminfo:)
}
```

`GhosttyConfig.init` reads the toggle from `SyncPreferences` and passes through:

```swift
try ConfigStore.writeManagedConfig(useGhosttyTerminfo: prefs.installTerminfoEnabled)
```

When the user toggles, `SyncPreferences.installTerminfoEnabled.didSet` calls `try? ConfigStore.writeManagedConfig(useGhosttyTerminfo: ...)` â€” the next libghostty surface picks it up. Existing surfaces keep their old config until reconnect (acceptable; the user just toggled, they understand a reconnect is coming).

### 5.4 Toggle storage

Add to `SyncPreferences`:

```swift
@Published public var installTerminfoEnabled: Bool {
    didSet {
        userDefaults.set(installTerminfoEnabled, forKey: Self.installTerminfoKey)
        try? ConfigStore.writeManagedConfig(useGhosttyTerminfo: installTerminfoEnabled)
    }
}

private static let installTerminfoKey = "com.caterm.preferences.installTerminfoEnabled"
```

Hydrate in `init`:

```swift
self.installTerminfoEnabled = userDefaults.bool(forKey: Self.installTerminfoKey)
// Note: bool(forKey:) returns false when key is absent â€” that's the
// default we want, no fallback needed.
```

Naming mismatch with `SyncPreferences` documented in Â§2 non-goals â€” accepted as v1.7 cleanup.

### 5.5 Session spawn site

Wherever `SSHCommandBuilder.build(...)` is currently called (most likely `SessionStore` or the host-launch path in `MainWindow`/`HostListSidebar`), the caller must:

1. Hold a reference to `SyncPreferences` (already injected via `@EnvironmentObject` per v1.4)
2. Read `prefs.installTerminfoEnabled` synchronously at spawn time
3. Pass through as `installTerminfo:` to `SSHCommandBuilder.build`

The plan stage will identify the exact call site and grep for current usages. The change is one parameter threaded one level â€” no architectural plumbing needed.

### 5.6 Settings UI

In `SyncSettingsView`, between the existing "Background sync" Toggle group and the account section:

```swift
Section("Terminal") {
    Toggle("Install Ghostty terminfo on remote hosts",
           isOn: $preferences.installTerminfoEnabled)
    Text("Provides full Ghostty rendering features (true colors, hyperlinks). Falls back to standard terminfo automatically if installation isn't possible.")
        .font(.caption)
        .foregroundStyle(.secondary)
}
```

No "explain what terminfo is" essay â€” the caption is two sentences, focused on outcome ("rendering features", "falls back"). A user who doesn't understand terminfo doesn't need to.

## 6. Failure mode matrix

| # | Trigger | Wrapper behavior | User-visible effect |
|---|---|---|---|
| 1 | Already installed (subsequent connect) | Skip install block | Silent; ~10ms `infocmp` check |
| 2 | Fresh install succeeds | `tic -x -` writes `~/.terminfo/x/xterm-ghostty` | Silent; ~50ms `tic` write |
| 3 | `tic` not present | `command -v tic` fails | Silent; `TERM=xterm-256color` in this session |
| 4 | `tic -x` flag unsupported | `tic` exits non-zero | Silent; same fallback as #3 |
| 5 | `tic` runtime fails (EROFS, EACCES) | Same as #4 | `tic: ...` stderr flashes one line; same fallback |
| 6 | Local terminfo bundle missing | SSHCommandBuilder skips wrapper, NSLogs once | Toggle effectively no-op; user sees `xterm-ghostty` in TERM but no terminfo on remote (latent issue) |
| 7 | Network/auth failure during connect | No different from baseline | Existing reconnect overlay handles |

### 7.3 Manual smoke

1. Toggle ON, reconnect to GEN2 (the user's host that surfaced the original `'xterm-ghostty': unknown terminal type` issue): `clear` works, `vim` works, `tmux new` works, `ls --color` works.
2. Toggle ON, immediately toggle OFF, reconnect: `echo $TERM` returns `xterm-256color`; remote `~/.terminfo/x/xterm-ghostty` from the prior connect is still there but unused (acceptable â€” we don't clean up).
3. Toggle ON, reconnect to a known-broken host (e.g. a docker `alpine:latest` container with no `ncurses`): session opens, prompt appears, `echo $TERM` returns `xterm-256color`.

## 8. Open questions

- **8-OQ-1:** Does `tic -x -` from BSD `ncurses` (macOS-bundled) produce a terminfo source that GNU `ncurses` (most Linux) can re-parse? The bundle is generated from BSD `infocmp` on the build mac. Risk: extended capability syntax differs between BSD and GNU. Resolution: integration test runs `tic -x -` on a real GNU/Linux Alpine container with the bundled dump; if it fails, regenerate from a portable source (probably ghostty upstream's `terminfo.src`). Test is the gate.
- **8-OQ-2:** Does single-quote escape in `ShellQuote.posix` survive a 3-KB wrapper containing its own newlines, dollar signs, and backslashes? `ShellQuote.posix` was designed for usernames / hostnames (short). Resolution: SSHCommandBuilderTests must include a fixture asserting the full wrapper round-trips through `posix(...)` and back via `bash -c`, verifying the wrapper executes locally.
- **8-OQ-3:** Where exactly is `SSHCommandBuilder.build(...)` called from? Plan stage greps and pins. If the call lives in `SessionStore`, threading the preference is one line. If it lives in multiple places, we may need to centralize first â€” but that's a small pre-task, not a scope expansion.

## 9. Decisions log

| # | Decision | Rationale |
|---|---|---|
| 1 | B' inline pattern, not pre-flight | Single auth, no state, self-healing. See brainstorming session. |
| 2 | Bundle terminfo source as resource, not shell out at runtime | Most users lack Ghostty.app; predictable build-time pinning. |
| 3 | Single global toggle, default OFF | YAGNI on per-host; user agreed default OFF respects "don't mutate remote without consent". |
| 4 | `term =` value driven by toggle (not always `xterm-ghostty`) | Toggle OFF must produce the exact v1.5 baseline shape including the `term = xterm-256color` line. |
| 5 | Bundle-missing degrades to toggle-OFF behavior, not hard error | Build-time issue shouldn't crash users at runtime. |
| 6 | POSIX `/bin/sh` syntax in wrapper, not `bash`-only | Works on Dash, Ash, BusyBox sh â€” covers minimal containers. |
| 7 | `<<'TERMINFO_EOF'` single-quoted delim | Prevents shell expansion in terminfo body (which contains escape sequences and `$<...>` delays). |
| 8 | `exec $SHELL -l` ending | Mirrors default `ssh user@host` behavior: login shell, in-place replacement. |
| 9 | Add `installTerminfoEnabled` to `SyncPreferences` despite naming mismatch | Pragmatic; rename to `Preferences` deferred to v1.7 to keep v1.6 scope tight. |
| 10 | `didSet` on toggle calls `ConfigStore.writeManagedConfig` synchronously | Existing surfaces keep old config until reconnect â€” that's fine; toggling implies user intent to reconnect. |
| 11 | No "explain terminfo" body text in Settings | Two-sentence caption; users who care can read docs. |
| 12 | No clean-up on toggle-OFF (don't try to `rm` remote `~/.terminfo`) | Cleanup adds risk (deleting user's own files), little reward. Stale terminfo is harmless. |
