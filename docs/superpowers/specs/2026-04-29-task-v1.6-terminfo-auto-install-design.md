# v1.6 Remote terminfo auto-install — Design

**Status:** Approved (brainstorming → spec → review round 1, 2026-04-29)
**Branch:** `barcelona-v1` (continuation)
**Predecessor:** v1.5 (`d73359f` defaults `term = xterm-256color` for compat)

## 1. Goal

Let users opt into Ghostty's full terminfo (`xterm-ghostty`) on remote SSH hosts without running any install commands themselves. When the toggle is ON, every SSH connection ensures `xterm-ghostty` terminfo is available on the remote — installing it idempotently if missing — and falls back to `xterm-256color` automatically when installation isn't possible. When the toggle is OFF (default), behavior matches the current v1.5 release: `TERM=xterm-256color` everywhere, no remote mutation.

## 2. Non-goals (deferred)

- **Per-host override** of the toggle. YAGNI — single global preference.
- **Manual "Reinstall terminfo on this host" button.** B' is self-healing; redundant.
- **ControlMaster connection multiplexing.** B' uses a single SSH session, so dedup is unnecessary.
- **Renaming `SyncPreferences` → `Preferences`.** Naming mismatch acknowledged; cleanup deferred to v1.7.
- **Cloud-syncing the toggle.** Pure local UX preference; no cross-device need.
- **Auto-detecting remote shell.** POSIX `/bin/sh` syntax used in the wrapper — works on Bash / Zsh / Dash / Ash / BusyBox sh; `-l` flag widely supported in modern shells (verified in shell-matrix smoke, §7.3).
- **Local TERM upgrade for non-SSH commands.** This v1.6 toggle is scoped to *remote* hosts only; local libghostty rendering already surfaces full features regardless of `$TERM`. A separate "Use Ghostty terminfo locally" preference would require app restart (§5.3 design fact) and is left for v1.7+.

## 3. Architecture

The toggle drives **two coordinated changes at SSH-command-build time** — never at libghostty-config-load time:

| `installTerminfoEnabled` | SSHCommandBuilder.Output.env | SSH command shape |
|---|---|---|
| `false` (default) | (no `TERM` override; libghostty PTY env carries `TERM=xterm-256color` from v1.5 managed config) | `ssh ... 'user'@'host'` (current v1.5 shape) |
| `true` | adds `("TERM", "xterm-ghostty")` | `ssh ... -t 'user'@'host' '<inline wrapper>'` |

**Critical design fact**: `GhosttyApp.shared` is a process-level singleton; `GhosttyConfig` is loaded exactly once at app launch and never reloaded (`apps/macos/Sources/TerminalEngine/GhosttyApp.swift:19,38`). Therefore the toggle does **not** rewrite the Caterm-managed config file at runtime — the managed config stays at the v1.5 baseline (`term = xterm-256color`) regardless of toggle state. Instead, the SSH subprocess gets its `TERM` overridden via the per-surface env vars that libghostty propagates into the PTY (`apps/macos/Sources/TerminalEngine/GhosttySurface.swift` env_vars handling).

The terminfo source dump is **bundled as a static resource** in the SSHCommandBuilder SwiftPM target. We deliberately do *not* rely on a local Ghostty.app install or shell out to `infocmp` at runtime — most Caterm users won't have Ghostty.app, and shelling out introduces a startup-time dependency.

The bundle source is **generated from upstream ghostty's portable terminfo source** (`ghostty/src/terminfo/ghostty.terminfo` or equivalent), not reverse-derived via `infocmp -x` on macOS. BSD `infocmp` output and GNU `tic` parser have known incompatibilities for extended capabilities; using the portable upstream source avoids this entire class of issue. The Docker E2E integration test on a real GNU/Linux Alpine container is the gate (§7.2). When `Frameworks/GhosttyKit.xcframework` is bumped, the bundle must be regenerated by hand. A short `Resources/README.md` documents the regen one-liner and pins the source URL.

## 4. Component summary

| Component | Kind | Responsibility |
|---|---|---|
| `SSHCommandBuilder/Resources/xterm-ghostty.terminfo` | New static resource | Upstream ghostty portable terminfo source (~3 KB). |
| `SSHCommandBuilder/Resources/README.md` | New static doc | Regen instruction (curl from upstream URL) + libghostty version pin. |
| `TerminfoSource.terminfoDump()` | New internal func in SSHCommandBuilder target | Loads bundle resource once, caches in a `static let`. Returns `String?` — `nil` only on missing-bundle build error. |
| `SSHCommandBuilder.build(..., installTerminfo:)` | Existing func, new param | When `false`: existing behavior. When `true` *and* bundle non-nil: emit `-t` flag, append wrapper as remote command, **and** add `("TERM", "xterm-ghostty")` to `Output.env`. When `true` *and* bundle nil: silently degrade to `false` shape (no `-t`, no env override) — managed config still says `xterm-256color`, so remote correctly sees `xterm-256color`, no latent issue. |
| `SyncPreferences.installTerminfoEnabled` | New `@Published` field | Default `false`; persisted to UserDefaults via `didSet`. **Does not touch `ConfigStore`** (see §5.3). View-side observers fire on change. |
| `SyncSettingsView` toggle | New UI row | Inside a new `Section("Terminal")` placed *after* `Section("Sync")` in the existing form. |
| `TerminalSurfaceRepresentable` env-object addition | Existing struct, new `@EnvironmentObject` | Adds `preferences: SyncPreferences`; reads `installTerminfoEnabled` in `makeNSView`; passes through to `SessionStore.surfaceConfig`. |
| `SessionStore.surfaceConfig(for:installTerminfo:)` | Existing func, new param | Forwards the bool to `SSHCommandBuilder.build`. |

## 5. Detail

### 5.1 Inline wrapper (POSIX sh, runs on remote)

```sh
if ! infocmp xterm-ghostty >/dev/null 2>&1; then
  if command -v tic >/dev/null 2>&1; then
    tic -x - <<'TERMINFO_EOF'
<terminfo dump>
TERMINFO_EOF
    [ $? -ne 0 ] && export TERM=xterm-256color
  else
    export TERM=xterm-256color
  fi
fi
exec "${SHELL:-/bin/sh}" -l
```

**Why each piece:**

- `infocmp xterm-ghostty >/dev/null 2>&1` — idempotency check. If user (or a previous Caterm connect) already installed it, skip the work and the data transfer.
- `command -v tic` — guard against `tic` missing. BusyBox sh has it as a builtin only sometimes; Alpine without `ncurses` ships no `tic`. We don't try to install ncurses.
- `<<'TERMINFO_EOF'` — **single-quoted** delimiter. Disables `$` expansion and backtick substitution in the body. Critical because terminfo source contains escape sequences (`\E`, `\033`) and `$<...>` delay markers that look like shell variables.
- `tic -x -` — read terminfo source from stdin (`-`), enable extended capabilities (`-x`). When `-x` flag is unsupported (very old ncurses), `tic` exits non-zero — caller treats this identically to other failures and falls back to `xterm-256color` (we **do not** retry without `-x` and we do not accept partial installs).
- `[ $? -ne 0 ] && export TERM=xterm-256color` — runtime install failure (read-only home, EPERM, unsupported `-x`). Falls back in the same shell session before the user gets a prompt. **Note**: `tic`'s stderr is *not* suppressed — on failure the user sees a one-line `tic: ...` diagnostic flash by before the prompt. This is intentional UX (silent failure would leave the user with no signal that something went sideways) and is consistent across all four failure rows in the §6 matrix.
- `exec "${SHELL:-/bin/sh}" -l` — replace wrapper script with user's login shell. `${SHELL:-/bin/sh}` quoted defensively (a path with spaces is rare but possible on some weird systems and free to handle). `-l` requests login-shell behavior, supported by Bash/Zsh/Dash/Ash/BusyBox sh; on truly pure POSIX `sh` (extremely rare in practice) `-l` is unrecognized — the shell may error or ignore. The shell-matrix manual smoke (§7.3) covers Bash/Zsh/Dash/Ash/BusyBox; if a real-world deployment hits a non-supporting shell, that's a v1.6.x patch (drop `-l`, accept non-login semantics).

**Why no `set -e` / no `trap`:** every failure is locally handled (each branch ends in either `tic` success, `export TERM=xterm-256color`, or skipping the block). `set -e` would abort the wrapper on the `infocmp` exit-1 path (the "not installed" detection itself).

**Why no `[ -t 0 ]` / `tput init`:** SSH `-t` already allocates a TTY. The wrapper executes inside that TTY.

### 5.2 SSHCommandBuilder integration

Current builder ends with:
```
ssh ... -p 22 'user'@'host'
```

With `installTerminfo: true` AND `terminfoDump()` returning non-nil:
```
ssh ... -p 22 -t 'user'@'host' '<wrapper-string>'
```
…and `Output.env` gains `("TERM", "xterm-ghostty")`.

With `installTerminfo: true` BUT `terminfoDump()` returning nil (build/CI failure didn't catch a bundling regression):
- Output equals `installTerminfo: false` shape (no `-t`, no remote command)
- `Output.env` does **not** add the TERM override
- Caller logs once via `NSLog`
- Net effect: toggle silently no-ops; remote sees `xterm-256color` (from managed config) and works. No latent "advertised xterm-ghostty but no terminfo on remote" trap.

This combined with the **CI gate** in §7.1 ensures the bundle-missing path is never reached in shipped builds — the test build fails when `Resources/xterm-ghostty.terminfo` is empty or absent.

The wrapper string is built as:

```swift
let wrapper = """
if ! infocmp xterm-ghostty >/dev/null 2>&1; then
  if command -v tic >/dev/null 2>&1; then
    tic -x - <<'TERMINFO_EOF'
\(terminfoDump)
TERMINFO_EOF
    [ $? -ne 0 ] && export TERM=xterm-256color
  else
    export TERM=xterm-256color
  fi
fi
exec "${SHELL:-/bin/sh}" -l
"""
```

`ShellQuote.posix` escapes embedded single quotes correctly (the `'\''` dance), so the wrapper — including the heredoc body — is safe inside the SSH command's outer single quotes. **However**, `ShellQuote.posix` was sized for short strings (usernames, hostnames). The integration test (§7.2) exercises the full pipeline by parsing the assembled command through a real `/bin/sh -c` and asserting the inner `ssh` argv matches expectation, not just round-tripping the wrapper string through `posix(...)` and back.

### 5.3 ConfigStore — no changes from v1.5

The v1.5 `ConfigStore.managedConfigContent` and `writeManagedConfig` stay exactly as committed in `d73359f`. The managed config always pins `term = xterm-256color`. The toggle does not rewrite the file; `SyncPreferences.installTerminfoEnabled.didSet` does **not** call `ConfigStore`. This eliminates:

- The need for `HostSyncStore` (which owns `SyncPreferences`) to depend on `ConfigStore` in its `Package.swift` target deps.
- A whole class of "did the running app pick up the new term= line" bugs (it can't — `GhosttyApp.shared` is a process-singleton, see §3).

The TERM override flows entirely through SSHCommandBuilder.Output.env into the PTY env-vars libghostty propagates into the SSH subprocess. Local libghostty rendering of non-SSH commands continues to use the managed config's `xterm-256color` regardless of toggle state.

### 5.4 Toggle storage

Add to `SyncPreferences`:

```swift
@Published public var installTerminfoEnabled: Bool {
    didSet {
        userDefaults.set(installTerminfoEnabled, forKey: Self.installTerminfoKey)
    }
}

private static let installTerminfoKey = "com.caterm.preferences.installTerminfoEnabled"
```

Hydrate in `init`:

```swift
self.installTerminfoEnabled = userDefaults.bool(forKey: Self.installTerminfoKey)
// bool(forKey:) returns false when key is absent — that's the
// default we want, no fallback needed.
```

`didSet` only persists. No `ConfigStore` interaction (see §5.3). The view tree observes `@Published` and re-renders; the next `SSHCommandBuilder.build` call reads the new value via the threaded parameter. Naming mismatch with `SyncPreferences` documented in §2 non-goals — accepted as v1.7 cleanup.

### 5.5 Session spawn site

Verified call chain (from `SessionStore.swift:124`, `TerminalContainerView.swift:40,41`):

```
TerminalContainerView (View, has @EnvironmentObject store: SessionStore)
   └── TerminalSurfaceRepresentable (NSViewRepresentable, has @EnvironmentObject store)
         └── makeNSView(context:)
               └── store.surfaceConfig(for: tabId)
                     └── SSHCommandBuilder.build(host:askpassPath:knownHostsCaterm:knownHostsUser:)
```

Changes:

1. `TerminalSurfaceRepresentable` adds `@EnvironmentObject var preferences: SyncPreferences`. Reads `preferences.installTerminfoEnabled` synchronously inside `makeNSView`. Passes the value as a new arg to `surfaceConfig`.
2. `SessionStore.surfaceConfig(for:)` adds an `installTerminfo: Bool` parameter, forwarded to `SSHCommandBuilder.build`.
3. `SyncPreferences` is already injected at the WindowGroup level (per v1.4 — see `apps/macos/Sources/Caterm/Views/SyncStatusRow.swift:11`), so the env-object is already in scope; adding it to the representable is a one-line declaration.

`SessionStore` does **not** gain a property reference to `SyncPreferences`. Read-then-pass at the representable layer keeps SessionStore decoupled.

### 5.6 Settings UI

In `SyncSettingsView`, add a new section **after** the existing `Section("Sync")` (file currently has `Section("Server")`, `Section("Account")`, `Section("Sync")`):

```swift
Section("Terminal") {
    Toggle("Install Ghostty terminfo on remote hosts",
           isOn: $preferences.installTerminfoEnabled)
    Text("Provides full Ghostty rendering features (true colors, hyperlinks). Falls back to standard terminfo automatically if installation isn't possible.")
        .font(.caption)
        .foregroundStyle(.secondary)
}
```

No "explain what terminfo is" essay — the caption is two sentences, focused on outcome.

## 6. Failure mode matrix

| # | Trigger | Wrapper behavior | User-visible effect |
|---|---|---|---|
| 1 | Already installed (subsequent connect) | Skip install block | Silent; ~10ms `infocmp` check |
| 2 | Fresh install succeeds | `tic -x -` writes `~/.terminfo/x/xterm-ghostty` | Silent; ~50ms `tic` write |
| 3 | `tic` not present | `command -v tic` fails | Silent; `TERM=xterm-256color` in this session |
| 4 | `tic -x` flag unsupported | `tic` exits non-zero | One-line `tic: ...` stderr flashes before prompt; `TERM=xterm-256color` fallback |
| 5 | `tic` runtime fails (EROFS, EACCES) | Same as #4 | Same as #4 |
| 6 | Local terminfo bundle missing (build/CI bug — gated by §7.1) | SSHCommandBuilder skips wrapper *and* skips TERM env override (§5.2) | Effective toggle-OFF behavior; remote sees `xterm-256color` and works; no latent "advertised xterm-ghostty without terminfo" trap |
| 7 | Network/auth failure during connect | No different from baseline | Existing reconnect overlay handles |

## 7. Test surface

### 7.1 Unit (SwiftPM `swift test`)

| Test target | Scenario |
|---|---|
| `SSHCommandBuilderTests` | `installTerminfo: false` output equals current v1.5 baseline byte-for-byte (regression guard) |
| `SSHCommandBuilderTests` | `installTerminfo: true` adds `-t` flag, appends a quoted remote command containing `infocmp xterm-ghostty`, `tic -x -`, the dump, and `exec "${SHELL:-/bin/sh}" -l`; **and** `Output.env` contains `("TERM", "xterm-ghostty")` |
| `SSHCommandBuilderTests` | Wrapper's `<<'TERMINFO_EOF'` delimiter is single-quoted (asserts the exact literal in the emitted command) |
| `SSHCommandBuilderTests` | **End-to-end argv test**: assemble the full command, hand it to `Process(launchPath: "/bin/sh", arguments: ["-c", cmd + " --debug-print-argv"])` where the test fixture provides a stub `ssh` script on PATH that prints its argv to stdout; assert the stub receives the expected sequence including the wrapper as a single argument. This validates that `ShellQuote.posix` survives a 3-KB heredoc-containing payload through real `/bin/sh` parsing — not just internal round-trips. (Resolves 8-OQ-2.) |
| `SSHCommandBuilderTests` | Bundle-missing fallback: when `terminfoDump()` returns nil, output equals `installTerminfo: false` shape **and** `Output.env` does not contain a `TERM` key |
| `TerminfoSourceTests` | `terminfoDump()` returns non-nil non-empty string with `xterm-ghostty\|` header — **CI gate**: this test failing is a build-blocker, ensures shipped builds always have a valid bundle |
| `SyncPreferencesTests` | `installTerminfoEnabled` round-trips through UserDefaults; `didSet` does **not** invoke `ConfigStore` (no shared spy needed — `ConfigStore` isn't a test dependency of `HostSyncStore`) |

### 7.2 Integration (Docker E2E, gated by `CATERM_E2E_DOCKER=1`)

| Scenario | Verify |
|---|---|
| Fresh GNU/Linux Alpine container with `ncurses` + toggle ON + connect | `~/.terminfo/x/xterm-ghostty` exists on remote post-connect; `infocmp xterm-ghostty` exits 0 on remote; second connect skips install (asserted via inotify or just file-mtime stability) |
| Fresh Alpine container *without* `ncurses` (no `tic`) + toggle ON + connect | Session opens; remote `echo $TERM` returns `xterm-256color`; no `tic: not found` blocks the prompt |
| Toggle OFF + connect to either container | `~/.terminfo` does not exist on remote post-connect (zero remote mutation, regression guard); `echo $TERM` returns `xterm-256color` |

(Resolves 8-OQ-1: GNU/Linux container parsing the bundled portable terminfo source with GNU `tic` is the gate. If this fails, the bundle source is wrong and must be regenerated from a more portable origin.)

### 7.3 Manual smoke

1. Toggle ON, reconnect to GEN2 (the user's host that originally surfaced `'xterm-ghostty': unknown terminal type`): `clear`, `vim`, `tmux new`, `ls --color` all work.
2. Toggle ON, immediately toggle OFF, reconnect: `echo $TERM` returns `xterm-256color`; remote `~/.terminfo/x/xterm-ghostty` from the prior connect is still there but unused (acceptable — we don't clean up).
3. Toggle ON, reconnect to a known-broken host (e.g. `docker run alpine:latest` with no `ncurses`): session opens, prompt appears, `echo $TERM` returns `xterm-256color`.
4. **Shell matrix**: with toggle ON, log in to hosts whose login shell is each of: bash, zsh, dash, ash, busybox `sh`. Verify in each: shell prompt appears (login init ran), `echo $TERM` returns `xterm-ghostty`, and `clear` works. If any shell errors out on `-l`, file as v1.6.x patch.

## 8. Open questions

- **8-OQ-1 (resolved → integration test gates):** Does the bundled terminfo source parse cleanly with GNU `tic` on a GNU/Linux remote? Resolution: **bundle is generated from upstream ghostty's portable terminfo source (not BSD `infocmp -x` output)**, and the §7.2 integration test on Alpine + ncurses is the merge gate. Bundle regen instructions in `Resources/README.md` document the portable source URL.
- **8-OQ-2 (resolved → end-to-end argv test):** Does `ShellQuote.posix` survive a 3-KB wrapper through real `/bin/sh` parsing? Resolution: §7.1 includes an end-to-end argv test that pipes the assembled command through `/bin/sh -c` against a stub `ssh` and asserts argv equality, not just internal round-trip.
- **8-OQ-3 (resolved → call site pinned):** Where is `SSHCommandBuilder.build(...)` called from? Resolution: only `SessionStore.surfaceConfig(for:)` at `apps/macos/Sources/SessionStore/SessionStore.swift:124`. v1.6 reads the preference in `TerminalSurfaceRepresentable.makeNSView` (which already has the env-object infrastructure from v1.4), passes through. No new property on `SessionStore`.

(All three v1.6 open questions resolved at spec time. The plan stage should not need a spike round.)

## 9. Decisions log

| # | Decision | Rationale |
|---|---|---|
| 1 | B' inline pattern, not pre-flight | Single auth, no state, self-healing. See brainstorming session. |
| 2 | Bundle terminfo source as resource, not shell out at runtime | Most users lack Ghostty.app; predictable build-time pinning. |
| 3 | Single global toggle, default OFF | YAGNI on per-host; user agreed default OFF respects "don't mutate remote without consent". |
| 4 | TERM override via SSHCommandBuilder.Output.env, **not** managed config rewrite | `GhosttyApp.shared` is a process-singleton; managed config rewrites have zero runtime effect (verified `apps/macos/Sources/TerminalEngine/GhosttyApp.swift:19,38`). Per-SSH-command env injection is the only path that takes effect within a running app. |
| 5 | Bundle-missing degrades to **toggle-off-equivalent** behavior (no `-t`, no env override) | Drops the latent issue from spec round-0 (`xterm-ghostty` advertised but no terminfo installed). Combined with §7.1 CI gate this should never ship, but defensive runtime fallback is cheap. |
| 6 | POSIX `/bin/sh` syntax in wrapper, `exec "${SHELL:-/bin/sh}" -l` ending | Maximum compat; defensive `$SHELL` quoting; `-l` for login-shell init; documented limitation on truly pure POSIX `sh` covered by manual shell-matrix smoke. |
| 7 | `<<'TERMINFO_EOF'` single-quoted delim | Prevents shell expansion in terminfo body (which contains escape sequences and `$<...>` delays). |
| 8 | `tic` non-zero **always** falls back, never partial install | If `-x` is unsupported, we don't retry without it — cleaner mental model than "dropped caps but partial works". |
| 9 | Bundle source generated from **upstream ghostty portable terminfo source**, not BSD `infocmp` reverse | Avoids known BSD/GNU `tic` parser incompatibilities for extended capabilities. Docker E2E gate (§7.2). |
| 10 | Add `installTerminfoEnabled` to `SyncPreferences` despite naming mismatch | Pragmatic; rename to `Preferences` deferred to v1.7. |
| 11 | `didSet` only persists to UserDefaults — no `ConfigStore` call | Avoids cross-target dependency; managed config doesn't change anyway (Decision #4). |
| 12 | `TerminalSurfaceRepresentable` reads preference, passes through to `SessionStore.surfaceConfig` | Keeps SessionStore decoupled from `SyncPreferences`; representable already has env-object infra (v1.4). |
| 13 | No "explain terminfo" body text in Settings | Two-sentence caption; users who care can read docs. |
| 14 | No clean-up on toggle-OFF (don't `rm` remote `~/.terminfo`) | Cleanup adds risk (deleting user's own files), little reward. Stale terminfo is harmless. |
| 15 | `tic` stderr is **not** suppressed | Silent failure on a write that mutates remote filesystem is worse UX than a one-line stderr flash before the prompt. Matrix #4/#5 honest about it. |
