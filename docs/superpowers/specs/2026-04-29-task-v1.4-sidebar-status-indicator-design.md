# v1.4 Sidebar Sync Status Indicator Design

**Date**: 2026-04-29
**Branch**: `barcelona-v1`
**Status**: Spec, ready for plan
**Builds on**: v1.3 (`docs/superpowers/specs/2026-04-28-task-v1.3-failure-visibility-design.md`)

## 0. Goal & Non-goals

**Goal**: Make sync state **always visible** in the main UI — not only inside the SyncSettings sheet — so a user with a stuck or failing sync notices without having to manually open settings.

v1.3 surfaced failure inside `SyncSettingsView` and added an opt-in notification. v1.4 closes the remaining gap: a user who keeps the toggle off and rarely opens settings can still have sync stuck failing for hours. An always-visible sidebar row is the persistent ambient layer.

**Non-goals (explicitly deferred)**:
- **Menu bar (NSStatusItem)** — deferred to v1.5. Rationale: Caterm's sync state only matters when the user is using Caterm; the system-wide indicator adds NSStatusItem lifecycle, popover NSWindow plumbing, and a "show in menu bar" preference toggle, none of which carry their weight in v1.4.
- **Configurable interval picker** — deferred. v1.3 §0 still defers it; no user demand yet.
- **Refresh token / silent re-auth** — deferred to v1.5 (depends on better-auth server-side support; v1.2 §8 carry-forward).
- **Per-op retry policy** — deferred to v1.5 (needs telemetry on real failure shapes).
- **Network reachability gating (`NWPathMonitor`)** — permanently deferred per v1.2 §2. One failed `URLSessionDataTask` is cheaper than polling reachability.
- **Per-host (host-row) sync state badges** — deferred to v1.5. The reconciler is atomic (all-or-nothing apply); per-host failure surface is a separate product concern.
- **Notification "granted-then-disabled-in-System-Settings" drift detection** — deferred to v1.5 (v1.3 §6.5 carry-forward).
- **In-flight cancel button in popover** — deferred. `Sync Now` and `Try Again` start a sync; cancelling mid-flight is a separate concern.

### Data flow (per cycle, see §3 for full enumeration)

1. App launches → `HostListSidebar` mounts → `SyncStatusRow` reads `syncStore.isSyncing` / `failingSince` / `lastSyncErrorKind` / `lastSyncedAt` and `authSession.isSignedIn` through `TimelineView(.periodic(by: 30))`.
2. `syncIndicatorState(...)` derives one of 5 cases. Row body switches on the case.
3. User taps row:
   - `.signedOut` → calls `openSettingsSheet()` (sets a `@Binding settingsSheetPresented = true` provided by `CatermApp`).
   - `.failing(reason: .auth, ...)` → same path as `.signedOut`.
   - `.healthy` / `.failing(reason: .other, ...)` → toggles a local `@State popoverPresented = true`.
   - `.syncing` → toggles popover (popover shows a static "Syncing…" line, no buttons).
4. Popover `Sync Now` / `Try Again` button calls `Task { try? await syncStore.sync() }`. Popover stays open through the sync transition; auto-closes 1 second after the sync resolves (success or failure).

### Invariants

- **State priority**: `signedOut > syncing > failing > healthy`. Tested explicitly (see §5.3).
- **`isSyncing == true` masks the failing state in the row** but does NOT clear `failingSince` in the store (v1.3 store-side semantics preserved). When sync completes, the row falls back to `healthy` (success) or `failing` (still broken).
- **`TimelineView(.periodic(by: 30))` is load-bearing for "Synced 2m ago" and "Sync failed · 8m"** (parallel to v1.3 §3.3): without it, `formatLastSynced(...)` and `formatRelativeShort(...)` resolve at body construction and the relative phrase freezes between renders.
- **Tap routing for `.signedOut` and `.failing(reason: .auth, ...)` skips popover entirely** — the only useful action is "open settings to sign in", and a popover with one button is worse than direct sheet present.
- **Healthy state visual weight is intentionally low** (secondary gray, not green). v1.4 narrative is "show me the failure"; an attention-grabbing healthy state would invert priorities.
- **`SyncIndicatorState` lives in the `HostSyncStore` SwiftPM target**, NOT in the Caterm app target — the free function `syncIndicatorState(...)` is unit-tested without a SwiftUI harness, parallel to how `syncFailureState` is tested in v1.3.
- **No new persisted state**. `isSyncing` is transient (live property); all other inputs (`lastSyncedAt`, `failingSince`, `lastSyncErrorKind`) are already persisted or derived per v1.2/v1.3.

## 2. Components

### 2.1 `HostSyncStore` extension

#### 2.1.1 New `@Published` property

Add to the existing block of published properties (`apps/macos/Sources/HostSyncStore/HostSyncStore.swift` around line 95):

```swift
/// True while a `sync()` pass — manual or auto — is in flight on this store.
/// Single flag (not split per-trigger) because the indicator only cares about
/// "is anything happening right now". Toggled at task boundaries; `defer`
/// guarantees it resets on throw.
@Published public private(set) var isSyncing: Bool = false
```

#### 2.1.2 Toggle site (single point)

`isSyncing` is toggled in **one place only**: inside `private func startSync() -> Task<Void, Error>` (around line 254). Public `sync()`, `syncIfSignedIn()`, `scheduleAutoSync()`, and `handleSystemWake()` all funnel into `startSync()`, so a single toggle covers every entry. **Do not edit `sync()`** — its existing body (which calls `try await startSync().value` and replays `pendingAutoAfterManual` in `defer`) is correct as-is.

The chained "cancel-and-drain" pattern in `startSync()` is the load-bearing constraint. Naive per-task toggling (`isSyncing = true` at task entry, `isSyncing = false` in `defer`) breaks: when a new sync arrives, the new task starts (sets `true`), `prev?.cancel()` causes the old task to exit, and the **old task's `defer` flips `isSyncing = false` while the new task is mid-flight** — UI shows "not syncing" while a sync is actually running. This is the failure mode the review caught.

**Solution: generation token gate.** Each call to `startSync()` increments a generation counter; only the latest generation's `defer` is permitted to clear `isSyncing`:

```swift
private var syncGeneration: Int = 0

@discardableResult
private func startSync() -> Task<Void, Error> {
    let prev = inFlight
    syncGeneration += 1
    let myGeneration = syncGeneration
    isSyncing = true
    let new = Task { [weak self] in
        guard let self else { return }
        defer {
            Task { @MainActor [weak self] in
                guard let self else { return }
                // Only the latest startSync() generation may clear the flag.
                // An older task draining after being cancelled must not toggle
                // isSyncing while a newer task is still running.
                if self.syncGeneration == myGeneration {
                    self.isSyncing = false
                }
            }
        }
        prev?.cancel()
        _ = await prev?.result   // existing drain
        try Task.checkCancellation()
        try await self.performSync()
    }
    inFlight = new
    return new
}
```

**Invariants this preserves**:
- A single sync pass: `isSyncing` flips `false → true` at start, `true → false` after `defer`. ✓
- Chained sync (mutation arrives while previous is in flight): generation increments, `isSyncing` stays `true` across the cancel-drain handoff. The cancelled prev task's defer runs but `syncGeneration != myGeneration_prev` so it skips the clear. The new task's defer eventually clears. ✓
- Throw or cancellation: `defer` still runs; gate logic still correct. ✓

**Why not active-count**: an active counter (`+1` at startSync, `-1` in defer, clear when 0) achieves the same outcome but is more fragile under unbalanced increments (e.g. early `return` between `+1` and the Task body). Generation token is more robust because the token is immutable per call.

**Why not toggle in `sync()` separately**: `sync()` already calls `startSync()`. Toggling in both places would double-fire `isSyncing = true` (harmless idempotent) but require coordinated clearing across two layers — the source of the original review concern. One toggle site, one defer.

#### 2.1.3 `isAuthShape` free function (lift refactor)

Currently the auth-error 4-case shape is duplicated:
- `HostSyncStore.classifySyncError(_:)` (private method, around line 314) — switches on the error to decide `.auth` vs `.other`.
- `SyncSettingsView.isAuthFailure(_:)` (private function, around `apps/macos/Sources/Caterm/Views/SyncSettingsView.swift` line 24) — duplicates the same 4 cases against `ServerSyncError`.

Lift to a single free function in the `HostSyncStore` module. Note the actual `ServerSyncError` cases use `status:` (not `httpStatus:`) and `authFailed(code:message:)` (not bare `.authFailed`) — see `apps/macos/Sources/ServerSyncClient/SyncErrors.swift`:

```swift
import ServerSyncClient

/// Returns true if the given `ServerSyncError` is the "401 / unauthenticated"
/// shape that should classify as `SyncErrorKind.auth`. Single source of truth —
/// both the store's `classifySyncError` and the view layer's auth-failure
/// check go through here. New auth-shape cases (e.g. a future server-side
/// token-expired code) are added here once.
///
/// Takes `ServerSyncError` (not `Error`) because:
/// - The view-layer call site has a typed `ServerSyncError?` (per
///   `SyncSettingsView.accountState(... lastSyncError: ServerSyncError?)`).
/// - The store's `classifySyncError(_ error: Error)` does the
///   `error as? ServerSyncError` cast at the call site, then calls this
///   function. Non-ServerSyncError types (URLError, etc.) are inherently
///   `.other` and never reach this function.
public func isAuthShape(_ error: ServerSyncError) -> Bool {
    switch error {
    case .http(status: 401, _),
         .orpc(_, status: 401, _),
         .authFailed,
         .notSignedIn:
        return true
    default:
        return false
    }
}
```

Update `classifySyncError` to call it:

```swift
private func classifySyncError(_ error: Error) -> SyncErrorKind {
    if let serverError = error as? ServerSyncError, isAuthShape(serverError) {
        return .auth
    }
    return .other
}
```

#### View-side swap (§2.6 detail)

Delete `SyncSettingsView.isAuthFailure(_:)`. Update the single `accountState(...)` call site:

```swift
// before:
if isAuthFailure(lastSyncError) || lastSyncErrorKind == .auth { return .sessionExpired }

// after:
if (lastSyncError.map(isAuthShape) ?? false) || lastSyncErrorKind == .auth {
    return .sessionExpired
}
```

The `.map(isAuthShape) ?? false` form handles the `ServerSyncError?` → `Bool` lift cleanly. `isAuthShape` is `public` from the `HostSyncStore` module; `SyncSettingsView` already imports it via the existing `import HostSyncStore` (added in v1.3 round-5 fix).

### 2.2 `SyncIndicatorState` module

New file `apps/macos/Sources/HostSyncStore/SyncIndicatorState.swift`:

```swift
import Foundation

/// User-facing state of the sidebar sync indicator. Derived from a snapshot of
/// the store + auth session at the time of body evaluation. Priority order
/// (highest first): signedOut > syncing > failing > healthy.
public enum SyncIndicatorState: Equatable, Sendable {
    case signedOut
    case syncing
    /// Mirrors `SyncFailureState.failing` payload. Includes auth shape so the
    /// view can route `.auth` taps to the settings sheet directly.
    case failing(reason: SyncErrorKind, since: Date)
    /// `lastSyncedAt` may be nil for a freshly-signed-in user that has not yet
    /// completed a sync — the row shows "Never synced" (see §2.3).
    case healthy(lastSyncedAt: Date?)
}

/// Pure derivation. Tested without any SwiftUI harness.
///
/// Priority: signedOut > syncing > failing > healthy. The store's
/// `failingSince` may be set during a sync, but while `isSyncing` is true the
/// indicator shows `.syncing` — when the in-flight sync resolves, the store
/// will either clear `failingSince` (recovery) or keep it (still broken),
/// and the next render will reflect that.
public func syncIndicatorState(
    now: Date,
    isSignedIn: Bool,
    isSyncing: Bool,
    lastSyncedAt: Date?,
    lastSyncAttemptedAt: Date?,
    lastSyncErrorKind: SyncErrorKind?,
    failingSince: Date?,
    periodicSyncEnabled: Bool,
    failingThreshold: TimeInterval
) -> SyncIndicatorState {
    if !isSignedIn { return .signedOut }
    if isSyncing { return .syncing }
    let failure = syncFailureState(
        now: now,
        lastSyncedAt: lastSyncedAt,
        lastSyncAttemptedAt: lastSyncAttemptedAt,
        lastSyncErrorKind: lastSyncErrorKind,
        periodicSyncEnabled: periodicSyncEnabled,
        failingThreshold: failingThreshold
    )
    switch failure {
    case .normal:
        return .healthy(lastSyncedAt: lastSyncedAt)
    case .failing(let reason, let attemptedSince):
        // Prefer `failingSince` (timestamp of FIRST failure of this run) over
        // `attemptedSince` (most recent attempt). Without this, retry-failed
        // cycles would advance the displayed "stuck N minutes" — the opposite
        // of what we want. SyncSettingsView already does the same fallback at
        // line 126 (`failingSince ?? attempted`) — Decision #21 makes the
        // pattern explicit at the derivation layer.
        let stableSince = failingSince ?? attemptedSince
        return .failing(reason: reason, since: stableSince)
    }
}
```

**Why this signature mirrors `syncFailureState` (v1.3 §2)**: the indicator delegates threshold / `periodicSyncEnabled` gating to v1.3's free function so semantics stay identical (Decision #10). Adding a second derivation here would risk drift. The `failingSince` parameter is layered on top of `syncFailureState`'s output to preserve "stuck N minutes" stability across retry-failed cycles (Cycle 7 §3).

### 2.3 `SyncStatusRow` view

New file `apps/macos/Sources/Caterm/Views/SyncStatusRow.swift`:

```swift
import SwiftUI
import HostSyncStore

struct SyncStatusRow: View {
    @EnvironmentObject var syncStore: HostSyncStore
    @EnvironmentObject var preferences: SyncPreferences

    @State private var popoverPresented = false
    @State private var popoverAutoCloseTask: Task<Void, Never>?

    var body: some View {
        TimelineView(.periodic(from: .now, by: 30)) { context in

                periodicSyncEnabled: preferences.periodicSyncEnabled,
                failingThreshold: syncStore.periodicInterval
            )
            statusRowBody(state: state, now: context.date)
        }
    }

    @ViewBuilder
    private func statusRowBody(state: SyncIndicatorState, now: Date) -> some View {
        Button(action: { handleTap(state: state) }) {
            HStack(spacing: 8) {
                stateIcon(state)
                stateLabel(state, now: now)
                Spacer()
                stateChevron(state)
            }
            .padding(.horizontal, 12)
            .padding(.vertical, 8)
            .contentShape(Rectangle())
        }
        .buttonStyle(.plain)
        .background(Color.clear)
        .accessibilityLabel(accessibilityLabel(for: state, now: now))
        .popover(isPresented: $popoverPresented,
                 arrowEdge: .trailing) {
            popoverContent(state: state, now: now)
        }
    }

    private func handleTap(state: SyncIndicatorState) {
        switch tapAction(for: state) {
        case .openSettings:
            popoverPresented = false
            NotificationCenter.default.post(
                name: .catermOpenSyncSettings, object: nil)
        case .togglePopover:
            popoverPresented.toggle()
        }
    }

    // (icon / label / chevron / popoverContent / accessibility helpers below)
}
```

**Why `@EnvironmentObject` and not `@ObservedObject`**: per code review, `AuthSession` is **not** an `ObservableObject` (it's a plain `public final class` in the `ServerSyncClient` module). `@ObservedObject var authSession: AuthSession` would not compile. The fix is two-fold:

1. **Drop the `authSession` parameter entirely.** Add `public var isSignedIn: Bool { authSession.isSignedIn }` to `HostSyncStore` (a computed property that proxies the underlying `AuthSessionProtocol.isSignedIn`). The view reads `syncStore.isSignedIn`.

2. **Trust the indirect re-render path.** SwiftUI re-evaluates `body` whenever any subscribed `ObservableObject` publishes. `syncStore` is the subscribed object; reading `syncStore.isSignedIn` inside `body` picks up the current cookie state on every re-eval. Sign-in / sign-out boundaries always poke `syncStore` (sign-in → `syncIfSignedIn()` → `startSync()` → `isSyncing` flips; sign-out → `clearAuthError()` → `lastSyncErrorKind` flips), so the body re-evaluates and the `.signedOut` case derives correctly. This is the same indirect pattern `SyncSettingsView` already uses (line 71 reads `authSession.isSignedIn` without `@ObservedObject`).

Add to `HostSyncStore.swift` (alongside the other public computed properties):

```swift
/// Computed proxy so SwiftUI views can read sign-in state without holding a
/// direct AuthSession reference (AuthSession is not ObservableObject).
/// Re-render is driven by other @Published properties on this store; sign-in
/// / sign-out boundaries always touch one of them, so body re-evaluation
/// picks up the new value.
public var isSignedIn: Bool { authSession.isSignedIn }
```

`@EnvironmentObject` is appropriate for `syncStore` and `preferences` because both are `ObservableObject`. They are injected at the WindowGroup level (§2.4).

#### Tap-routing free function

Tap routing is the most likely place to write a regression silently — the `switch state { ... }` decides between "open settings sheet" and "toggle popover", and an off-by-one case branch breaks core behavior. Lift it to a pure function so it can be unit-tested without a SwiftUI harness:

```swift
public enum SyncStatusTapAction: Equatable, Sendable {
    /// Posts `.catermOpenSyncSettings`. Used for `.signedOut` and
    /// `.failing(reason: .auth, ...)` — both cases have a single useful
    /// action (sign in), and a popover with one button would be worse UX.
    case openSettings
    /// Toggles the row's `@State popoverPresented`. Used for
    /// `.healthy`, `.failing(reason: .other, ...)`, and `.syncing`.
    case togglePopover
}

public func tapAction(for state: SyncIndicatorState) -> SyncStatusTapAction {
    switch state {
    case .signedOut,
         .failing(reason: .auth, since: _):
        return .openSettings
    case .healthy,
         .failing(reason: .other, since: _),
         .syncing:
        return .togglePopover
    }
}
```

Lives in the same file as `SyncIndicatorState` (`SyncIndicatorState.swift`). Tested by `SyncStatusRowTests` (§5.6).

#### 2.3.2 Free formatters (testable without a view harness)

```swift
/// "Synced 2m ago" / "Synced just now" / "Never synced" — short relative form.
public func formatLastSynced(_ lastSyncedAt: Date?, now: Date) -> String {
    guard let lastSyncedAt else { return "Never synced" }
    let elapsed = now.timeIntervalSince(lastSyncedAt)
    if elapsed < 60 { return "Synced just now" }
    let formatter = RelativeDateTimeFormatter()
    formatter.unitsStyle = .abbreviated
    return "Synced \(formatter.localizedString(for: lastSyncedAt, relativeTo: now))"
}

/// "8m" / "1h" — short interval used in "Sync failed · 8m" suffix.
public func formatRelativeShort(since: Date, now: Date) -> String {
    let elapsed = max(0, now.timeIntervalSince(since))
    if elapsed < 60 { return "now" }
    if elapsed < 3600 { return "\(Int(elapsed / 60))m" }
    return "\(Int(elapsed / 3600))h"
}
```

Both live in `SyncStatusRow.swift`, both `public` so the `CatermTests` target can hit them without ViewInspector.

#### 2.3.3 Popover content

```swift
@ViewBuilder

                settingsSheetPresented = true

        case .syncing:
            Label("Syncing…", systemImage: "arrow.triangle.2.circlepath")
                .foregroundStyle(.secondary)

        case .signedOut, .failing(reason: .auth, since: _):
            // Should not be reached — these states open the sheet directly.
            EmptyView()
        }
    }
    .padding(14)
    .frame(minWidth: 240)

```

`formatNextSyncIn(now:)` reads `syncStore.lastSyncedAt + syncStore.periodicInterval - now`:
- If `lastSyncedAt == nil` or computed remainder ≤ 0, render `"soon"`.
- Otherwise render short form (`"13m"`, `"45s"`).

`describe(errorKind:)` returns `"Authentication required"` / `"Network or server error"` for `.auth` / `.other`.

#### 2.3.4 `triggerSync` + popover auto-close

```swift
private func triggerSync() {
    popoverAutoCloseTask?.cancel()
    Task { @MainActor in
        try? await syncStore.sync()
        popoverAutoCloseTask = Task { @MainActor in
            try? await Task.sleep(nanoseconds: 1_000_000_000)
            if !Task.isCancelled { popoverPresented = false }
        }
    }
}
```

When the user clicks `Sync Now` / `Try Again`:
1. Cancel any pending auto-close.
2. Start `sync()`. While in flight, the row outside the popover flips to `.syncing`; the popover content re-renders to show "Syncing…" because the same `state` switch covers it.
3. After sync resolves (success or failure), schedule a 1-second auto-close. If the user clicks again before then, the new `triggerSync` cancels the timer.

### 2.4 `HostListSidebar` integration

**Reality check (per code review)**: `HostListSidebar()` is currently constructed with **no parameters** at two call sites — `MainWindow.swift:19` and `CatermApp.swift:153` (LandingView). Its only state input is `@EnvironmentObject var store: SessionStore`, injected via the WindowGroup's `.environmentObject(store)` chain. There is no current pass of `syncStore` / `authSession` / `preferences`.

Two wiring choices to add the new dependencies:

## 3. Data Flow (per cycle)

### Cycle 1: Cold start, signed out

1. `CatermApp.init` constructs `syncStore` / `authSession` / `preferences`.
2. `authSession.isSignedIn == false` (no persisted cookie).
3. `HostListSidebar` mounts → `SyncStatusRow.body` runs → `syncIndicatorState(...)` returns `.signedOut`.
4. Row shows: `↗ Sign in to sync` (tint color).
5. User taps row → `handleTap(.signedOut)` → `settingsSheetPresented = true`.
6. Existing `.sheet` modifier presents `SyncSettingsView` with `SignInView` as the active form.

### Cycle 2: Sign in → first sync

1. User signs in successfully via `SignInView`.
2. `authSession.isSignedIn` flips → `.signedOut` → `.healthy(lastSyncedAt: nil)` momentarily ("Never synced").
3. v1.1 2.8 `onSignedIn` calls `syncStore.syncIfSignedIn()` → `scheduleAutoSync()` → `inFlight` task starts.
4. `assignIsSyncing(true)` runs on main actor → `@Published isSyncing = true` → row re-renders → `.syncing` → "Syncing…" with spinner.
5. `performSync()` succeeds → `lastSyncedAt = Date()` → `failingSince = nil` (no failure) → defer flips `isSyncing = false` → row re-renders → `.healthy(lastSyncedAt: <now>)` → "Synced just now".

### Cycle 3: Healthy → mutation triggers debounced auto-sync

1. User edits a host in `HostFormView` → `sessionStore.mutationsForSync` publishes.
2. v1.1 2.10 debounce (2s) fires `scheduleAutoSync()`.
3. Same flow as Cycle 2 step 4-5: row shows `.syncing`, then `.healthy`.
4. `TimelineView` 30s tick keeps "Synced 2m ago" advancing to "Synced 5m ago" without new sync events.

### Cycle 4: Auto-sync fails → threshold elapses → failing.other

1. Network drops. Periodic timer fires (15 min) → `scheduleAutoSync()` → `performSync()` throws (e.g. `URLError(.notConnectedToInternet)`).
2. v1.3 catch path: `lastSyncErrorKind = .other`, `failingSince = lastSyncAttemptedAt` (first failure of this run), rethrows.
3. `isSyncing = false`. Row re-renders.
4. `syncFailureState(...)` returns `.normal` because `now - lastSyncedAt < periodicInterval` (15 min). Row shows `.healthy("Synced 14m ago")` — **deliberately quiet** for the first 15 minutes (v1.3 `failingThreshold = periodicInterval` semantics).
5. Periodic timer fires again 15 min later → fails again. Now `now - lastSyncedAt > 15min` → `syncFailureState` returns `.failing(reason: .other, since: failingSince)`.
6. Row re-renders → `.failing(reason: .other, since: ...)` → "⚠ Sync failed · 15m" (red).
7. User taps row → `handleTap(.failing(reason: .other, ...))` → `popoverPresented.toggle() = true`.
8. Popover renders: "Last attempt failed 15m ago", "Most recent error: Network or server error", `[Try Again]`, `[Open Settings]`.

### Cycle 5: 401 mid-sync → failing.auth

1. Auto-sync hits `/api/rpc/sshHost/listHosts` and gets 401 (server session expired).
2. `classifySyncError(error)` calls `isAuthShape(error)` → returns true → `lastSyncErrorKind = .auth`.
3. v1.3 catch path: `failingSince = lastSyncAttemptedAt`. Rethrows. `isSyncing = false`.
4. Threshold elapses (15min) → next render → `syncFailureState` returns `.failing(reason: .auth, since: ...)` → indicator state becomes `.failing(reason: .auth, ...)`.
5. Row shows "🔒 Sign in again" (red).
6. User taps row → `handleTap(.failing(reason: .auth, ...))` → `settingsSheetPresented = true` (skips popover, see Decision #13).
7. v1.3 `onSignedIn` (or user dismisses) calls `clearAuthError()` → `failingSince = nil`, `lastSyncErrorKind = nil` → next render → `.healthy(...)` again.

### Cycle 6: User clicks `Try Again` in popover, succeeds

1. Indicator is `.failing(reason: .other, since: <past>)`. Popover is open.
2. User clicks `[Try Again]` → `triggerSync()` → `Task { try? await syncStore.sync() }`.
3. `sync()` body sets `isSyncing = true` → row + popover both re-render → `.syncing` → row shows "Syncing…", popover shows "Syncing…" (no buttons).
4. `performSync()` succeeds → `lastSyncedAt = Date()`, `failingSince = nil`, `lastSyncErrorKind = nil`. Defer flips `isSyncing = false`.
5. Row + popover re-render → `.healthy(lastSyncedAt: <now>)` → row shows "Synced just now", popover shows "Synced just now / Next sync in 15m / [Sync Now]".
6. After 1 second, `popoverAutoCloseTask` fires → `popoverPresented = false`. User has seen the success state.

### Cycle 7: User clicks `Try Again` in popover, fails again

1. Same as Cycle 6 steps 1-3.
2. `performSync()` throws → `lastSyncErrorKind = .other` (or `.auth`), `failingSince` unchanged (still the original first-failure timestamp). Defer flips `isSyncing = false`.
3. Row + popover re-render → still `.failing(reason: ..., since: <same>)`. The "since" timestamp does NOT advance — this preserves the "stuck N minutes" semantics from v1.3.
4. After 1 second, `popoverAutoCloseTask` fires → `popoverPresented = false`. The user retries by tapping again.

### Edge matrix

| isSignedIn | isSyncing | failingSince | lastSyncErrorKind | now-lastSyncedAt | indicator state |
|---|---|---|---|---|---|
| false | * | * | * | * | `.signedOut` |
| true | true | * | * | * | `.syncing` |
| true | false | nil | * | < 15m | `.healthy` |
| true | false | set | .other | > 15m | `.failing(.other, since)` |
| true | false | set | .auth | > 15m | `.failing(.auth, since)` |
| true | false | set | * | < 15m | `.healthy` (failure subthreshold, v1.3 semantics) |
| true | false | set | nil | > 15m | `.failing(.other, since)` (v1.3 fallback) |

## 4. Error Handling & Edge Cases

### 4.1 Tap during in-flight sync

If the user taps the row while `.syncing`, `popoverPresented.toggle()` opens a popover showing static "Syncing…". No buttons. When the sync resolves, the popover content auto-updates to the new state (`.healthy` or `.failing`) without closing — the user sees the transition.

### 4.2 Tap during popover-already-open

`popoverPresented.toggle()` closes the popover. No side effect on `syncStore`. The user can re-open with a second tap.

### 4.3 Concurrent `Sync Now` clicks

If the user double-taps `[Sync Now]` faster than `isSyncing` flips to true, both clicks call `syncStore.sync()`. The store's existing `if let existing = currentManualTask` branch handles this — second caller awaits the same task. `isSyncing` is already true from the first call. `popoverAutoCloseTask` is cancelled and re-scheduled on each call.

### 4.4 Sheet binding conflict between `⌘⇧,` and tap

Both paths set `settingsSheetPresented = true`. `.sheet(isPresented:)` is idempotent — flipping a true binding to true is a no-op. If the sheet is already open and the user taps a `.signedOut` row from another window's sidebar, nothing changes (sheet stays open). Verified by manual smoke test (§5.7).

### 4.5 Multi-window: each `HostListSidebar` mounts its own `SyncStatusRow`

Per v1.1 2.8 + Phase 1.5, Caterm supports multiple windows via `NSWindow native multi-tab + ⌘N`. Each window mounts its own `HostListSidebar` and therefore its own `SyncStatusRow`. All rows read the same `syncStore` (`@StateObject` lives in `CatermApp`). All rows update synchronously when `@Published` properties change. Popover state (`@State popoverPresented`) is per-row — opening a popover in window A does not affect window B. This is the correct behavior: each window has its own click affordance.

### 4.6 `TimelineView(.periodic(by: 30))` jank

macOS 14+ `TimelineView` is the standard mechanism. With one `TimelineView` per window's sidebar (typically 1-3 windows max), the 30-second tick is negligible. Validated empirically in v1.2 / v1.3 settings view; v1.4 sidebar usage is parallel.

### 4.7 Popover auto-close cancelled by user re-tapping

If the user clicks `Sync Now`, the auto-close task is scheduled. If the user clicks `Sync Now` again before the 1s elapses, the new `triggerSync` cancels the pending task and schedules a new one. If the user clicks anywhere outside the popover, AppKit's automatic dismissal closes the popover and the auto-close timer becomes a no-op (`if !Task.isCancelled` — but even if it fires, `popoverPresented = false` is idempotent against an already-closed popover).

### 4.8 `lastSyncedAt == nil` in healthy state

A freshly signed-in user has `lastSyncedAt == nil` until the first sync completes. `formatLastSynced(nil, now:)` returns `"Never synced"`. The popover (if opened) shows "Never synced" + "Next sync in N min" using `lastSyncedAt + periodicInterval`. If `lastSyncedAt` is nil, "Next sync in" is computed against the periodic timer's next fire (best-effort: render "soon" if we can't compute exactly).

### 4.9 Server session expired but user is "signed in" locally

This is the v1.3 `.failing(reason: .auth, ...)` case. The auth session cookie is still on disk but the server rejects it with 401. `authSession.isSignedIn` returns true (we have a cookie), but every sync request fails. v1.4 keeps the v1.3 model: the indicator surfaces this as `.failing(reason: .auth, ...)` after threshold, and tapping opens settings. v1.3's `clearAuthError` semantics still apply (called from `Sign Out` and `onSignedIn`).

### 4.10 Indicator under reduced motion / dynamic type

`ProgressView` respects `.reduceMotion` automatically. The row's body uses standard SwiftUI `.font(.caption)` which respects dynamic type. No custom motion or sizes — the row inherits accessibility settings.

### 4.11 Manual sync started elsewhere while popover is open

If `syncStore.sync()` is invoked from a different code path (e.g. v1.4.5 `⌘R Refresh` shortcut, future scope) while the popover is open and showing `.healthy`, `isSyncing` flips to true → popover content re-renders to "Syncing…". User sees the transition without clicking anything. No bug.

```swift
final class FakeServerSyncClient: ServerSyncClient {
    /// Test sets this before invoking sync. When the store calls listHosts,
    /// the fake parks on the continuation. Test asserts mid-flight state,
    /// then calls `releaseListHosts()` to let the call return.
    private var parkedListHostsCont: CheckedContinuation<Void, Never>?
    var listHostsDidPark: (@Sendable () -> Void)?

    func listHosts(...) async throws -> ... {
        if listHostsDidPark != nil {
            await withCheckedContinuation { cont in
                parkedListHostsCont = cont
                listHostsDidPark?()
            }
        }
        return /* whatever the test pre-loaded */
    }

    func releaseListHosts() {
        parkedListHostsCont?.resume()
        parkedListHostsCont = nil
    }
}
```

## 7. Open Questions

- **Q1**: Should the row be hidden when the sidebar is otherwise empty (no hosts)? — Decision: NO. The empty-sidebar state is the most common "first run signed-out" case, and the row's `Sign in to sync` CTA is exactly the right onboarding affordance there.
- **Q2**: Should `formatNextSyncIn` show "now" if the periodic cycle is overdue? — YES, but only if `isSyncing == false` (an in-flight sync is not "overdue"). The popover only renders this for `.healthy` so this is well-defined.
- **Q3**: Does v1.4 need a `disabled` visual treatment for the row when `periodicSyncEnabled == false`? — NO. With `periodicSyncEnabled == false`, `syncFailureState` already returns `.normal`, so the row shows `.healthy(lastSyncedAt: …)` reflecting the most recent sync. The user toggled periodic off intentionally; the row showing the last sync time is correct ambient feedback. No special "paused" state in v1.4.

## 10. Locked Decisions

Listed in order of impact (highest first):

1. **Sidebar-only placement; menu bar deferred to v1.5.** Caterm's sync state matters when the user is using Caterm. Menu bar adds NSStatusItem lifecycle, popover NSWindow plumbing, and a "show in menu bar" preference; not justified in v1.4.

2. **Indicator priority order: signedOut > syncing > failing > healthy.** Tested explicitly in `SyncIndicatorStateTests`. Inverting any pair leads to confusing UX (e.g. showing "failing" while a sync is in flight implies the sync is hopeless).

3. **`SyncIndicatorState` lives in `HostSyncStore` SwiftPM target, not Caterm app.** Keeps the derivation testable without a SwiftUI view harness, parallel to `syncFailureState` (v1.3 §2.2.5). The view layer is dumb display + tap routing.

4. **Popover not inline disclosure expand.** macOS-native `NSPopover` is the right idiom; sidebar height is constrained and inline expand would push the host list. Popover lets us show 2-3 lines + buttons without crowding.

5. **`Sync Now` button in popover, not row.** Style C ("status panel" with always-visible Sync Now button) was rejected: it adds vertical sidebar real estate cost and creates a misclick risk. Two-step (tap row → click button) is the right flow for an action that hits the network.

6. **Lift `isAuthShape` to a free function in `HostSyncStore` module** (carry-forward from v1.3 §10 lock-step concern). Single source of truth for the 4-case auth shape. Both `classifySyncError` and `SyncSettingsView` go through it. New auth-shape cases (e.g. server-side `.tokenExpired`) are added once. Includes equivalence test in §5.5.

7. **Single `isSyncing` flag, not split per-trigger (manual vs auto).** The indicator only needs to know "is anything happening". Splitting would require the view layer to OR the two flags, adding complexity for no UX gain. The store internally still tracks `manualInProgress` separately for v1.3 notification gating; v1.4 introduces the unified `isSyncing` orthogonal to that.

8. **Healthy state visual weight is intentionally low** (secondary gray text, no green badge). v1.4 narrative is "show me failures"; an attention-grabbing healthy state would invert priorities. Failure breaks the visual norm — that's the design intent.

9. **Sheet binding owned by `CatermApp`**, threaded through `HostListSidebar` to `SyncStatusRow`. Both `⌘⇧,` and tap routing use the same binding. Idempotent against double-set.

10. **`failingThreshold == periodicInterval` carry-forward from v1.3.** v1.4's indicator uses the same threshold as v1.3 settings to avoid divergent gating. If v1.5 introduces an interval picker, this auto-links.

11. **`TimelineView(.periodic(by: 30))` mandatory** (carry-forward from v1.2 §3.3 / v1.3 §3.3). Without it, "Synced 2m ago" and "Sync failed · 8m" freeze. Tested by reading the spec — no automated way to assert TimelineView is present without snapshot tests.

12. **No host-level indicator** (Decision #1 of §0 non-goals lifted up here for visibility). The reconciler is atomic — per-host sync state is meaningless until per-host apply is introduced (separate concern).

13. **Tap routing**: `signedOut` and `failing(reason: .auth, ...)` open the sheet directly (no popover). `healthy`, `failing(reason: .other, ...)`, and `syncing` toggle the popover. Rationale: a popover with a single "Open Settings" button is worse than just opening settings.

14. **Popover auto-close after 1 second on success or fail** (Cycle 6/7). Lets the user see the new state, then gets out of the way. Cancellable: subsequent button click cancels and re-schedules.

15. **Multi-window: each `SyncStatusRow` is independent for popover state, shared for sync state.** All rows update synchronously from `@Published` — no manual broadcast.

16. **`isAuthShape` accepts `Error`, returns Bool.** Doesn't classify into `.auth` / `.other` directly because the caller (`classifySyncError`) does the binary mapping. Keeps the lifted function single-purpose.

17. **No new persisted state.** All state is either pre-existing (`lastSyncedAt`, `failingSince`, `lastSyncErrorKind`) or transient (`isSyncing`). No new UserDefaults keys.

18. **`formatLastSynced` and `formatRelativeShort` are public free functions in `SyncStatusRow.swift`.** Lets `SyncStatusRowTests` (CatermTests target) hit them without a view harness, parallel to v1.3's `formatFailingSince`.

19. **Spec name shifted from "Sidebar / menu-bar status indicator" to "Sidebar status indicator".** Menu bar carved out of v1.4 scope; renaming makes the deferral explicit (Decision #1).

20. **No `disabled` visual state for paused periodic sync.** When `periodicSyncEnabled == false`, the row shows the last successful sync time as healthy ambient feedback. The user paused intentionally — no need to nag.

21. **`syncIndicatorState` accepts `failingSince` and prefers it over `syncFailureState`'s `attemptedSince`** for the `.failing` case payload. Without this carve-out, retry-failed cycles would advance the user-visible "stuck N min" timestamp because `lastSyncAttemptedAt` updates per attempt while `failingSince` stays anchored at the first failure of the run. `SyncSettingsView` already does this fallback inline (line 126: `failingSince ?? attempted`); v1.4 makes the pattern explicit at the derivation layer so both views share identical semantics.

22. **`isSyncing` toggled in one place — `startSync()` — gated by a generation token.** A naive per-task `defer { isSyncing = false }` breaks under the chained cancel-and-drain pattern: when a new sync arrives, `prev?.cancel()` causes the old task to drain, and its `defer` would clear `isSyncing` while the new task is still running. The generation gate (`if syncGeneration == myGeneration`) ensures only the latest call's defer can clear the flag. Active counters were considered but rejected as more fragile under unbalanced increments. **Public `sync()` is NOT edited** — its existing `try await startSync().value` path inherits the toggle naturally, including the `pendingAutoAfterManual` replay in its `defer`.

23. **`HostSyncStore.isSignedIn` computed proxy** lets views read sign-in state via the store's @Published surface without making `AuthSession` an `ObservableObject` (out of scope for v1.4). Sign-in / sign-out boundaries always touch other published properties (sign-in → `isSyncing` flips via `syncIfSignedIn`; sign-out → `lastSyncErrorKind` flips via `clearAuthError`), so SwiftUI body re-evaluation picks up the new value indirectly. Same pattern `SyncSettingsView` already uses (line 71).

24. **NotificationCenter `.catermOpenSyncSettings` for tap → sheet trigger** instead of threading a `Binding<Bool>` through `MainWindow` / `LandingView` / `HostListSidebar`. Matches the existing `.catermAddHost` / `.catermNewWindow` pattern. Avoids editing `MainWindow.swift` and `LandingView` body. CatermApp owns `@State showSyncSettings` and observes the notification.

25. **`isAuthShape` accepts `ServerSyncError` (not `Error`).** The view-side call site (`accountState`) has a typed `ServerSyncError?`; the store-side call site (`classifySyncError(_ error: Error)`) does the `as? ServerSyncError` cast at the top. Non-ServerSyncError types (URLError, etc.) are inherently `.other` and never reach `isAuthShape`. Narrower input type makes the function easier to reason about and test.

26. **`tapAction(for:)` free function lifted out of view** for tap-routing testability. Hand-coding `switch state` inside `handleTap` would put the most regression-prone control flow somewhere unit tests can't reach. Lifting to a pure function makes 5-state coverage trivial.
