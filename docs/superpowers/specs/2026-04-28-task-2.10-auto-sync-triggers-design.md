# Task 2.10 — Auto-sync Triggers (Design)

**Date:** 2026-04-28
**Author:** brainstorm session continuing Phase 2 v1.1
**Status:** approved
**Scope:** macOS Caterm app
**Phase:** 2.10 (follow-up to v1.1 baseline + 2.9)

## 1. Goal

Today, sync only happens when the user explicitly clicks "Sync Now" in `SyncSettingsView`. After 2.10, the app:

- (a) syncs once at app launch when the user is signed in
- (b) syncs after any user-driven local mutation (`addHost / updateHost / deleteHost`), with a 2 s debounce so rapid edits coalesce into a single round-trip

Auto-sync runs silently. Failures never surface in the main UI; users discover them only by opening Sync Settings (where the existing manual-sync error path is reused, with one new affordance for session expiration). This closes the freshly-launched-second-device dead-end (signed in, host list empty, must remember to click Sync Now) and the device-A→device-B propagation gap.

## 2. Non-goals

- **(c) Periodic background sync** every N minutes — deferred to a future plan. Justification: the (a)+(b) pair already covers the common multi-device flows; (c) would require timer state, app-lifecycle awareness (don't drain battery in background), and a settings toggle to disable it. Pure cost-vs-benefit.
- **Auto-sync settings toggle** — hard-wired on when signed in. Signing in *is* the consent. A user who doesn't want sync signs out.
- **Backoff / retry logic** — see §4.5.
- **Refresh-token / silent re-auth** — out of scope. 401 handling is "Sign In Again" UI per §4.3.
- **`lastSyncedAt` timestamp display** — not surfaced. If users report "I don't know if my devices are in sync", revisit in v1.2.

## 3. Architecture

### 3.1 Lifecycle: `HostSyncStore` becomes long-lived

`apps/macos/Sources/Caterm/CatermApp.swift:20` currently has:

```swift
@MainActor
private var syncStore: HostSyncStore {
    HostSyncStore(client: syncClient, sessionStore: store)
}
```

— a computed property that builds a fresh instance on every access. This is incompatible with debounced + de-duplicated sync because `inFlight` Task state, Combine subscriptions, and debounce timers must persist across mutation events.

`HostSyncStore` adopts `ObservableObject`, and `CatermApp` holds it as `@StateObject`, parallel to the existing `@StateObject store: SessionStore`. Both live for the app's whole lifetime.

**Load-bearing invariant (re-stated in §4.1):** if anyone reverts `syncStore` to a computed property, every cancellation/dedup guarantee in this design becomes nominal. The mutation-publisher tests pin the expected behavior, but a test failing because someone "fixed" the lifetime back to short-lived must be treated as a regression, not a flaky test.

### 3.2 Trigger source: a `Combine` subject on `SessionStore`

`SessionStore` exposes a `public let mutationsForSync = PassthroughSubject<Void, Never>()`. Three methods send on it: `addHost`, `updateHost`, `deleteHost` — and only after `HostPersistence.save(...)` returns, so failed persistence does not trigger sync.

The following methods explicitly **do not** send:

- `setCredentialOnly` — credential is a device-local overlay (per spec §7.1.2), never propagates, must not produce server ops
- `addRemoteHost` / `applyRemoteMetadata` / `setServerId` — these are the *outputs* of a sync pass. Sending here would create an echo loop where each apply triggers another sync

This split is the single most important correctness property in the design. It is enforced by a unit test per method (§5.1) so a future contributor can't add a `mutationsForSync.send()` to the wrong place silently.

### 3.3 Subscription: HostSyncStore subscribes with debounce

In `HostSyncStore.init`, the store subscribes to `sessionStore.mutationsForSync` with a `.debounce(for: .seconds(debounceInterval), scheduler: DispatchQueue.main)` operator. The sink calls `self.scheduleSync()`. Trailing 2 s window: 5 rapid `addHost`s within 2 s of each other coalesce into one sync, fired 2 s after the last edit. Subscription is stored in `cancellables: Set<AnyCancellable>`.

`debounceInterval` is a `TimeInterval` parameter on `HostSyncStore.init` with a default of `2.0`. Tests pass `0.05` and wait `~0.1 s` for fire — no scheduler-abstraction layer is introduced.

### 3.4 Concurrency guard: `inFlight: Task<Void, Error>?`

`scheduleSync()` calls `inFlight?.cancel()` (no-op when nil), then assigns `inFlight = Task { try? await self.sync() }`. At most one auto-sync task runs at a time. A new mutation arriving mid-sync cancels the running pass and starts a fresh one based on the latest `sessionStore.hosts` snapshot.

Manual `Sync Now` does not go through `scheduleSync()`. It calls `try await syncStore.sync()` directly — the resulting Task is not stored in `inFlight`, is never cancelled by the auto path, and propagates errors to `SyncSettingsView`.

### 3.5 Startup trigger: `WindowGroup.task { ... }` modifier

Adding `.task { await syncStore.syncIfSignedIn() }` to the root `Group` inside `WindowGroup` runs once when the view first appears and auto-cancels on disappear. `syncIfSignedIn()` is a sync-named-async wrapper that checks `authSession.isSignedIn` (returns immediately when not signed in) and calls `scheduleSync()` when signed in. We piggyback the same `inFlight` guard so a startup sync mid-flight is properly torn down if a user mutation arrives within 2 s of launch.

### 3.6 No echo loop, restated

- Startup sync calls `sync()` → `addRemoteHost` / `applyRemoteMetadata` / `setServerId` are invoked → these methods **do not** send `mutationsForSync` → debounce is not retriggered → no recursion.
- Post-mutation sync calls `sync()` → same chain → same outcome.

## 4. Components

### 4.1 `SessionStore.swift` (modify)

Add a public Combine subject and call `send()` from three methods.

```swift
public let mutationsForSync = PassthroughSubject<Void, Never>()
```

In `addHost`, `updateHost`, `deleteHost`, the trailing line — placed *after* `try HostPersistence.save(...)` and before the method returns — is:

```swift
mutationsForSync.send()
```

Rationale for placement: persistence failure throws *before* `send()` is reached, so the UI sees the error and the sync subsystem is not falsely told "host changed". `Combine` is already imported by this file (`SessionStore` is `ObservableObject`), so no new import.

**Load-bearing test:** `SessionStoreMutationPublisherTests` (§5.1) pins which methods do and don't emit. A future contributor adding a new mutation method must decide explicitly.

### 4.2 `HostSyncStore.swift` (modify)

Class declaration changes:

```swift
@MainActor
public final class HostSyncStore: ObservableObject {
    private let client: ServerSyncClient
    private let sessionStore: SessionStore
    private let authSession: AuthSession
    private var cancellables: Set<AnyCancellable> = []
    private var inFlight: Task<Void, Error>?

    public init(client: ServerSyncClient,
                sessionStore: SessionStore,
                authSession: AuthSession,
                debounceInterval: TimeInterval = 2.0) {
        self.client = client
        self.sessionStore = sessionStore
        self.authSession = authSession

        sessionStore.mutationsForSync
            .debounce(for: .seconds(debounceInterval),
                      scheduler: DispatchQueue.main)
            .sink { [weak self] in self?.scheduleSync() }
            .store(in: &cancellables)
    }

    /// Manual entry point. Throws on failure; SyncSettingsView surfaces the error.
    public func sync() async throws {
        let remote = try await client.listHosts()
        try Task.checkCancellation()
        let ops = HostSyncReconciler.reconcile(local: sessionStore.hosts,
                                                remote: remote)
        for op in ops {
            try Task.checkCancellation()
            try await apply(op)
        }
    }

    /// Startup entry point. No-op when signed out; otherwise schedule a sync.
    public func syncIfSignedIn() {
        guard authSession.isSignedIn else { return }
        scheduleSync()
    }

    private func scheduleSync() {
        inFlight?.cancel()
        inFlight = Task { [weak self] in
            try? await self?.sync()
        }
    }

    private func apply(_ op: SyncOperation) async throws {
        // ★ Critical invariant: do NOT insert Task.checkCancellation()
        // between client.createHost(...) and sessionStore.setServerId(...)
        // — see §4.2.1.
        switch op {
        case let .createRemote(localHostId):
            guard let host = sessionStore.hosts.first(where: { $0.id == localHostId }) else { return }
            let input = RemoteHostCreateInput(
                name: host.name, hostname: host.hostname,
                port: host.port, username: host.username
            )
            let out = try await client.createHost(input)
            try sessionStore.setServerId(out.id, for: localHostId)
        case let .createLocal(remote):
            try sessionStore.addRemoteHost(remote)
        case let .updateRemote(localHostId, serverId):
            guard let host = sessionStore.hosts.first(where: { $0.id == localHostId }) else { return }
            let input = RemoteHostUpdateInput(
                id: serverId, name: host.name, hostname: host.hostname,
                port: host.port, username: host.username
            )
            try await client.updateHost(input)
        case let .updateLocal(localHostId, remote):
            try sessionStore.applyRemoteMetadata(localHostId: localHostId, remote: remote)
        case let .deleteLocal(localHostId):
            try sessionStore.deleteHost(id: localHostId)
        }
    }
}
```

#### 4.2.1 Cancellation invariant in `apply(.createRemote)`

`Task.cancel()` in Swift Concurrency is *cooperative*: an `await` does not auto-throw on cancellation; the awaited operation must check cancellation itself. `URLSession.data(for:)` does respond to cancellation (translates to `NSURLErrorCancelled`), so `client.listHosts()` and `client.createHost(...)` will throw promptly when cancelled. Pure local methods like `setServerId` do not check.

The forbidden code is:

```swift
case let .createRemote(localHostId):
    let out = try await client.createHost(input)
    try Task.checkCancellation()                       // ★ NEVER PUT THIS HERE
    try sessionStore.setServerId(out.id, for: localHostId)
```

If a cancel arrives between `createHost` returning and `setServerId` being called, the server has the host but the local Host has no `serverId`. On the next sync the reconciler sees a local Host with no `serverId` and creates *another* server-side host — a duplicate. The window is small but not zero: cancellations can fire from arbitrary mutations, so this is a real concern.

The correct shape is the one in §4.2: cancel checkpoints between `listHosts` and the op loop, and between ops, but never *inside* `apply(.createRemote)` after the server call.

`apply(.createLocal)`, `apply(.updateLocal)`, and `apply(.deleteLocal)` are pure local writes and have no `await`, so cooperative cancellation can't fire mid-op. `apply(.updateRemote)` and `apply(.deleteLocal)` end with the await/local op respectively and have no second-step server-binding to protect.

### 4.3 `CatermApp.swift` (modify)

Three changes:

**1.** Replace the computed `syncStore` with `@StateObject`. Because `syncStore` depends on `store`, both `@StateObject` properties must be initialized in `init()`:

```swift
@StateObject var store: SessionStore
@StateObject var syncStore: HostSyncStore

init() {
    try? ConfigStore.ensureExists(at: ConfigStore.defaultPath)
    let session = SessionStore.makeForApp()  // existing makeStore() factory, renamed for clarity
    let auth = AuthSession(baseURL: ServerURL.current)
    let client = URLSessionServerSyncClient(baseURL: ServerURL.current)
    _store = StateObject(wrappedValue: session)
    _syncStore = StateObject(wrappedValue: HostSyncStore(
        client: client,
        sessionStore: session,
        authSession: auth
    ))
    self.authSession = auth
    self.syncClient = client
}
```

`makeStore()` already exists at `CatermApp.swift:148` as a private free function. Keep it that way — call it from `init()`. The pseudocode above shows `SessionStore.makeForApp()` for readability, but the actual implementation can just be `let session = makeStore()`. Note `_store = StateObject(wrappedValue: ...)` is the underscore-prefixed property-wrapper init: required because `@StateObject` cannot be assigned via the synthesized `self.store = ...` syntax in `init`.

The current `private let authSession = AuthSession(baseURL: ServerURL.current)` and `private let syncClient: ServerSyncClient = URLSessionServerSyncClient(baseURL: ServerURL.current)` (default-initialized at property declaration) become `private let authSession: AuthSession` and `private let syncClient: ServerSyncClient`, both assigned in `init()`. `authSession` stays the *concrete* type in CatermApp because `SyncSettingsView` consumes signIn/signOut methods on the concrete class. Passing the concrete `authSession` to `HostSyncStore.init(authSession: AuthSessionProtocol, ...)` works via structural conformance — Swift accepts a concrete value where a protocol type is required.

**2.** Add the `.task { ... }` modifier to the root view inside `WindowGroup`:

```swift
WindowGroup(for: UUID.self) { $tabId in
    Group {
        ...
    }
    .environmentObject(store)
    .background(OpenTabBridge(store: store))
    .task { await syncStore.syncIfSignedIn() }      // ← new
    .sheet(isPresented: $showSyncSettings) { ... }
}
```

`.task` runs once on appear, auto-cancels on disappear. Multi-window note: `WindowGroup` may instantiate the closure multiple times (one per window); `.task` will fire per-window. That's fine — `syncIfSignedIn()` calls `scheduleSync()`, which cancels any in-flight sync and starts a new one. Worst case is two windows opening within microseconds of each other → second one cancels the first's sync, then the second runs to completion. No correctness issue. (We don't want to put `.task` outside `WindowGroup` because there's no obvious place; per-window `.task` is acceptable.)

**3.** Update `syncStore` references where the SyncSettingsView is presented (line ~46) — no API change for the view itself other than the new `authSession` parameter on the sync store's init, which is internal to CatermApp.

### 4.4 `SyncSettingsView.swift` (modify, focused)

Two surface changes: error type widens to `ServerSyncError?`, Account section dispatches on a 3-state derivation.

```swift
@State private var lastSyncError: ServerSyncError?    // was String?
```

The display still works because `ServerSyncError` is `CustomStringConvertible`:

```swift
if let lastSyncError {
    Text(lastSyncError.description).foregroundColor(.red).font(.caption)
}
```

In `syncNow()`:

```swift
private func syncNow() async {
    isSyncing = true
    defer { isSyncing = false }
    lastSyncError = nil
    do {
        try await syncStore.sync()
    } catch let err as ServerSyncError {
        lastSyncError = err
    } catch {
        lastSyncError = .http(status: 0, body: "\(error)")  // fallback
    }
}
```

Account section becomes 3-state, driven by a free function `accountState(isSignedIn:lastSyncError:) -> AccountState`:

```swift
enum AccountState: Equatable {
    case signedOut
    case signedIn
    case sessionExpired
}

func accountState(isSignedIn: Bool, lastSyncError: ServerSyncError?) -> AccountState {
    if isAuthFailure(lastSyncError) { return .sessionExpired }
    return isSignedIn ? .signedIn : .signedOut
}

private func isAuthFailure(_ err: ServerSyncError?) -> Bool {
    guard let err else { return false }
    switch err {
    case .http(status: 401, _): return true
    case .authFailed:            return true
    case .notSignedIn:           return true
    default:                     return false
    }
}
```

The view renders:

```swift
Section("Account") {
    switch accountState(isSignedIn: authSession.isSignedIn,
                        lastSyncError: lastSyncError) {
    case .signedOut:
        Button("Sign In…") { showSignIn = true }
    case .signedIn:
        Button("Sign Out") { Task { await signOut() } }
            .disabled(isSigningOut)
    case .sessionExpired:
        VStack(alignment: .leading, spacing: 4) {
            Text("Session expired").font(.caption).foregroundColor(.secondary)
            Button("Sign In Again…") {
                Task {
                    try? await authSession.signOut()  // clear local cookie
                    showSignIn = true
                    lastSyncError = nil               // hide the red text once user moves
                }
            }
        }
    }
}
```

`accountState` and `isAuthFailure` live `internal` in `SyncSettingsView.swift` so the `CatermTests` target can `@testable import Caterm` and unit-test the derivation directly (per §5.3).

## 5. Data flow

### 5.1 Path A — Startup sync

```
CatermApp body first render
  → WindowGroup root Group .task fires
  → await syncStore.syncIfSignedIn()
       │
       ├─ authSession.isSignedIn == false
       │     → return; no client call; UI shows whatever the local hosts.json holds
       └─ true
             → scheduleSync()
                  ├─ inFlight?.cancel()        // typically nil at boot
                  └─ inFlight = Task { try? await sync() }
                                                    │
                                                    ├─ list → reconcile → apply ops
                                                    │     SessionStore mutations
                                                    │     (addRemoteHost / applyRemoteMetadata
                                                    │      / setServerId) DO NOT emit on
                                                    │     mutationsForSync → no echo
                                                    └─ throws → swallowed by try?
```

### 5.2 Path B — Post-mutation debounced sync

```
HostListSidebar Save → SessionStore.addHost (or updateHost / deleteHost)
  → try HostPersistence.save(...)
  → mutationsForSync.send()
       │
       └─ Combine debounce(2s, .main)
              │
              ├─ another send within 2s → reset timer
              └─ 2s silence → sink → scheduleSync()
                                        ├─ inFlight?.cancel()
                                        └─ inFlight = Task { try? await sync() }
```

### 5.3 Path C — Manual Sync Now (unchanged surface, error type widened)

```
user clicks Sync Now in SyncSettingsView
  → syncNow() async
  → try await syncStore.sync()
       ├─ success → lastSyncError = nil
       └─ throws ServerSyncError → lastSyncError = err
              → Account section may switch to .sessionExpired (when err is 401/auth)
```

### 5.4 Invariants

1. **No echo loop** — `addRemoteHost / applyRemoteMetadata / setServerId` never `send()`. Tested by `testRemoteApplyDoesNotEmit` (§5.1).
2. **Credential change does not sync** — `setCredentialOnly` never `send()`. Tested by `testSetCredentialOnlyDoesNotEmit` (§5.1).
3. **Manual sync is never cancelled by auto path** — manual goes through `try await syncStore.sync()` directly, never assigned to `inFlight`. Tested by `testManualSyncBypassesInFlight` (§5.2).
4. **At most one auto sync at a time** — `inFlight?.cancel()` before assignment. Tested by `testInFlightCancellationOnNewSchedule` (§5.2).
5. **`createHost` → `setServerId` is uninterruptible at the cancellation level** — no `Task.checkCancellation()` between them. Pinned by code review against §4.2.1; not a unit-testable property at the public API, but the code shape is small enough to enforce via review.
6. **No `mutationsForSync.send()` on persistence failure** — `send()` line is *after* `try HostPersistence.save(...)`. Tested implicitly by `testAddHostEmits` (which expects exactly one emit on success); a write failure throws before the send, leaving subscriber count at 0 (could be added as a 6th test if we want to pin it explicitly).

## 6. Error handling

### 6.1 Categorized table (recap of §4 of brainstorm)

| Path | Error class | Behavior |
|---|---|---|
| Startup / Post-mutation | offline / DNS / 5xx | `try?` swallows. UI silent. |
| Startup / Post-mutation | 401 / auth-failed | `try?` swallows. UI silent until user opens Sync Settings. |
| Startup / Post-mutation | `CancellationError` (new mutation interrupted) | `try?` swallows. New Task takes over. |
| Startup / Post-mutation | reconcile partial fail | `try?` swallows. Partial ops have already been applied; remaining ops will be picked up by the next sync. |
| Manual Sync Now | any | throws → `lastSyncError = err` → red `description` text. |
| Manual Sync Now | 401 / auth-failed / notSignedIn | as above + Account section flips to `.sessionExpired`. |

### 6.2 Auth invariant: `isSignedIn` lies after server-side expiration

`AuthSession.isSignedIn` reads `HTTPCookieStorage` and returns true as long as `better-auth.session_token` is present locally. The server can expire the session while the cookie still lives. Both auto paths therefore can fail with 401 even though `isSignedIn == true`; this is by design — auto paths swallow, and the user discovers it via Sync Settings.

The `accountState` derivation (§4.4) is the bridge: when *manual* Sync Now lands a 401, the view shows "Sign In Again…" instead of "Sign Out", short-cutting the otherwise-required two-step (Sign Out, then Sign In) recovery.

### 6.3 No backoff / retry loop

Triggers are bounded by `launch ∪ user-mutation ∪ manual-click`. There is no timer-driven retry. Therefore:

- N consecutive failures cost ≤ N × (one TCP-fast-fail or HTTP round trip), bounded by user activity.
- No server-side rate-limiting concerns from a runaway client.
- Silent failures don't accumulate visible noise.

If 2.x adds (c) periodic background sync, this design must be revisited — periodic + no-backoff would hammer a degraded server.

### 6.4 Cancellation propagation

`Task.cancel()` is cooperative. `await` calls in `sync()` that respond:

- `client.listHosts()` — `URLSessionDataTask` cancellation produces `NSURLErrorCancelled`, surfaced as a thrown error
- `client.createHost / updateHost / deleteHost` — same

Calls that don't auto-respond (and therefore need the explicit `Task.checkCancellation()` before them):

- `apply(_:)` body for `.createLocal / .updateLocal / .deleteLocal` — pure local SessionStore writes, no await

Per §4.2 / §4.2.1, `try Task.checkCancellation()` is placed:

- Once between `client.listHosts()` and the op loop — guarantees we don't apply a stale op set if cancelled during reconciliation
- Once at the top of each loop iteration — guarantees we don't start a new op after cancellation

It is **never** placed inside `apply(.createRemote)` between the server `createHost` and the local `setServerId`.

## 8. AuthSession protocol vs. subclass

`HostSyncStore` only needs `authSession.isSignedIn` (a `Bool`). Two ways to make it testable:

**Option A (chosen) — extract a protocol:**

```swift
public protocol AuthSessionProtocol: AnyObject {
    var isSignedIn: Bool { get }
}
extension AuthSession: AuthSessionProtocol {}
```

`HostSyncStore.init` takes `authSession: AuthSessionProtocol`. Tests pass a stub conforming to the protocol with no superclass init. Real app passes the existing `AuthSession`.

**Option B (rejected) — drop `final` on `AuthSession`:** subclassing forces tests to provide a working `URL` to `super.init` and inherit all the unrelated machinery (cookie storage, sign-in/out HTTP). Drift surface is wider; the protocol is a smaller, more honest API for what `HostSyncStore` actually depends on.

Net change: `AuthSession` stays `final`; new protocol added in the same file (`AuthSession.swift`); `HostSyncStore.init` uses the protocol type.
