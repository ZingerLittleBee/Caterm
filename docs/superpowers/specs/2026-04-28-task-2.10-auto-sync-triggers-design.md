# Task 2.10 ‚Äî Auto-sync Triggers (Design)

**Date:** 2026-04-28
**Author:** brainstorm session continuing Phase 2 v1.1
**Status:** approved
**Scope:** macOS Caterm app
**Phase:** 2.10 (follow-up to v1.1 baseline + 2.9)

## 1. Goal

Today, sync only happens when the user explicitly clicks "Sync Now" in `SyncSettingsView`. After 2.10, the app:

- (a) syncs once at app launch when the user is signed in
- (b) syncs after any user-driven local mutation (`addHost / updateHost / deleteHost`), with a 2 s debounce so rapid edits coalesce into a single round-trip

Auto-sync runs silently. Failures never surface in the main UI; users discover them only by opening Sync Settings (where the existing manual-sync error path is reused, with one new affordance for session expiration). This closes the freshly-launched-second-device dead-end (signed in, host list empty, must remember to click Sync Now) and the device-A‚Üídevice-B propagation gap.

## 2. Non-goals

- **(c) Periodic background sync** every N minutes ‚Äî deferred to a future plan. Justification: the (a)+(b) pair already covers the common multi-device flows; (c) would require timer state, app-lifecycle awareness (don't drain battery in background), and a settings toggle to disable it. Pure cost-vs-benefit.
- **Auto-sync settings toggle** ‚Äî hard-wired on when signed in. Signing in *is* the consent. A user who doesn't want sync signs out.
- **Backoff / retry logic** ‚Äî see ¬ß4.5.
- **Refresh-token / silent re-auth** ‚Äî out of scope. 401 handling is "Sign In Again" UI per ¬ß4.3.
- **`lastSyncedAt` timestamp display** ‚Äî not surfaced. If users report "I don't know if my devices are in sync", revisit in v1.2.

## 3. Architecture

### 3.1 Lifecycle: `HostSyncStore` becomes long-lived

`apps/macos/Sources/Caterm/CatermApp.swift:20` currently has:

```swift
@MainActor
private var syncStore: HostSyncStore {
    HostSyncStore(client: syncClient, sessionStore: store)
}
```

‚Äî a computed property that builds a fresh instance on every access. This is incompatible with debounced + de-duplicated sync because `inFlight` Task state, Combine subscriptions, and debounce timers must persist across mutation events.

`HostSyncStore` adopts `ObservableObject`, and `CatermApp` holds it as `@StateObject`, parallel to the existing `@StateObject store: SessionStore`. Both live for the app's whole lifetime.

**Load-bearing invariant (re-stated in ¬ß4.1):** if anyone reverts `syncStore` to a computed property, every cancellation/dedup guarantee in this design becomes nominal. The mutation-publisher tests pin the expected behavior, but a test failing because someone "fixed" the lifetime back to short-lived must be treated as a regression, not a flaky test.

### 3.2 Trigger source: a `Combine` publisher on `SessionStore`

`SessionStore` exposes the **read side** of an event stream ‚Äî a `public var mutationsForSync: AnyPublisher<Void, Never>` backed by a `private let mutationsForSyncSubject = PassthroughSubject<Void, Never>()`. Subscribers (today: `HostSyncStore`) get the publisher; nobody outside `SessionStore` can call `.send()` and forge a sync trigger. Three methods call `mutationsForSyncSubject.send()` internally: `addHost`, `updateHost`, `deleteHost` ‚Äî and only after `HostPersistence.save(...)` returns, so failed persistence does not trigger sync.

The following methods explicitly **do not** send:

- `setCredentialOnly` ‚Äî credential is a device-local overlay (per spec ¬ß7.1.2), never propagates, must not produce server ops
- `addRemoteHost` / `applyRemoteMetadata` / `setServerId` ‚Äî these are the *outputs* of a sync pass. Sending here would create an echo loop where each apply triggers another sync

This split is the single most important correctness property in the design. It is enforced by a unit test per method (¬ß5.1) so a future contributor can't add a `mutationsForSync.send()` to the wrong place silently.

### 3.3 Subscription: HostSyncStore subscribes with debounce

In `HostSyncStore.init`, the store subscribes to `sessionStore.mutationsForSync` with a `.debounce(for: .seconds(debounceInterval), scheduler: DispatchQueue.main)` operator. The sink calls `self.scheduleAutoSync()` (defined in ¬ß3.4). Trailing 2 s window: 5 rapid `addHost`s within 2 s of each other coalesce into one sync, fired 2 s after the last edit. Subscription is stored in `cancellables: Set<AnyCancellable>`.

`debounceInterval` is a `TimeInterval` parameter on `HostSyncStore.init` with a default of `2.0`. Tests pass `0.05` and wait `~0.1 s` for fire ‚Äî no scheduler-abstraction layer is introduced.

### 3.4 Concurrency guard: serialized cancel-and-drain + manual gate

A naive `inFlight?.cancel(); inFlight = Task { ... }` does **not** guarantee mutual exclusion: `Task.cancel()` is cooperative, and the cancelled task keeps running until it hits a checkpoint or a cancellation-aware `await`. During that window two tasks can both be inside `apply(_:)` ‚Äî fatal for `createRemote` where the same unsynced local host can be POSTed twice and end up with two different `serverId`s on the server.

The fix is **chained serialization with cancel-and-drain**:

```swift
private var inFlight: Task<Void, Error>?

private func startSync() -> Task<Void, Error> {
    let prev = inFlight
    let new = Task { [weak self] in
        guard let self else { return }
        prev?.cancel()
        _ = await prev?.result   // drain ‚Äî wait for previous to fully exit
        try Task.checkCancellation()  // we may have been replaced too
        try await self.performSync()
    }
    inFlight = new
    return new
}
```

Each new task waits for the previous task's `.result` (which always resolves ‚Äî to success, to thrown error, or to `CancellationError`) before doing its own work. There is no overlap between consecutive sync passes.

**Manual vs. auto coexistence** ‚Äî manual `Sync Now` and the auto path share the same chain, but the manual path is uncancellable from the user's perspective. Two flags coordinate this:

```swift
private var manualInProgress: Bool = false
private var pendingAutoAfterManual: Bool = false
```

- `scheduleAutoSync()` (called from the debounce sink and from `syncIfSignedIn`) checks `manualInProgress` first. If true, sets `pendingAutoAfterManual = true` and returns without scheduling. The mutation's effect is captured in `sessionStore.hosts`; manual's own reconcile may pick it up, and the post-manual replay covers the worst case where it doesn't.
- `sync()` (manual entry point) sets `manualInProgress = true` for its duration. Its `defer` flips it back to `false` and, if `pendingAutoAfterManual`, calls `scheduleAutoSync()` once.

This means:
- Auto running + user clicks Sync Now: manual cancels auto via the chain, drains it, runs its own pass; auto's CancellationError is internal to the auto path's `try?` and never surfaces.
- Manual running + user mutation arrives: scheduleAutoSync skips, sets `pendingAutoAfterManual`. After manual completes, one auto sync runs to capture any mutation that landed too late for manual's reconcile to see.
- Two manuals concurrently: prevented by `SyncSettingsView`'s existing `disabled(isSyncing)` on the Sync Now button.

The chained `await prev?.result` does not deadlock because `prev` was stored *before* assigning to `inFlight`; the new task captures it as a local immutable.

### 3.5 Startup trigger: `WindowGroup.task { ... }` modifier

`.task { syncStore.syncIfSignedIn() }` on the root `Group` inside `WindowGroup` runs once when the view first appears and **the modifier's own wrapping Task auto-cancels on disappear**. That is *not* the same as cancelling the actual sync work: `syncIfSignedIn()` is a synchronous (non-`async`) function that creates an unstructured `Task` via `scheduleAutoSync()`. Unstructured tasks are not children of the `.task` modifier; they outlive view disappearance.

This is intentional. **Cancellation ownership lives in `HostSyncStore`** (the `inFlight`-chain in ¬ß3.4), not in the SwiftUI view. View lifecycle is the right place to *fire* the trigger; HostSyncStore's chain is the right place to manage it from there. If a startup sync is mid-flight and a mutation arrives within 2 s, the mutation's debounced `scheduleAutoSync()` call will replace the startup sync via the chain, just like any auto-vs-auto sequence.

`syncIfSignedIn()` itself: checks `authSession.isSignedIn` (returns immediately when not signed in) and calls `scheduleAutoSync()` when signed in. Multi-window note: `WindowGroup`'s closure may instantiate per window, so `.task` can fire multiple times in rapid succession when the user opens additional windows. Each call goes through `scheduleAutoSync()` and is serialized by the chain ‚Äî at most one sync runs to completion. No correctness issue.

### 3.6 No echo loop, restated

- Startup sync calls `sync()` ‚Üí `addRemoteHost` / `applyRemoteMetadata` / `setServerId` are invoked ‚Üí these methods **do not** send `mutationsForSync` ‚Üí debounce is not retriggered ‚Üí no recursion.
- Post-mutation sync calls `sync()` ‚Üí same chain ‚Üí same outcome.

## 4. Components

### 4.1 `SessionStore.swift` (modify)

Add a private Combine subject + a public `AnyPublisher` (read-only outside the class), and call `send()` from three methods.

```swift
private let mutationsForSyncSubject = PassthroughSubject<Void, Never>()
public var mutationsForSync: AnyPublisher<Void, Never> {
    mutationsForSyncSubject.eraseToAnyPublisher()
}
```

In `addHost`, `updateHost`, `deleteHost`, the trailing line ‚Äî placed *after* `try HostPersistence.save(...)` and before the method returns ‚Äî is:

```swift
mutationsForSyncSubject.send()
```

Rationale for placement: persistence failure throws *before* `send()` is reached, so the UI sees the error and the sync subsystem is not falsely told "host changed". `Combine` is already imported by this file (`SessionStore` is `ObservableObject`), so no new import. Exposing only `AnyPublisher` (not the bare subject) prevents any other target from manufacturing fake sync triggers via `.send()`.

**Load-bearing test:** `SessionStoreMutationPublisherTests` (¬ß5.1) pins which methods do and don't emit. A future contributor adding a new mutation method must decide explicitly.

### 4.2 `HostSyncStore.swift` (modify)

Add `import Combine` at the top of the file (`AnyCancellable` and `ObservableObject`'s default conformance both require it; the existing file imports only Foundation/etc.).

Class declaration changes:

```swift
import Combine
import Foundation
import ServerSyncClient
import SSHCommandBuilder
import SessionStore

@MainActor
public final class HostSyncStore: ObservableObject {
    private let client: ServerSyncClient
    private let sessionStore: SessionStore
    private let authSession: AuthSessionProtocol
    private var cancellables: Set<AnyCancellable> = []
    private var inFlight: Task<Void, Error>?
    private var manualInProgress: Bool = false
    private var pendingAutoAfterManual: Bool = false

    public init(client: ServerSyncClient,
                sessionStore: SessionStore,
                authSession: AuthSessionProtocol,
                debounceInterval: TimeInterval = 2.0) {
        self.client = client
        self.sessionStore = sessionStore
        self.authSession = authSession

        sessionStore.mutationsForSync
            .debounce(for: .seconds(debounceInterval),
                      scheduler: DispatchQueue.main)
            .sink { [weak self] in self?.scheduleAutoSync() }
            .store(in: &cancellables)
    }

    // MARK: - Public entry points

    /// Manual entry point (Sync Now button). Drains any in-flight auto sync,
    /// then runs and propagates errors to the caller.
    public func sync() async throws {
        manualInProgress = true
        defer {
            manualInProgress = false
            if pendingAutoAfterManual {
                pendingAutoAfterManual = false
                scheduleAutoSync()
            }
        }
        let task = startSync()
        try await task.value
    }

    /// Startup entry point. No-op when signed out; otherwise schedule a sync.
    /// Synchronous (non-async) ‚Äî the `.task` modifier wraps it; the actual
    /// sync work runs as an unstructured Task owned by HostSyncStore.
    public func syncIfSignedIn() {
        guard authSession.isSignedIn else { return }
        scheduleAutoSync()
    }

    // MARK: - Internal serialization

    private func scheduleAutoSync() {
        guard !manualInProgress else {
            pendingAutoAfterManual = true
            return
        }
        _ = startSync()
    }

    /// Append a new sync onto the serialized chain. The new task cancels
    /// the previous one and waits for it to fully exit (drain) before
    /// running its own work ‚Äî guarantees mutual exclusion across
    /// consecutive sync passes.
    @discardableResult
    private func startSync() -> Task<Void, Error> {
        let prev = inFlight
        let new = Task { [weak self] in
            guard let self else { return }
            prev?.cancel()
            _ = await prev?.result   // drain ‚Äî always resolves
            try Task.checkCancellation()  // we may have been replaced too
            try await self.performSync()
        }
        inFlight = new
        return new
    }

    // MARK: - Sync work

    private func performSync() async throws {
        let remote = try await client.listHosts()
        try Task.checkCancellation()
        let ops = HostSyncReconciler.reconcile(local: sessionStore.hosts,
                                                remote: remote)
        for op in ops {
            try Task.checkCancellation()
            try await apply(op)
        }
    }

    private func apply(_ op: SyncOperation) async throws {
        // ‚òÖ Critical invariant: do NOT insert Task.checkCancellation()
        // between client.createHost(...) and sessionStore.setServerId(...)
        // ‚Äî see ¬ß4.2.1.
        switch op {
        case let .createRemote(localHostId):
            guard let host = sessionStore.hosts.first(where: { $0.id == localHostId }) else { return }
            let input = RemoteHostCreateInput(
                name: host.name, hostname: host.hostname,
                port: host.port, username: host.username
            )
            let out = try await client.createHost(input)
            try sessionStore.setServerId(out.id, for: localHostId)
        case let .createLocal(remote):
            try sessionStore.addRemoteHost(remote)
        case let .updateRemote(localHostId, serverId):
            guard let host = sessionStore.hosts.first(where: { $0.id == localHostId }) else { return }
            let input = RemoteHostUpdateInput(
                id: serverId, name: host.name, hostname: host.hostname,
                port: host.port, username: host.username
            )
            try await client.updateHost(input)
        case let .updateLocal(localHostId, remote):
            try sessionStore.applyRemoteMetadata(localHostId: localHostId, remote: remote)
        case let .deleteLocal(localHostId):
            try sessionStore.deleteHost(id: localHostId)
        }
    }
}
```

#### 4.2.1 Cancellation invariant in `apply(.createRemote)`

`Task.cancel()` in Swift Concurrency is *cooperative*: an `await` does not auto-throw on cancellation; the awaited operation must check cancellation itself. `URLSession.data(for:)` does respond to cancellation (translates to `NSURLErrorCancelled`), so `client.listHosts()` and `client.createHost(...)` will throw promptly when cancelled. Pure local methods like `setServerId` do not check.

The forbidden code is:

```swift
case let .createRemote(localHostId):
    let out = try await client.createHost(input)
    try Task.checkCancellation()                       // ‚òÖ NEVER PUT THIS HERE
    try sessionStore.setServerId(out.id, for: localHostId)
```

If a cancel arrives between `createHost` returning and `setServerId` being called, the server has the host but the local Host has no `serverId`. On the next sync the reconciler sees a local Host with no `serverId` and creates *another* server-side host ‚Äî a duplicate. The window is small but not zero: cancellations can fire from arbitrary mutations, so this is a real concern.

The correct shape is the one in ¬ß4.2: cancel checkpoints between `listHosts` and the op loop, and between ops, but never *inside* `apply(.createRemote)` after the server call.

`apply(.createLocal)`, `apply(.updateLocal)`, and `apply(.deleteLocal)` are pure local writes and have no `await`, so cooperative cancellation can't fire mid-op. `apply(.updateRemote)` and `apply(.deleteLocal)` end with the await/local op respectively and have no second-step server-binding to protect.

### 4.4 `SyncSettingsView.swift` (modify, focused)

Two surface changes: error type widens to `ServerSyncError?`, Account section dispatches on a 3-state derivation.

```swift
@State private var lastSyncError: ServerSyncError?    // was String?
```

The display still works because `ServerSyncError` is `CustomStringConvertible`:


In `syncNow()`:

```swift
private func syncNow() async {
    isSyncing = true
    defer { isSyncing = false }
    lastSyncError = nil
    do {
        try await syncStore.sync()

        lastSyncError = err
```

Account section becomes 3-state, driven by a free function `accountState(isSignedIn:lastSyncError:) -> AccountState`:

```swift
enum AccountState: Equatable {
    case signedOut
    case signedIn
    case sessionExpired
}

func accountState(isSignedIn: Bool, lastSyncError: ServerSyncError?) -> AccountState {
    if isAuthFailure(lastSyncError) { return .sessionExpired }
    return isSignedIn ? .signedIn : .signedOut
}

private func isAuthFailure(_ err: ServerSyncError?) -> Bool {
    guard let err else { return false }
    switch err {
    case .http(status: 401, _):       return true
    case .orpc(_, status: 401, _):    return true   // oRPC envelope wraps 401 here, NOT .http
    case .authFailed:                  return true
    case .notSignedIn:                 return true
    default:                           return false
    }
}
```

**Why both `.http(401)` and `.orpc(_, 401, _)`**: `URLSessionServerSyncClient` parses the response body ‚Äî if it decodes as an oRPC error envelope (the common case for `/api/rpc/sshHost/list` returning 401 from better-auth), the error is thrown as `.orpc("UNAUTHORIZED", 401, ...)`. Only when the body fails to decode as an oRPC envelope does the client fall through to raw `.http(status:body:)`. The `ServerSyncClientHTTPTests.swift:58` test pins the oRPC shape:

```swift
catch let ServerSyncError.orpc(code, status, _) {
    XCTAssertEqual(code, "UNAUTHORIZED")
    XCTAssertEqual(status, 401)
}
```

`isAuthFailure` must match both shapes or the "Sign In Again‚Ä¶" button silently fails to appear in production while the unit tests look green.

The view renders:

```swift
Section("Account") {
    switch accountState(isSignedIn: authSession.isSignedIn,
                        lastSyncError: lastSyncError) {
    case .signedOut:
        Button("Sign In‚Ä¶") { showSignIn = true }
    case .signedIn:
        Button("Sign Out") { Task { await signOut() } }
            .disabled(isSigningOut)
    case .sessionExpired:
        VStack(alignment: .leading, spacing: 4) {
            Text("Session expired").font(.caption).foregroundColor(.secondary)
            Button("Sign In Again‚Ä¶") {
                Task {
                    try? await authSession.signOut()  // clear local cookie
                    showSignIn = true
                    lastSyncError = nil               // hide the red text once user moves
                }
            }
        }
    }
}
```

`accountState` and `isAuthFailure` live `internal` in `SyncSettingsView.swift` so the `CatermTests` target can `@testable import Caterm` and unit-test the derivation directly (per ¬ß5.3).

### 5.4 Invariants

1. **No echo loop** ‚Äî `addRemoteHost / applyRemoteMetadata / setServerId` never call `mutationsForSyncSubject.send()`. Tested by `testRemoteApplyDoesNotEmit` (¬ß7.1).
2. **Credential change does not sync** ‚Äî `setCredentialOnly` never sends. Tested by `testSetCredentialOnlyDoesNotEmit` (¬ß7.1).
3. **Manual sync is not cancelled by subsequent auto** ‚Äî auto schedules during `manualInProgress` go to `pendingAutoAfterManual`; they don't cancel manual via the chain. Tested by `testAutoSyncDeferredDuringManual` (¬ß7.2).
4. **At most one sync at a time across the chain** ‚Äî every new task `await prev?.result` before starting `performSync()`. Tested by `testChainSerializesPasses` (¬ß7.2).
5. **`createHost` ‚Üí `setServerId` is uninterruptible at the cancellation level** ‚Äî no `Task.checkCancellation()` between them. Pinned by code review against ¬ß4.2.1; not a unit-testable property at the public API, but the code shape is small enough to enforce via review.
6. **No `mutationsForSyncSubject.send()` on persistence failure** ‚Äî `send()` line is *after* `try HostPersistence.save(...)`. Tested implicitly by `testAddHostEmits` (which expects exactly one emit on success); a write failure throws before the send.
7. **Manual replay catches missed mutations** ‚Äî a mutation arriving during a manual sync is captured by `pendingAutoAfterManual` and replayed in manual's `defer`. Tested by `testPendingAutoReplaysAfterManual` (¬ß7.2).
8. **Write side of mutation publisher is private** ‚Äî only `SessionStore` can `send()`; outside callers see `AnyPublisher<Void, Never>`. Compile-time guarantee, no runtime test needed.

## 6. Error handling

### 6.1 Categorized table (recap of ¬ß4 of brainstorm)

| Path | Error class | Behavior |
|---|---|---|
| Startup / Post-mutation | offline / DNS / 5xx | `try?` swallows. UI silent. |
| Startup / Post-mutation | 401 / auth-failed | `try?` swallows. UI silent until user opens Sync Settings. |
| Startup / Post-mutation | `CancellationError` (new mutation interrupted) | `try?` swallows. New Task takes over. |
| Startup / Post-mutation | reconcile partial fail | `try?` swallows. Partial ops have already been applied; remaining ops will be picked up by the next sync. |
| Manual Sync Now | any | throws ‚Üí `lastSyncError = err` ‚Üí red `description` text. |
| Manual Sync Now | 401 / auth-failed / notSignedIn | as above + Account section flips to `.sessionExpired`. |

### 6.2 Auth invariant: `isSignedIn` lies after server-side expiration

`AuthSession.isSignedIn` reads `HTTPCookieStorage` and returns true as long as `better-auth.session_token` is present locally. The server can expire the session while the cookie still lives. Both auto paths therefore can fail with 401 even though `isSignedIn == true`; this is by design ‚Äî auto paths swallow, and the user discovers it via Sync Settings.

The `accountState` derivation (¬ß4.4) is the bridge: when *manual* Sync Now lands a 401, the view shows "Sign In Again‚Ä¶" instead of "Sign Out", short-cutting the otherwise-required two-step (Sign Out, then Sign In) recovery.

### 6.3 No backoff / retry loop

Triggers are bounded by `launch ‚à™ user-mutation ‚à™ manual-click`. There is no timer-driven retry. Therefore:

- N consecutive failures cost ‚â§ N √ó (one TCP-fast-fail or HTTP round trip), bounded by user activity.
- No server-side rate-limiting concerns from a runaway client.
- Silent failures don't accumulate visible noise.

If 2.x adds (c) periodic background sync, this design must be revisited ‚Äî periodic + no-backoff would hammer a degraded server.

### 6.4 Cancellation propagation

`Task.cancel()` is cooperative. `await` calls in `performSync()` that respond:

- `client.listHosts()` ‚Äî `URLSessionDataTask` cancellation produces `NSURLErrorCancelled`, surfaced as a thrown error
- `client.createHost / updateHost / deleteHost` ‚Äî same

Calls that don't auto-respond (and therefore need the explicit `Task.checkCancellation()` before them):

- `apply(_:)` body for `.createLocal / .updateLocal / .deleteLocal` ‚Äî pure local SessionStore writes, no await

Per ¬ß4.2 / ¬ß4.2.1, `try Task.checkCancellation()` is placed:

- Once between `client.listHosts()` and the op loop ‚Äî guarantees we don't apply a stale op set if cancelled during reconciliation
- Once at the top of each loop iteration ‚Äî guarantees we don't start a new op after cancellation
- Once at the top of `startSync()`'s Task body, after `await prev?.result` ‚Äî guarantees we don't run a sync that was already replaced before its predecessor finished draining

It is **never** placed inside `apply(.createRemote)` between the server `createHost` and the local `setServerId`.

## 8. AuthSession protocol vs. subclass

`HostSyncStore` only needs `authSession.isSignedIn` (a `Bool`). Two ways to make it testable:

**Option A (chosen) ‚Äî extract a protocol:**

```swift
public protocol AuthSessionProtocol: AnyObject {
    var isSignedIn: Bool { get }
}
extension AuthSession: AuthSessionProtocol {}
```

`HostSyncStore.init` takes `authSession: AuthSessionProtocol`. Tests pass a stub conforming to the protocol with no superclass init. Real app passes the existing `AuthSession`.

**Option B (rejected) ‚Äî drop `final` on `AuthSession`:** subclassing forces tests to provide a working `URL` to `super.init` and inherit all the unrelated machinery (cookie storage, sign-in/out HTTP). Drift surface is wider; the protocol is a smaller, more honest API for what `HostSyncStore` actually depends on.

Net change: `AuthSession` stays `final`; new protocol added in the same file (`AuthSession.swift`); `HostSyncStore.init` uses the protocol type.
