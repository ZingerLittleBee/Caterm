# Task v1.5 — Terminal Mouse / Clipboard / IME / URL — Design

Status: Draft (brainstorming-approved 2026-04-29)
Owner: barcelona-v1 / TerminalEngine
Predecessor: Phase 2 v1.4 (sidebar status indicator) — done
Possible v1.6+ follow-ups (not committed in this spec): configurable `clipboard-trust`, find-in-scrollback (⌘F), Select All if libghostty exposes the API, accessibility tree, image / rich-content drop, mouse-reporting URL preview hover popovers.

## 1. Problem

Phase 1 wired only the **keyboard** path between AppKit and libghostty. After SSH connects, users cannot:

- Click to position cursor or interact with mouse-aware TUIs (`htop`, `vim`, `lazygit`, …)
- Drag-select text in scrollback
- Scroll
- ⌘C copy / ⌘V paste / right-click pasteboard menu
- Use Chinese / Japanese / Korean input (no IME)
- ⌘-click URLs printed by remote programs

Symptoms appear "after SSH" only because the prior smoke path was `$SHELL`, where mouse-less use is acceptable. **The gap is not SSH-specific.**

## 2. Goals

Bring the macOS app to "complete parity" (tier C) with stock Terminal.app / Ghostty for terminal-side interaction:

- Mouse selection, click, drag, scroll, mouse-reporting forwarding
- ⌘C / ⌘V / right-click context menu, Edit-menu integration
- OSC 52 — `write = allow`, `read = ask` (hardcoded for v1)
- Scrollback keyboard navigation (`⌘↑/↓ ⌘⇞/⇟ ⌘Home/End ⌘K`)
- IME marked-text + commit via `NSTextInputClient`
- URL hover underline + ⌘-click open via `NSWorkspace`
- File-drop → shell-quoted path paste

## 3. Non-Goals (v1.5)

- Configurable `clipboard-trust` UI (defer until users ask)
- Find-in-scrollback (⌘F)
- macOS Services menu integration
- Image / rich-content drop (only `.fileURL` and `.string` accepted)
- VoiceOver / a11y tree
- Tab-scoped clipboard isolation (system `NSPasteboard.general` is shared as-is)
- ⌘-click semantics inside alt-screen apps (libghostty disables hover; we don't touch)
- macOS Sequoia "Writing Tools" / smart quotes / autocorrect inside terminal — explicitly disabled via `NSTextInputClient` defaults
- **Select All in Edit menu** — `ghostty_surface_select_all` does not exist in the header. Will revisit in v1.6+ if libghostty adds it or if we want to implement local selection ourselves.
- **`SurfaceProtocol` abstraction for testability** — IME / clipboard coverage is manual smoke only in v1.5; `final class GhosttySurface` is not opened for mocking.

## 4. Architecture

```
GhosttySurfaceNSView (AppKit responder, NSTextInputClient, NSDraggingDestination)
        │ method calls (@MainActor)
        ▼
GhosttySurface (Swift wrapper)
        │ ghostty_surface_* C calls
        ▼
libghostty (selection state, mouse reporting, preedit, URL detection)
        │ runtime callbacks (see §6 — assume-main for action_cb,
        │                    explicit DispatchQueue.main hop for clipboard cb)
        ▼
GhosttyApp (clipboard / action callbacks → NSPasteboard / NSWorkspace / NSAlert)
```

## 5. Components

### 5.1 Mouse (task 1)

**`GhosttySurface+Mouse.swift`** exposes:

```swift
@MainActor extension GhosttySurface {
    func sendMouseButton(state: MouseState, button: MouseButton, mods: NSEvent.ModifierFlags) -> Bool
    func sendMousePos(x: Double, y: Double, mods: NSEvent.ModifierFlags)
    func sendMouseScroll(deltaX: Double, deltaY: Double, mods: ScrollMods)
    func sendMousePressure(stage: UInt32, pressure: Double)   // optional
    var isMouseCaptured: Bool { get }
}
```

`MouseState` / `MouseButton` / `ScrollMods` are Swift enums 1:1 with the ghostty C enums (`ghostty_input_mouse_state_e`, `ghostty_input_mouse_button_e`, `ghostty_input_scroll_mods_t`).

**`GhosttySurfaceNSView+Mouse.swift`**:

- `acceptsFirstMouse(for:)` returns `true`.
- `updateTrackingAreas()` removes prior areas and installs one with `[.activeInKeyWindow, .mouseMoved, .mouseEnteredAndExited, .inVisibleRect, .cursorUpdate]`.
- Five button pairs forward to `sendMouseButton(.press / .release, …)`. `otherMouseDown/Up` switch on `event.buttonNumber` to map middle / back / forward.
- `mouseDragged` and `mouseMoved` → `sendMousePos` with view-local coordinates (view is `isFlipped == true`, so y is already top-origin).
- `scrollWheel` converts `scrollingDeltaX/Y`; if `!hasPreciseScrollingDeltas`, multiplies by a cached `cellSize`. There is no sync getter for cell metrics in `ghostty.h`; instead `GhosttyApp.actionCallback` handles `GHOSTTY_ACTION_CELL_SIZE` and stores the latest `(width, height)` on the destination surface. View reads `surface.cellSize` for line-step computation; default to `(8, 16)` until the first `CELL_SIZE` action arrives.
- libghostty internally decides whether each event is consumed for selection vs. forwarded to PTY as mouse-reporting escapes (DECSET 1000/1002/1006). We forward unconditionally; Shift-to-override-mouse-reporting is libghostty's default binding.
- Cursor shape comes from `GHOSTTY_ACTION_MOUSE_SHAPE`; `cursorUpdate(with:)` pushes the current `NSCursor`. Visibility from `GHOSTTY_ACTION_MOUSE_VISIBILITY` calls `NSCursor.hide()/unhide()`.

### 5.2 Copy / Paste (task 2)

**Header reality check** — the only paste-relevant APIs in `ghostty.h` are:

```c
void ghostty_surface_text(ghostty_surface_t, const char*, uintptr_t);
bool ghostty_surface_binding_action(ghostty_surface_t, const char*, uintptr_t);
```

There is **no** `ghostty_surface_paste_text`. To get bracketed-paste / size-limit / dangerous-content prompting, paste must be routed via libghostty's internal "paste_from_clipboard" binding, which in turn calls *us* back through `read_clipboard_cb` to fetch the data. The data flow inverts vs. the previous draft:

```
⌘V (or right-click Paste, or NSDraggingDestination drop)
  └─ surface.triggerBindingAction("paste_from_clipboard")
       └─ libghostty internally: needs clipboard data → calls read_clipboard_cb(STANDARD, state)
            └─ our cb reads NSPasteboard.general.string → returns to libghostty
                 └─ libghostty wraps in bracketed-paste / chunks / writes to PTY
```

`ghostty_surface_text` is reserved for IME-committed text in §5.5 — **not** used for paste.

**`GhosttySurface+Clipboard.swift`**:

```swift
@MainActor extension GhosttySurface {
    var hasSelection: Bool { get }
    func readSelection() -> String?
    func triggerBindingAction(_ action: String) -> Bool   // wraps ghostty_surface_binding_action
}
```

`readSelection` calls `ghostty_surface_read_selection`, copies bytes from the returned `ghostty_text_s` into a Swift `String`, then frees via `ghostty_surface_free_text(surface, &text)` (header line 1163). No OQ — confirmed in header.

**`GhosttySurfaceNSView+ContextMenu.swift`**:

- `@objc copy(_:)` reads selection → writes to `NSPasteboard.general` (`.string` type). Pure local; no libghostty round-trip.
- `@objc paste(_:)` raises `surface.pendingLocalPaste = true`, then calls `surface.triggerBindingAction("paste_from_clipboard")`. The actual pasteboard read happens later, inside `read_clipboard_cb` (§5.4). If `triggerBindingAction` returns `false` (libghostty rejected the binding), `pendingLocalPaste` is rolled back to `false` immediately to avoid leaving stale state.
- `@objc pasteAsPlainText(_:)` — same as `paste(_:)` (terminals don't render styled text either way).
- Select All omitted from v1.5: `ghostty_surface_select_all` is not in the header (verified). Listed in §3 Non-Goals; v1.6 candidate.
- `validateMenuItem(_:)` greys Copy when no selection, Paste/Paste-as-plain-text when pasteboard string is nil.
- `menu(for: NSEvent)` returns a fresh `NSMenu` with Copy / Paste / Separator / Clear; AppKit auto-validates. Clear invokes the same `triggerBindingAction("clear_screen")` path as the `super+k` keybind from §5.3 (one wrapper, two callers).

**`CatermApp.swift`** SwiftUI `.commands`:

```swift
CommandGroup(replacing: .pasteboard) {
    Button("Copy")  { NSApp.sendAction(#selector(NSText.copy(_:)),  to: nil, from: nil) }.keyboardShortcut("c")
    Button("Paste") { NSApp.sendAction(#selector(NSText.paste(_:)), to: nil, from: nil) }.keyboardShortcut("v")
    Button("Paste and Match Style") { NSApp.sendAction(#selector(NSText.pasteAsPlainText(_:)), to: nil, from: nil) }.keyboardShortcut("v", modifiers: [.command, .option, .shift])
}
```

(No "Select All" entry — see above.)

### 5.3 Scroll (task 3)

§5.1 forwards events; task 3 ensures behavior across three PTY states is correct (handled inside libghostty given correct forwarding):

| PTY state | Effect | Handled by |
|---|---|---|
| Main screen | Scroll scrollback buffer | libghostty |
| Alt-screen, no mouse-reporting | Translates to `↑/↓` PTY bytes | libghostty |
| Mouse-reporting on (DECSET 1000/1002/1006) | Encodes SGR mouse escape | libghostty |

**Keybind injection** — libghostty exposes only file-load APIs (`ghostty_config_load_file`, `_default_files`, `_recursive_files`, `_cli_args`); there is **no** runtime string-set API. So Caterm-managed keybinds must come from a file. Two files are needed:

- **User config** (`Application Support/Caterm/config`) — `ConfigStore.defaultConfig`. User can edit; user wins on conflict.
- **Caterm-managed snapshot** (new, `Application Support/Caterm/caterm-managed.config`) — written on every launch by `ConfigStore.writeManagedConfig()`. Holds policies that aren't user-editable in v1 (the scrollback keybinds below). Loaded **after** the user config so it takes precedence — no, wait: ghostty's keybind semantics are last-loaded-wins per binding key, so loading managed *after* user means user-defined `super+up = …` would be silently overridden, which is bad UX. Resolution: **load managed first, then user**, so the user file naturally overrides our defaults.

`GhosttyConfig.init` becomes:

```swift
ghostty_config_load_default_files(cfg)                     // ghostty stock defaults
ghostty_config_load_file(cfg, managedConfigPath)           // Caterm scrollback / locked-down
if FileManager.default.fileExists(atPath: userConfigPath) {
    ghostty_config_load_file(cfg, userConfigPath)          // user overrides
}
ghostty_config_finalize(cfg)
```

`ConfigStore.writeManagedConfig()` writes the file atomically (no-op if content unchanged) so Time Machine / fsevents don't churn on every launch.

The managed file content for v1.5:

```
# Caterm-managed; do not edit. Override in your user config at:
#   ~/Library/Application Support/Caterm/config
keybind = super+up=scroll_page_lines:-1
keybind = super+down=scroll_page_lines:1
keybind = super+page_up=scroll_page_fractional:-1
keybind = super+page_down=scroll_page_fractional:1
keybind = super+home=scroll_to_top
keybind = super+end=scroll_to_bottom
keybind = super+k=clear_screen
```

Action names verified against the bundled `GhosttyKit` version at task-3 spike — see Open Question 5.3-OQ-1.

Scroll-on-input ("typing jumps to bottom; remote output does not") is libghostty's default; we add no override.

### 5.4 OSC 52 + clipboard runtime (task 2)

Strategy: `write = allow`, `read = ask`, `supports_selection_clipboard = false`. Hardcoded; no config surface in v1.

**Header ground truth**:

```c
typedef struct {
  const char *mime;
  const char *data;            // null-terminated C string
} ghostty_clipboard_content_s;

typedef enum {
  GHOSTTY_CLIPBOARD_REQUEST_PASTE,
  GHOSTTY_CLIPBOARD_REQUEST_OSC_52_READ,
  GHOSTTY_CLIPBOARD_REQUEST_OSC_52_WRITE,
} ghostty_clipboard_request_e;

typedef bool (*ghostty_runtime_read_clipboard_cb)(
    void* surface_userdata, ghostty_clipboard_e kind, void* state);
typedef void (*ghostty_runtime_confirm_read_clipboard_cb)(
    void* surface_userdata, const char* data, void* request,
    ghostty_clipboard_request_e);
typedef void (*ghostty_runtime_write_clipboard_cb)(
    void* surface_userdata, ghostty_clipboard_e kind,
    const ghostty_clipboard_content_s* contents, size_t count, bool confirm);

void ghostty_surface_complete_clipboard_request(
    ghostty_surface_t, const char* data, void* state, bool confirmed);
```

Two semantic facts to remember:

- **First arg is the surface userdata**, not the runtime userdata (per Ghostty upstream `src/apprt/embedded.zig`). Our `GhosttySurface.init` already sets `surfaceConfig.userdata = Unmanaged.passUnretained(hostView).toOpaque()`, so the cb can recover the NSView (and its surface) without any side table.
- **`size_t` in `write_clipboard_cb` is the array length of `contents`, not a byte count.** Iterate `0..<count` and pick the entry whose `mime` matches the type we want; each `data` is a null-terminated C string.

**Read flow (the part the previous draft got wrong)**

`read_clipboard_cb` is not a "do you have data?" probe. It is the synchronous data-source. Returning `false` means *the request failed*, not "defer to confirm". The correct contract:

```
1. PTY writes OSC 52 read (or user triggers paste_from_clipboard binding)
   → libghostty calls read_clipboard_cb(view_userdata, kind, state)
2. We synchronously read the pasteboard (or, for drag-drop paste, a per-surface
   pendingPasteText), then call:
       ghostty_surface_complete_clipboard_request(surface, ptr, state, confirmed: false)
   (`confirmed: false` here means "data is provided but not yet user-confirmed";
    libghostty itself decides whether to gate on confirm.)
   Return `true`.
3. libghostty inspects the data:
   - For paste (binding-triggered): may flag UnsafePaste / UnauthorizedPaste
     (multi-line, dangerous escapes) and then call confirm_read_clipboard_cb.
   - For OSC 52 read: **expected** to call confirm_read_clipboard_cb so we can
     honor the read=ask policy. If libghostty's heuristic deems the read safe and
     skips confirm, we'd silently leak the clipboard — see 5.4-OQ-1 for the
     spike that confirms this and the in-cb fallback (gate at read_cb itself
     when `pendingLocalPaste == false`).
4. Our confirm cb dispatches to main, presents NSAlert (only for OSC_52_READ kind;
   PASTE kind is silently approved — bracketed-paste safety is libghostty's job),
   then calls complete_clipboard_request again with `confirmed: true` or `false`.
```

The previous draft had step 2 returning `false` and trying to "defer" to step 4. That's not how the API works — `false` discards the state and the request fails.

**Per-surface paste override (drag-drop)**

For §5.7 drag-drop, we need `read_clipboard_cb` to use the dropped path string instead of `NSPasteboard.general`. We also need `read_cb` to know whether the request originated locally (⌘V / drag-drop) or remotely (OSC 52 read) — the cb signature only carries `ghostty_clipboard_e` (STANDARD/SELECTION), no request kind, so we keep two side-channel fields on the surface:

```swift
// On the main GhosttySurface class declaration in GhosttySurface.swift
// (Swift extensions cannot add stored properties; the +Clipboard extension
// only adds methods that read these fields.)
public final class GhosttySurface {
    // ...existing fields...
    @MainActor public var pendingPasteText: String?    // drag-drop / programmatic paste source
    @MainActor public var pendingLocalPaste: Bool = false  // raised before user-initiated paste binding
}
```

Both fields are set on main and read on main (read_cb threading guarded by 6-OQ-2). `pendingLocalPaste` is set to `true` immediately before any local `triggerBindingAction("paste_from_clipboard")` call — that includes ⌘V, the right-click Paste menu, and drag-drop. It is cleared the moment `read_cb` consumes it. If `read_cb` fires *without* the token raised, the request is treated as a remote OSC 52 read; see 5.4-OQ-1 for how that branch resolves.

The cb body (paste path only):

```swift
if surface.pendingLocalPaste {
    surface.pendingLocalPaste = false
    let text = surface.pendingPasteText ?? NSPasteboard.general.string(forType: .string) ?? ""
    surface.pendingPasteText = nil
    text.withCString { ptr in
        ghostty_surface_complete_clipboard_request(surface.raw, ptr, state, /*confirmed:*/ false)
    }
    return true
}
// else: remote read — see "Remote OSC 52 read branch" below.
```

Any code path that wants to paste sets `pendingLocalPaste = true`, optionally `pendingPasteText`, then calls `triggerBindingAction("paste_from_clipboard")`. There is no `surface.pasteText(_:)` API.

**Three callback bodies** (`GhosttyApp.swift`):

```swift
// WRITE — libghostty hands us a *list* of (mime, data) entries. Pick text/plain (or text/utf-8).
private static let writeClipboardCallback: ghostty_runtime_write_clipboard_cb = {
    userdata, kind, contentsPtr, count, _confirm in
    guard kind == GHOSTTY_CLIPBOARD_STANDARD,
          let contentsPtr else { return }

    // Walk the array, find first text-y entry, copy its bytes (C string) into Swift.
    var picked: String?
    for i in 0..<Int(count) {
        let entry = contentsPtr[i]
        guard let dataCStr = entry.data else { continue }
        let mime = entry.mime.map { String(cString: $0) } ?? ""
        if mime.isEmpty || mime.hasPrefix("text/") {
            picked = String(cString: dataCStr)
            break
        }
    }
    guard let str = picked else { return }

    DispatchQueue.main.async {
        let pb = NSPasteboard.general
        pb.clearContents()
        pb.setString(str, forType: .string)
    }
}

// READ — synchronous; must read clipboard *and* call complete_request before returning true.
// Branches on the local-paste token (5.4-OQ-1): with token = local paste; without = remote OSC 52 read.
private static let readClipboardCallback: ghostty_runtime_read_clipboard_cb = {
    userdata, kind, state in
    // userdata == hostView pointer (set in GhosttySurface.init).
    // See §6: this cb's threading is a spike target (6-OQ-2). We assert main here
    // and gather evidence; if it ever fires off-main, redesign per spike outcome.
    guard let userdata else { return false }
    let view = Unmanaged<GhosttySurfaceNSView>.fromOpaque(userdata).takeUnretainedValue()
    guard let surface = view.surface else { return false }

    if surface.pendingLocalPaste {
        // Local paste path — provide data, let libghostty decide whether to confirm.
        surface.pendingLocalPaste = false
        let text = surface.pendingPasteText
            ?? NSPasteboard.general.string(forType: .string)
            ?? ""
        surface.pendingPasteText = nil
        text.withCString { ptr in
            ghostty_surface_complete_clipboard_request(surface.raw, ptr, state, /*confirmed:*/ false)
        }
        return true
    }

    // Remote OSC 52 read path — conservative deny until 5.4-OQ-1 confirms
    // confirm_read_clipboard_cb is reliable for a sheet-driven approval.
    ghostty_surface_complete_clipboard_request(surface.raw, nil, state, false)
    return false
}

// CONFIRM — libghostty escalated; show sheet only for OSC_52_READ. PASTE auto-approves.
private static let confirmReadClipboardCallback: ghostty_runtime_confirm_read_clipboard_cb = {
    userdata, dataCStr, request, requestKind in
    guard let userdata else { return }
    // Snapshot the data string while we're still synchronous (data lifetime is
    // call-scoped per ghostty embedded runtime).
    let snapshot = dataCStr.map { String(cString: $0) } ?? ""

    DispatchQueue.main.async {
        let view = Unmanaged<GhosttySurfaceNSView>.fromOpaque(userdata).takeUnretainedValue()
        guard let surface = view.surface else { return }

        switch requestKind {
        case GHOSTTY_CLIPBOARD_REQUEST_PASTE:
            // User-driven paste; auto-approve. libghostty already checked unsafe-paste
            // policy by the time it called us, so confirming is a green-light.
            snapshot.withCString { ptr in
                ghostty_surface_complete_clipboard_request(surface.raw, ptr, request, true)
            }

        case GHOSTTY_CLIPBOARD_REQUEST_OSC_52_READ:
            ClipboardConfirm.present(on: view.window) { allowed in
                if allowed {
                    snapshot.withCString { ptr in
                        ghostty_surface_complete_clipboard_request(surface.raw, ptr, request, true)
                    }
                } else {
                    ghostty_surface_complete_clipboard_request(surface.raw, nil, request, false)
                }
            }

        case GHOSTTY_CLIPBOARD_REQUEST_OSC_52_WRITE:
            // Matches write-cb policy: silent allow.
            snapshot.withCString { ptr in
                ghostty_surface_complete_clipboard_request(surface.raw, ptr, request, true)
            }

        default:
            ghostty_surface_complete_clipboard_request(surface.raw, nil, request, false)
        }
    }
}
```

**`ClipboardConfirm.swift`** (~30 LOC):

- Sheet-modal `NSAlert`, style `.warning`.
- Buttons: **Deny** (default — Enter/Esc denies), **Allow Once**.
- No "Allow for this session" persistence in v1.
- If the host view has no window, deny silently.

**Thread safety**: `read_clipboard_cb` *must* run synchronously (it returns data inline); we cannot hop. We assert main thread at entry (and instrument per §6 6-OQ-2). `write_clipboard_cb` and `confirm_read_clipboard_cb` are async-ack and hop via `DispatchQueue.main.async` before touching AppKit.

> **Implementation deviation (recorded 2026-04-29, post-implementation)** — the original line said "None of the three uses `MainActor.assumeIsolated`." That turned out to be impossible under Swift 6 strict concurrency: `read_clipboard_cb` is a `@convention(c)` closure that touches `@MainActor`-isolated state on `GhosttySurface` (`pendingLocalPaste`, `pendingPasteText`) and on the NSView. The only way to satisfy isolation without spawning a hop (which would defeat the synchronous contract) is to wrap the body in `MainActor.assumeIsolated { ... }`. The `Thread.isMainThread` assert remains as a runtime tripwire — `assumeIsolated` is an *assertion* about isolation, not a hop, so the precondition still holds. `write_clipboard_cb` and `confirm_read_clipboard_cb` are unchanged: they hop via `DispatchQueue.main.async`. See commit `990ed1a`.

### 5.5 IME (task 4)

The naive flow ("always `interpretKeyEvents`, route `insertText` → `surface_text`, route `doCommand` → `surface_key`") is **wrong**: it makes every printable keypress (`a`, `1`, `Return`) bypass `ghostty_surface_key`, which is where libghostty does keybind matching, modifier translation, and kitty-keyboard-protocol encoding. We'd lose all ghostty keybinds for normal characters and corrupt how the PTY sees printable input.

**Correct flow — ghostty-key-first, IME-only-when-composing**:

```
keyDown(event):
  let isComposing = inputContext?.handleEvent triggered setMarkedText? (tracked via field)

  if hasMarkedText OR event likely starts composition:
      // Hand fully to AppKit IME machinery; surface_key gets composing=true
      // so libghostty knows not to interpret as a binding.
      ghostty_surface_key(event, composing: true)        // ghostty sees the keystroke
      interpretKeyEvents([event])                         // AppKit drives setMarkedText / insertText
      // setMarkedText → surface.setPreedit(...)
      // insertText (committed)  → surface.sendText(...) + clear preedit
  else:
      // Normal printable / chord input. Let ghostty decide.
      let consumed = ghostty_surface_key(event, composing: false)
      if !consumed {
          // ghostty didn't bind it; AppKit gets a chance for accelerators
          // it owns (Edit-menu items already fired before keyDown via responder
          // chain, so this branch is mostly noop).
      }
```

**`GhosttySurface+IME.swift`**:

```swift
@MainActor extension GhosttySurface {
    func sendText(_ s: String)            // ghostty_surface_text — committed text only
    func setPreedit(_ s: String)          // ghostty_surface_preedit (nil/0 to clear)
    func imePoint() -> NSRect             // ghostty_surface_ime_point → view-local rect
    // sendKey is updated to take `composing: Bool`, fed into ghostty_input_key_s.composing.
}
```

**`GhosttySurfaceNSView+TextInput.swift`** implements:

- `insertText(_:replacementRange:)` → `sendText(string)`, then `setPreedit("")` and `markedString = ""`.
- `setMarkedText(_:selectedRange:replacementRange:)` → `setPreedit(string)`; track `markedString`.
- `unmarkText()` → `setPreedit("")` and clear `markedString`.
- `hasMarkedText / markedRange / selectedRange / attributedSubstring / characterIndex / validAttributesForMarkedText` — minimal correct implementations. `validAttributesForMarkedText` returns `[]` to opt out of styled marked text (terminals don't render styles).
- `firstRect(forCharacterRange:actualRange:)` → `surface.imePoint()` → `convert(toView: nil)` → `window.convertToScreen(_:)`.
- `doCommand(by:)` is a noop. (We don't synthesize keystrokes from selectors. The original event already went through `surface_key` *before* `interpretKeyEvents`.)

**Why surface_key fires before interpretKeyEvents** — libghostty needs every keystroke for binding lookup and protocol encoding, even during composition (it expects `composing=true` so it skips bindings but still tracks the keystroke for some keyboard protocols). Sending it first means we don't miss the event when AppKit decides to consume it.

**AppKit menu vs ghostty keybind ownership**:

| Owner | Bindings |
|---|---|
| AppKit Edit menu (responder-chain dispatch) | ⌘C, ⌘V, ⌘⌥⇧V (Paste & Match Style) |
| Caterm-managed config (§5.3) | ⌘↑, ⌘↓, ⌘⇞, ⌘⇟, ⌘Home, ⌘End, ⌘K |
| Ghostty stock defaults | everything else (font sizing, etc.) |

The two key chords overlap zero — both groups are explicit lists. We do **not** add `super+c / super+v` to the managed config; AppKit menu wins because it dispatches before `keyDown`.

**Test coverage** (per §8) must include — beyond control chords:
- printable ASCII (`a`, `1`, `?`) — round-trip to PTY unchanged
- Option/Alt-modified printable (`⌥e` for é dead-key on US-Intl, ` ⌥3` for `£` on UK)
- Function keys (`F1`–`F12`) — ghostty kitty-protocol encoding
- Pinyin / Japanese full IME flow (covered in manual smoke)
- Ghostty keybind not occluded by AppKit (e.g. user adds `keybind = ctrl+shift+t=...` to user config — must fire)

### 5.6 URL hover + ⌘-click (task 5)

libghostty does URL detection internally. We add:

**`GhosttyApp.actionCallback`** new cases:

- `GHOSTTY_ACTION_MOUSE_OVER_LINK`: read `mouse_over_link.url` (NULL means hover-leave) → `surface.handleHoverURL(_:)`.
- `GHOSTTY_ACTION_OPEN_URL`: read `open_url.url` and `kind` → `surface.handleOpenURL(_:kind:)`.

**`GhosttySurface`** publishes:

```swift
public var onHoverURL: ((String?) -> Void)?
public var onOpenURL:  ((String, ghostty_action_open_url_kind_e) -> Void)?
```

**`GhosttySurfaceNSView`** subscribes after surface creation:

- `onHoverURL` updates `hoveredURL` field and triggers `cursorUpdate`.
- `onOpenURL` validates scheme against a `private static let safeSchemes: Set<String> = ["http", "https", "mailto", "ssh", "ftp", "ftps"]` on the view → `NSWorkspace.shared.open(_:)`. Other schemes pass through `confirmAndOpen(_:)` (NSAlert sheet, default Deny).
- `flagsChanged(with:)` re-sends the current mouse position with updated mods and triggers `cursorUpdate` (so cursor flips to `.pointingHand` when ⌘ is held over a URL).

The actual ⌘-click triggers libghostty internally — we don't intercept; ghostty fires `OPEN_URL` for us.

### 5.7 File drop → paste path (task 2)

**`GhosttySurfaceNSView+Drag.swift`** implements `NSDraggingDestination`:

- `init` registers `[.fileURL, .string]`.
- `draggingEntered` → `.copy`.
- `performDragOperation` reads `NSURL`s; falls back to `.string`.
- File URLs are joined with single space; each path is shell-escaped by wrapping in single quotes and replacing `'` → `'\''`.
- The composed string is then routed through the same paste-binding pipeline used by ⌘V (no separate `pasteText` API exists — see §5.4):

  ```swift
  surface.pendingPasteText = composedShellString
  surface.pendingLocalPaste = true
  if !surface.triggerBindingAction("paste_from_clipboard") {
      // Binding rejected — clear both fields so a later real paste isn't tainted.
      surface.pendingPasteText = nil
      surface.pendingLocalPaste = false
  }
  ```

  When libghostty calls `read_clipboard_cb` it consumes `pendingPasteText` first (§5.4), so the system pasteboard is *not* read for drag-drop pastes. libghostty still wraps in bracketed-paste / chunks / runs unsafe-paste detection.

Image / rich content is rejected (out of scope).

## 6. Threading model

**Critical clarification** — `MainActor.assumeIsolated { … }` is a **runtime assertion**, not a thread hop. If the closure runs on a non-main thread, it crashes (in debug builds with `-warnings-as-errors-on` style) or produces undefined behavior touching AppKit. We must distinguish callbacks that ghostty *guarantees* on main from those that don't.

| Source | Thread | Egress rule |
|---|---|---|
| AppKit events (mouse / key / menu / drag) | main | direct call into `surface.*` (already `@MainActor`) |
| libghostty `actionCallback` | **observed-main** (Cocoa apprt drives ghostty work from the main runloop; verified by Phase-1 smoke) | continue using `MainActor.assumeIsolated { … }` — but treat as a load-bearing assumption documented here. If a future ghostty release violates it, every action-cb consumer must hop. |
| libghostty `read_clipboard_cb` | **must be main** (cb is synchronous and reads `NSPasteboard` / a `@MainActor` `pendingPasteText`) | assert `Thread.isMainThread` at entry; if it ever fires off-main the design has to change — see 6-OQ-2. No hop possible: the cb has to call `complete_clipboard_request` *before returning true*, so async hop would race the return value. |
| libghostty `confirm_read_clipboard_cb` (async ack) | any | snapshot the `data` C-string into a Swift `String` **inside the C body** (data is call-scoped), then `DispatchQueue.main.async { … }` before touching `NSPasteboard` / `NSAlert` / `surface.raw`. Capture `request` token (`void*`) into the closure. |
| libghostty `write_clipboard_cb` (async one-way) | any | walk the `contents` array and copy text into a Swift `String` **inside the C body** (entries are call-scoped), then `DispatchQueue.main.async { … }` before `NSPasteboard`. |
| `wakeupCallback` | any | currently noop; will become `RunLoop.main.perform` if used |

**Iron rules**:

1. `@convention(c)` callback bodies **copy any C data they need** before any thread hop. C-string lifetimes are not guaranteed past the cb return. The single exception is `read_clipboard_cb`, which doesn't hop and uses the C string up to the `complete_clipboard_request` call.
2. `assumeIsolated` is allowed *only* for action-cb (with the documented assumption above). Everywhere else, hop with `DispatchQueue.main.async`.
3. **Never** touch `NSPasteboard`, `NSAlert`, `NSCursor`, or any NSView/NSWindow inside the C body itself, **except** in `read_clipboard_cb` where the synchronous contract requires it (and the main-thread assertion guards it).

**Pre-flight checks at task 1 / task 2**:

- Task 1: `Thread.isMainThread` log at the top of `actionCallback`, kept in dev builds for the rest of v1.5.
- Task 2: same assertion at the top of `read_clipboard_cb`. If it ever fires off-main, halt and rework — the fallback is to pre-cache pasteboard text on every `NSPasteboard.changeCount` tick (or on key window changes) so the cb has a stale-but-valid value to return synchronously without touching AppKit. Decision deferred to spike outcome.

## 7. Errors / unsafe boundaries

- C-string lifetime: stack-scoped via `withCString` is sufficient for `surface_text / surface_preedit / binding_action / complete_clipboard_request` — libghostty reads synchronously during the call (see §5.4 spike note).
- `ghostty_text_s` (output of `read_selection`) ownership: explicit free via `ghostty_surface_free_text(surface, &text)` (header line 1163). Always pair the read.
- `ghostty_clipboard_content_s.data` is a null-terminated C string per entry. The `size_t` argument in `write_clipboard_cb` is the **count of `contents` entries** (array length), *not* a byte length — iterate `0..<count` and decode each entry's `data` independently.
- C strings from libghostty are decoded with `String(cString:)` and treated as UTF-8 since that's what ghostty emits. v1.5 does not separately validate; non-UTF-8 input will produce a garbage string which downstream `NSPasteboard.setString(_:)` accepts as opaque text. If a future hardening pass needs strict validation, switch to `String(validatingUTF8:)` and silent-drop invalid input.
- `pendingPasteText` (§5.4 + §5.7) is consumed-and-cleared on first read. If a drag-drop sets it but `read_clipboard_cb` never fires (e.g. the binding action is a noop because libghostty rejects it), the field stays set until the next paste, which would then unexpectedly use the stale path. Mitigation: `triggerBindingAction` returns `Bool` (`ghostty_surface_binding_action` return value — `false` means rejected); on `false`, immediately clear `pendingPasteText` before returning to the drag handler.

### Manual smoke checklist (added to `end-to-end-smoke.md`)

Shell commands are written for bash/zsh (use `$(…)`); fish users translate to `(…)`.

1. Local shell: drag-select text → highlight visible.
2. SSH host: enter `vim` → mouse click moves cursor; Shift-drag-select bypasses mouse-reporting.
3. SSH `htop` → mouse click sorts columns (mouse-reporting active).
4. ⌘C / ⌘V round-trip through `pbpaste` on local; round-trip across SSH via `pbcopy` on a remote macOS host.
5. `printf '\e]52;c;%s\a' "$(printf hello | base64)"` on remote → local pasteboard becomes `hello`. (Note: GNU `base64` and BSD `base64` differ; use `printf` to avoid `echo -n` portability traps.)
6. `printf '\e]52;c;?\a'` on remote → confirm sheet appears; Deny is default. Verify the sheet does **not** appear for ⌘V (only OSC 52 read).
7. Switch to Pinyin IME, type `nihao`, see preedit underline in terminal at cursor; Space commits `你好`; remote `cat` receives `你好` (not `nihao`).
8. IME candidate panel appears anchored to terminal cursor, not screen corner.
9. Remote `echo https://example.com`; hover with ⌘ → cursor flips to pointing hand → ⌘-click opens browser.
10. Remote `printf 'file:///etc/passwd\n'`; ⌘-click → confirmation sheet (non-whitelisted scheme).
11. Drag `My Doc.pdf` from Finder onto terminal → PTY receives `'/Users/.../My Doc.pdf'` (single-quoted, embedded space preserved).
12. `⌘↑` scrolls one line; `⌘⇞` one page; `⌘K` clears screen + scrollback.
13. Pure-keyboard regression: typing `a`, `1`, `?`, `Return` produces those bytes at PTY (verify with `cat | xxd`); `⌥e` then `e` produces `é` (US-Intl layout); a user-defined `keybind = ctrl+shift+t=…` in user config still fires.

## 9. Configuration

No new user-facing config fields in v1.5. Policies live in code or in a Caterm-managed file (not a user-visible schema):

- OSC 52: `write = allow`, `read = ask` — hardcoded in `GhosttyApp` callbacks (§5.4).
- URL scheme whitelist: `{http, https, mailto, ssh, ftp, ftps}` — hardcoded as `private static let safeSchemes` on `GhosttySurfaceNSView`.
- Scrollback keybinds: written each launch by `ConfigStore.writeManagedConfig()` to `Application Support/Caterm/caterm-managed.config`, loaded **before** the user config so user overrides win (§5.3).

Rationale: each user-facing config field requires schema migration in `ConfigStore` + UI surface; not worth the asymmetric cost before internal-beta feedback. The managed-config file is internal plumbing, not a user surface — its existence is documented inline as a comment but it isn't part of any persisted schema.

## 10. Task decomposition

Single plan, **5** sequential tasks (one PR each). Stopping at any task boundary leaves the app shippable. Task 2 bundles "everything required for ⌘C/⌘V/right-click/drag to actually work end-to-end" so paste is never half-wired across PR boundaries — internal commits inside the task are fine.

| # | Title | Files |
|---|---|---|
| 1 | Mouse infrastructure + test-target setup | `Package.swift` (new `TerminalEngineTests` target), `Tests/TerminalEngineTests/` (new), `GhosttySurface+Mouse.swift`, `GhosttySurfaceNSView+Mouse.swift`, mouse-shape / visibility / cell-size action cases in `GhosttyApp.swift`, `Thread.isMainThread` instrumentation in actionCallback (§6 pre-flight) |
| 2 | Clipboard end-to-end (copy + paste + drag + OSC 52) | `GhosttySurface+Clipboard.swift` (`readSelection`, `triggerBindingAction`, `pendingPasteText`), `GhosttySurfaceNSView+ContextMenu.swift`, `GhosttySurfaceNSView+Drag.swift`, `CatermApp.swift` Edit menu, **and** `GhosttyApp.swift` clipboard callback rewrite (write / read / confirm with surface userdata routing), `ClipboardConfirm.swift`. Internally splits into commits: (a) read_selection + Copy menu, (b) clipboard cb rewrite, (c) Paste binding + drag drop. |
| 3 | Scroll polish + managed-config keybinds | scroll mods refinement (precise vs. line, momentum), `ConfigStore.swift` (add `writeManagedConfig`, `managedConfigPath`), `GhosttyConfig.swift` (load order: defaults → managed → user), Caterm-managed config content for scrollback keybinds. |
| 4 | IME | `GhosttySurface+IME.swift`, `GhosttySurfaceNSView+TextInput.swift`, `keyDown` rewrite per §5.5 (ghostty-key-first, IME-only-when-composing), `sendKey` signature update (`composing: Bool`). |

Each task ends with: `swift test` green, manual smoke step(s) for that task documented and exercised.

**Why 2+4 merged**: the previous draft had paste split across two PRs (binding-call site in 2, libghostty round-trip in 4). That meant any commit between them shipped a broken Paste menu item — unacceptable for "stopping at task boundaries leaves the app shippable". Merging 2+4 trades a larger PR for a clean shippable boundary. Reviewer-confirmed.

**Why drag drop moved into task 2 (was task 6 part 2)**: it shares `pendingPasteText` plumbing with paste; splitting it across tasks would mean implementing the same field twice or leaving a half-wired override.
